### Q1. 정답: **(A) 19**

📌 **출제 의도:** 정수 나눗셈(`/`)과 나머지 연산(`%`)을 조합한 식 `a / b * b + a % b`가 항상 원래 값 `a`를 복원하는 **항등식(identity)**임을 이해하고, 연산 우선순위와 정수 나눗셈의 버림(truncation) 특성을 정확히 적용할 수 있는지 테스트

📎 **연계:** AP CSA Unit 1(Using Objects)의 핵심 기초 문제이자, 시험 전반에 걸쳐 등장하는 정수 연산의 근간. Unit 2(Selection/Iteration)에서 자릿수 분리(`n % 10`으로 일의 자리, `n / 10`으로 나머지 자릿수), Unit 4(Data Collections)에서 순환 인덱스 계산(`(i + 1) % arr.length`)과 2D 배열의 행/열 변환(`index / cols`, `index % cols`), 재귀에서 자릿수 합 문제 등에 직접 연결됨. 특히 이 항등식은 "잃어버린 정보를 복원할 수 있는가"라는 근본적인 질문과 연결되며, AP 시험에서 정수 연산 문제가 나올 때마다 떠올려야 할 핵심 공식

> **핵심 개념:** Java에서 `int / int`는 소수점 이하를 무조건 버리는(truncation) **정수 몫**을 반환하고, `int % int`는 **나머지**를 반환한다. `*`, `/`, `%`는 모두 같은 우선순위를 가지며 `+`, `-`보다 먼저 실행된다. 같은 우선순위 연산자끼리는 **왼쪽에서 오른쪽**으로 실행된다.

> **선행 지식:**
> - `/` 연산자 (정수끼리): 소수점 이하를 **버림** (truncation, 반올림 아님!). 예: `19 / 6` → 3.166...이 아니라 **3**
> - `%` 연산자: 나눗셈의 **나머지**를 반환. 예: `19 % 6` → 19를 6으로 나누면 몫 3, 나머지 **1**
> - `*`, `/`, `%`는 모두 동일 우선순위이며, `+`나 `-`보다 **먼저** 실행됨
> - 같은 우선순위 연산자는 **왼쪽에서 오른쪽(left-to-right)** 순으로 실행됨
> - 항등식: **`(a / b) * b + (a % b) == a`** — 이것은 정수 나눗셈의 수학적 정의에서 직접 유도됨. 어떤 정수 `a`를 `b`로 나누면, `a = 몫 × b + 나머지`라는 관계가 성립

**코드:**
```java
int a = 19;
int b = 6;
System.out.println(a / b * b + a % b);
```

**한 줄씩 추적:**

| 단계 | 실행 내용 | 계산 과정 | 결과 |
|---|---|---|---|
| 1 | `int a = 19;` | 정수 변수 `a`에 19 저장 | `a = 19` |
| 2 | `int b = 6;` | 정수 변수 `b`에 6 저장 | `b = 6` |
| 3 | `a / b` 계산 | `19 / 6` → 정수 나눗셈 → 3.166...의 소수점 버림 → **3** | 중간값: `3` |
| 4 | `(a / b) * b` 계산 | `3 * 6` → **18** (왼쪽→오른쪽이므로 `/` 먼저, 그 다음 `*`) | 중간값: `18` |
| 5 | `a % b` 계산 | `19 % 6` → 19를 6으로 나누면 몫 3, 나머지 **1** → 결과 `1` | 중간값: `1` |
| 6 | `(a / b * b) + (a % b)` | `18 + 1` → **19** (`*`와 `%`가 `+`보다 우선순위 높으므로 먼저 계산됨) | 최종: `19` |
| 7 | `println(...)` | **19** 출력 | 출력: `19` |

**수학적 증명:**

임의의 정수 `a`와 `b`(b != 0)에 대해, 정수 나눗셈의 정의에 의해:
- `a = (a / b) * b + (a % b)` 가 **항상** 성립한다.
- 이것은 "19 = 몫 × 나누는수 + 나머지" → "19 = 3 × 6 + 1"과 동일한 원리
- 따라서 `a / b * b + a % b`의 결과는 입력값 `a`와 `b`에 관계없이 **항상 a 자체**

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| **(A) 19** | ✅ 정답 | `3 * 6 + 1 = 18 + 1 = 19`. 항등식에 의해 원래 값 `a`가 복원됨 |
| (B) 18 | `a / b * b`만 계산하고 `a % b` 부분을 빼먹은 경우. 즉, `3 * 6 = 18`까지만 하고 나머지 `+1`을 잊음 | `a / b * b`는 `a`에서 나머지를 뺀 값(18)이고, 여기에 `a % b`(1)를 더해야 비로소 원래 값이 됨. 식 전체를 끝까지 계산하지 않은 실수 |
| (C) 20 | `19 / 6`을 **반올림**하여 `3.166... ≈ 3`이 아니라 뭔가 잘못 계산한 경우. 또는 연산 순서를 `a / (b * b) + a % b`로 잘못 읽어 `19 / 36 + 1 = 0 + 1 = 1`도 아니고 완전히 다른 계산을 한 경우 | Java 정수 나눗셈은 **반올림이 아니라 버림**. `19 / 6 = 3`이지 4가 아님 |
| (D) 17 | `a % b`를 잘못 계산한 경우. 예를 들어 `19 % 6 = -1`이라고 생각했거나, 연산 순서를 혼동하여 `a / b * b`를 `a / (b * b)`로 계산한 경우 | `19 % 6`은 명확히 `1`이며, 연산 순서는 좌→우로 `(19/6)*6 + 19%6` |

**비유:** 19개의 쿠키를 6개씩 봉지에 담는다고 상상해보자. 3봉지를 만들 수 있고(`19 / 6 = 3`), 봉지에 담긴 쿠키는 총 18개(`3 * 6 = 18`), 남은 쿠키는 1개(`19 % 6 = 1`). 봉지에 담긴 18개 + 남은 1개 = 원래의 **19개**. 당연하지 않은가? 이것이 바로 이 항등식의 직관이다. 아무리 나누고 남겨도, 합치면 원래대로 돌아온다.

> ⚠️ **함정:** 이 문제의 가장 큰 함정은 "계산이 복잡해 보여서 하나하나 계산하다 실수하는 것"이다. 항등식 `a / b * b + a % b == a`를 알고 있다면 계산 없이 바로 `19`라고 답할 수 있다. AP 시험에서는 이런 패턴을 알아보는 능력이 시간 절약의 열쇠다. 또한 `*`, `/`, `%`의 우선순위가 동일하다는 사실을 모르면 `a / (b * b)`로 잘못 해석할 수 있으니 주의!

> 💡 **꿀팁:** **`(a / b) * b + a % b == a`는 Java의 황금 공식이다.** 이 공식은 다음과 같은 상황에서 강력하게 쓰인다:
> - 자릿수 분리: `123 / 10 * 10 + 123 % 10 = 120 + 3 = 123` (원래 수 복원)
> - 검증: 나눗셈/나머지 계산 결과가 맞는지 빠르게 확인
> - AP 시험: 이 패턴이 보이면 계산할 필요 없이 바로 답이 `a`임을 알 수 있음
> 단, 이 공식은 `b != 0`일 때만 성립한다. `b = 0`이면 `ArithmeticException`(0으로 나누기 오류)이 발생한다.

### Q2. 정답: **(C) 1.5**

📌 **출제 의도:** `double` 타입 변수끼리의 나눗셈이 실수 나눗셈(floating-point division)을 수행한다는 것을 이해하는지 테스트. 특히 `int` 나눗셈과 `double` 나눗셈의 결과 차이를 명확히 구분할 수 있는지 확인

📎 **연계:** AP CSA Unit 1(Using Objects)의 기본 문제이면서, Unit 3(Class Creation)에서 메서드 반환 타입 결정 등에 연결됨. Q1의 정수 나눗셈과 대비하여, "같은 나눗셈 기호(`/`)인데 피연산자 타입에 따라 결과가 완전히 달라진다"는 Java의 핵심 규칙을 묻고 있음. 이 테스트의 Q9(Fraction 클래스)에서도 `(double) num / den`이라는 캐스팅이 등장하여 직접 연결됨

> **핵심 개념:** Java에서 나눗셈 연산자 `/`의 동작은 **피연산자의 타입**에 의해 결정된다. 양쪽 피연산자가 모두 `int`이면 **정수 나눗셈**(소수점 버림), 한쪽이라도 `double`이면 **실수 나눗셈**(소수점 유지)을 수행한다.

> **선행 지식:**
> - `int / int` → 정수 나눗셈 (소수점 버림). 예: `3 / 2 = 1`
> - `double / double` → 실수 나눗셈 (소수점 유지). 예: `3.0 / 2.0 = 1.5`
> - `double / int` 또는 `int / double` → `int`가 자동으로 `double`로 **승격(promotion)** 된 후 실수 나눗셈 수행
> - `double x = 3;` → 정수 리터럴 `3`이 `double`로 자동 변환되어 `x`에는 `3.0`이 저장됨 (widening conversion)

**코드:**
```java
double x = 3;
double y = 2;
System.out.println(x / y);
```

**한 줄씩 추적:**

| 단계 | 실행 내용 | 계산 과정 | 결과 |
|---|---|---|---|
| 1 | `double x = 3;` | 정수 리터럴 `3`이 `double`로 자동 변환 → `3.0` 저장 | `x = 3.0` |
| 2 | `double y = 2;` | 정수 리터럴 `2`가 `double`로 자동 변환 → `2.0` 저장 | `y = 2.0` |
| 3 | `x / y` | `3.0 / 2.0` → 양쪽 모두 `double`이므로 실수 나눗셈 → **1.5** | 중간값: `1.5` |
| 4 | `println(...)` | **1.5** 출력 | 출력: `1.5` |

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) 1 | `3 / 2`를 **정수 나눗셈**으로 계산한 경우. `int / int = 1`이지만, 이 문제에서 `x`와 `y`는 `double` 타입! | `double x = 3;`에서 `x`는 `3.0`으로 저장된다. `x / y`는 `3.0 / 2.0`이므로 실수 나눗셈 |
| (B) 1.0 | `3 / 2 = 1`(정수 나눗셈)을 먼저 수행한 후 `double`로 변환하여 `1.0`이라고 생각한 경우 | 변수 선언 시점에서 이미 `double`로 저장되었으므로, 나눗셈 시점에서는 `3.0 / 2.0`이 실행됨. 정수 나눗셈이 아님 |
| **(C) 1.5** | ✅ 정답 | `3.0 / 2.0 = 1.5` |
| (D) 2.0 | 나눗셈을 반대로 하거나(`2.0 / 3.0` 근처), 완전히 잘못된 계산을 한 경우 | `x / y`이므로 `3.0 / 2.0`이 맞음 |

**비유:** 피자 3판을 2명이 나눈다고 생각해보자. 각자 **1.5판**씩 먹는 것이 자연스럽다. 이것이 `double` 나눗셈이다. 반면 `int` 나눗셈은 "피자를 잘라서 나눌 수 없고 통째로만 줄 수 있다"는 규칙이라서, 한 명당 **1판**만 받고 나머지 1판은 버려지는 것과 같다.

> ⚠️ **함정:** `double x = 3;`에서 `3`은 정수 리터럴이지만, `double` 변수에 대입되면서 자동으로 `3.0`으로 변환된다. 이 과정을 **widening conversion(확장 변환)** 이라고 하며 Java에서 자동으로 수행된다. 만약 `int x = 3; int y = 2; System.out.println(x / y);`였다면 출력은 `1`이 되었을 것이다. 타입 선언 한 줄의 차이가 결과를 완전히 바꾼다!

> 💡 **꿀팁:** AP 시험에서 나눗셈 문제를 만나면 **가장 먼저 양쪽 피연산자의 타입을 확인**하라. 둘 다 `int`인가? 한쪽이라도 `double`인가? 이 한 가지 확인이 정답과 오답을 가른다. 기억법: "**한쪽이라도 소수점이면, 결과도 소수점!**"

### Q3. 정답: **(A) "burge"**

📌 **출제 의도:** `String.substring(beginIndex, endIndex)` 메서드에서 `beginIndex`는 포함(inclusive)이고 `endIndex`는 미포함(exclusive)이라는 핵심 규칙을 정확히 적용할 수 있는지 테스트

📎 **연계:** AP CSA Unit 1(Using Objects) — String 객체. `substring`은 AP CSA에서 가장 빈번하게 출제되는 String 메서드 중 하나이며, Unit 1의 거의 모든 문자열 문제에서 사용됨. 이 테스트의 Q4(substring 비교), Q20(substring + indexOf)을 비롯해, FRQ에서 문자열을 처리하는 거의 모든 문제에서 `substring`을 정확히 사용하는 능력이 요구됨

> **핵심 개념:** `s.substring(begin, end)`는 인덱스 `begin`부터 인덱스 `end - 1`까지의 문자를 추출하여 새로운 문자열을 반환한다. 즉, `begin`은 **포함(inclusive)**, `end`는 **미포함(exclusive)**. 결과 문자열의 길이는 `end - begin`이다.

> **선행 지식:**
> - 문자열 인덱스는 **0부터 시작** (첫 번째 문자 = 인덱스 0)
> - `substring(begin, end)`: 인덱스 `begin` 이상, `end` **미만**의 문자를 추출
> - `substring(begin)`: 인덱스 `begin`부터 **문자열 끝**까지 추출 (1-매개변수 버전)
> - 결과 문자열의 길이 = `end - begin`

**코드:**
```java
String s = "hamburger";
System.out.println(s.substring(3, 8));
```

**인덱스 맵핑:**
```
문자:  h   a   m   b   u   r   g   e   r
인덱스: 0   1   2   3   4   5   6   7   8
                    ^               ^
                  begin=3         end=8
                  (포함)          (미포함)
```

**한 줄씩 추적:**

| 단계 | 실행 내용 | 계산 과정 | 결과 |
|---|---|---|---|
| 1 | `String s = "hamburger";` | 문자열 `"hamburger"` 저장 (길이 9) | `s = "hamburger"` |
| 2 | `s.substring(3, 8)` | 인덱스 3(포함)부터 8(미포함)까지 추출 → 인덱스 3, 4, 5, 6, 7의 문자 → `b`, `u`, `r`, `g`, `e` | `"burge"` |
| 3 | `println(...)` | **"burge"** 출력 | 출력: `"burge"` |

**추출 과정 상세:**

| 인덱스 | 문자 | 포함 여부 |
|---|---|---|
| 0 | h | X (3보다 작음) |
| 1 | a | X |
| 2 | m | X |
| **3** | **b** | **O (begin = 3, 포함)** |
| **4** | **u** | **O** |
| **5** | **r** | **O** |
| **6** | **g** | **O** |
| **7** | **e** | **O (end - 1 = 7, 마지막 포함)** |
| 8 | r | X (end = 8, 미포함) |

결과 길이: `8 - 3 = 5`글자 → `"burge"`

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| **(A) "burge"** | ✅ 정답 | 인덱스 3~7의 5글자: b, u, r, g, e |
| (B) "burger" | `endIndex`를 포함한다고 잘못 생각한 경우. 즉 인덱스 3~8(6글자)을 추출한다고 착각 | `endIndex`는 **미포함(exclusive)**이므로 인덱스 8의 'r'은 포함되지 않음. `substring(3, 8)`은 5글자, `substring(3, 9)`가 "burger"(6글자) |
| (C) "burg" | `endIndex`를 "추출할 글자 수"로 잘못 해석하거나, 인덱스 3~6(4글자)만 추출한 경우. 또는 `substring(3, 7)`의 결과와 혼동 | `endIndex`는 "추출할 글자 수"가 아니라 "여기까지(미포함)" 인덱스. `8 - 3 = 5`글자가 정답 |
| (D) "mburge" | `beginIndex`를 "추출 시작 전 위치"(즉 인덱스 2부터 시작)로 잘못 해석한 경우 | `beginIndex = 3`이면 정확히 인덱스 3의 문자부터 시작. 인덱스 2의 'm'은 포함되지 않음 |

**비유:** `substring(begin, end)`를 울타리 기둥(fence post)으로 생각하자. 인덱스 번호가 붙은 기둥이 각 문자 **앞에** 서 있다. `substring(3, 8)`은 "3번 기둥부터 8번 기둥 **사이**에 있는 문자들을 꺼내라"는 뜻이다. 3번 기둥 뒤에는 `b`가, 7번 기둥 뒤에는 `e`가 있으므로, `b`~`e`까지 5글자를 꺼낸다. 8번 기둥 뒤의 `r`은 8번 기둥 바깥이므로 포함되지 않는다.

> ⚠️ **함정:** AP CSA에서 가장 흔한 실수는 **endIndex를 포함한다고 착각**하는 것이다. `substring(3, 8)`에서 인덱스 8의 문자는 포함되지 **않는다**! Java의 거의 모든 범위 표현은 "begin 포함, end 미포함"(half-open interval) 원칙을 따른다. 이 원칙은 `for (int i = 0; i < length; i++)`에서 `i < length`인 것과 같은 맥락이다.

> 💡 **꿀팁:** `substring`의 결과 길이를 빠르게 구하는 공식: **`end - begin`**. `substring(3, 8)` → `8 - 3 = 5`글자. 이 공식으로 선택지의 글자 수를 세어 빠르게 소거할 수 있다. "burge"는 5글자(O), "burger"는 6글자(X), "burg"는 4글자(X), "mburge"는 6글자(X).

### Q4. 정답: **(D) false**

📌 **출제 의도:** `substring`의 두 가지 형태(2-매개변수 vs 1-매개변수)를 정확히 구분하고, `equals` 메서드로 문자열 내용을 비교할 수 있는지 테스트. 특히 회문(palindrome)처럼 보이는 문자열에서 부분 문자열의 실제 내용이 다를 수 있음을 확인

📎 **연계:** Q3에서 배운 `substring` 인덱싱의 심화 적용. `substring(0, 2)`는 2-매개변수 형태(begin 포함, end 미포함), `substring(3)`은 1-매개변수 형태(begin부터 끝까지). 이 두 형태의 차이를 정확히 아는 것이 Unit 1(Using Objects) String 문제의 핵심이며, FRQ에서 문자열 조작 문제를 풀 때 필수

> **핵심 개념:** `substring(begin, end)`는 인덱스 `begin`~`end-1` 추출, `substring(begin)`은 인덱스 `begin`부터 **끝까지** 추출. `equals()`는 두 문자열의 **내용**을 비교하여 `boolean` 반환.

> **선행 지식:**
> - `substring(begin, end)`: 인덱스 `begin` 이상 `end` 미만 (Q3에서 배움)
> - `substring(begin)`: 인덱스 `begin`부터 **문자열 끝까지** 전부 추출
> - `equals()`: 문자열 내용 비교 (대소문자 구분). 같으면 `true`, 다르면 `false`
> - `==`는 참조 비교이므로 문자열 내용 비교에 사용하면 안 됨 (여기서는 `equals` 사용으로 올바름)

**코드:**
```java
String s = "ABCBA";
boolean result = s.substring(0, 2).equals(s.substring(3));
System.out.println(result);
```

**인덱스 맵핑:**
```
문자:  A   B   C   B   A
인덱스: 0   1   2   3   4
```

**한 줄씩 추적:**

| 단계 | 실행 내용 | 계산 과정 | 결과 |
|---|---|---|---|
| 1 | `String s = "ABCBA";` | 문자열 저장 (길이 5) | `s = "ABCBA"` |
| 2 | `s.substring(0, 2)` | 인덱스 0(포함)~2(미포함) → 인덱스 0, 1 → `"AB"` | `"AB"` |
| 3 | `s.substring(3)` | 인덱스 3부터 끝까지 → 인덱스 3, 4 → `"BA"` | `"BA"` |
| 4 | `"AB".equals("BA")` | 내용 비교: 첫 글자 'A' vs 'B' → 다름! → **false** | `false` |
| 5 | `println(...)` | **false** 출력 | 출력: `false` |

**왜 "AB"와 "BA"인가?**

`"ABCBA"`는 회문(palindrome)이라서 얼핏 보면 앞뒤가 같을 것 같지만:
- `substring(0, 2)` = 처음 2글자 = **"AB"** (A가 먼저, B가 나중)
- `substring(3)` = 인덱스 3부터 끝까지 = **"BA"** (B가 먼저, A가 나중)
- "AB"와 "BA"는 같은 문자로 구성되어 있지만 **순서가 다르므로** `equals`는 `false`

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) true | "ABCBA"가 회문이므로 앞부분과 뒷부분이 같다고 성급하게 판단한 경우. 또는 `substring(0, 2)`가 "AB"이고 `substring(3)`이 "BA"인데 문자 구성이 같다고 `equals`가 `true`라고 착각 | `equals`는 **순서까지 포함하여 완전히 동일**해야 `true`. "AB"와 "BA"는 순서가 다르므로 `false` |
| (B) 컴파일 에러 | `substring(3)`이 잘못된 호출이라고 생각한 경우. 또는 `boolean`과 `String` 타입 관련 오류를 걱정한 경우 | `substring(3)`은 합법적인 1-매개변수 형태. `equals`는 `boolean`을 반환하므로 `boolean result`에 대입 가능. 컴파일 오류 없음 |
| (C) 런타임 에러 | `substring(3)`이 범위를 벗어난다고 생각한 경우 | `"ABCBA"`는 길이 5이고, `substring(3)`은 인덱스 3~4를 추출하므로 범위 내. `StringIndexOutOfBoundsException` 발생하지 않음 |
| **(D) false** | ✅ 정답 | "AB".equals("BA") = false |

**비유:** "ABCBA"를 거울로 비춘다고 상상해보자. 전체 문자열은 거울에 비쳐도 같지만(회문), **앞쪽 조각 "AB"**와 **뒤쪽 조각 "BA"**를 떼어서 비교하면 순서가 뒤집혀 있다. 마치 "가나"와 "나가"가 같은 글자로 이루어져 있지만 다른 단어인 것처럼, "AB"와 "BA"도 다른 문자열이다.

> ⚠️ **함정:** 이 문제의 최대 함정은 **"ABCBA"가 회문이라는 사실에 현혹되는 것**이다. 회문이라는 것은 문자열 전체가 뒤집어도 같다는 뜻이지, 임의의 부분 문자열끼리 같다는 뜻이 아니다. `substring(0, 2)`는 "AB"(앞에서 2글자), `substring(3)`은 "BA"(뒤에서 2글자)인데, 회문에서 앞의 2글자와 뒤의 2글자는 **역순** 관계이지 동일하지 않다! 만약 `substring(0, 2).equals(substring(3).reverse())`였다면 `true`겠지만, `reverse()` 메서드는 String에 없다.

> 💡 **꿀팁:** `substring`의 1-매개변수 버전과 2-매개변수 버전을 헷갈리지 말자:
> - `s.substring(3)` = 인덱스 3부터 **끝까지** → "BA" (2글자)
> - `s.substring(3, 4)` = 인덱스 3부터 4 미만 → "B" (1글자)
> - `s.substring(3, 5)` = 인덱스 3부터 5 미만 → "BA" (2글자) — `substring(3)`과 같은 결과!
> 1-매개변수 버전은 `substring(begin, s.length())`와 동일하다고 기억하면 된다.

### Q5. 정답: **(C) "even big small"**

📌 **출제 의도:** **독립적인 `if`문 3개**와 **`if-else if` 체인**의 차이를 정확히 구분할 수 있는지 테스트. 이것은 AP CSA에서 가장 빈번하게 출제되고, 학생들이 가장 많이 틀리는 유형 중 하나

📎 **연계:** AP CSA Unit 2(Selection/Iteration)의 핵심 문제. `if-else if` 체인에서는 조건이 하나 만족되면 나머지는 건너뛰지만, 독립적인 `if`문에서는 **모든 조건을 하나하나 검사**한다. 이 차이를 이해하지 못하면 Unit 2의 거의 모든 조건문 문제에서 실수하게 됨. 이 테스트의 Q6(드모르간 법칙)과 함께 boolean 논리의 기초를 형성

> **핵심 개념:** Java에서 `if` / `if` / `if`는 **세 개의 독립적인 조건문**이다. 각각은 서로에게 아무런 영향을 주지 않으며, 앞의 `if`가 참이든 거짓이든 상관없이 뒤의 `if`도 **반드시 검사**한다. 반면 `if` / `else if` / `else if`는 **하나의 연결된 체인**으로, 앞의 조건이 참이면 뒤의 조건은 아예 검사하지 않고 건너뛴다.

> **선행 지식:**
> - **독립 `if`문**: 각 `if`가 별개의 문장. 모든 조건을 **항상** 검사
> - **`if-else if` 체인**: 조건들이 서로 연결됨. **첫 번째로 참인 조건**의 블록만 실행하고 나머지 건너뜀
> - `%` 연산자: `n % 2 == 0`이면 짝수
> - 비교 연산자: `>`, `<` 등

**코드:**
```java
int n = 42;
if (n % 2 == 0)
    System.out.print("even ");
if (n > 40)
    System.out.print("big ");
if (n < 50)
    System.out.print("small");
```

**이 코드가 `if-else if`가 아니라 독립 `if`인 이유:**
```java
// 이 문제의 코드 (독립 if 3개):
if (조건1) { ... }     // ← 독립 if #1
if (조건2) { ... }     // ← 독립 if #2 (else가 아님!)
if (조건3) { ... }     // ← 독립 if #3 (else가 아님!)

// 만약 if-else if였다면:
if (조건1) { ... }
else if (조건2) { ... }   // ← else가 붙어있음!
else if (조건3) { ... }   // ← else가 붙어있음!
```

**한 줄씩 추적:**

| 단계 | 조건 | n=42 대입 | 결과 | 출력 |
|---|---|---|---|---|
| 1 | `n % 2 == 0` | `42 % 2 == 0` → `0 == 0` → **true** | 실행! | `"even "` |
| 2 | `n > 40` | `42 > 40` → **true** | 실행! | `"even big "` |
| 3 | `n < 50` | `42 < 50` → **true** | 실행! | `"even big small"` |

세 조건이 모두 참이므로 세 `print`문이 모두 실행된다!

**만약 `if-else if`였다면:**
```java
if (n % 2 == 0)         // true → 이것만 실행!
    System.out.print("even ");
else if (n > 40)         // 건너뜀 (이미 위에서 참이었으므로)
    System.out.print("big ");
else if (n < 50)         // 건너뜀
    System.out.print("small");
// 출력: "even " (A)만 나옴
```

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) "even " | **독립 `if`를 `if-else if`로 착각한 경우.** 첫 번째 `if`가 참이므로 나머지를 건너뛴다고 생각 | `else`가 없으므로 각 `if`는 **독립적**. 첫 번째가 참이어도 두 번째, 세 번째 모두 검사함 |
| (B) "even big " | 세 번째 조건 `n < 50`을 검사하지 않거나 `42 < 50`을 거짓이라고 잘못 판단한 경우 | `42 < 50`은 명백히 참. 세 번째 `if`도 독립적이므로 반드시 검사됨 |
| **(C) "even big small"** | ✅ 정답 | 세 조건 모두 참 → 세 출력문 모두 실행 |
| (D) "big small" | 첫 번째 조건 `n % 2 == 0`을 거짓이라고 판단한 경우. `42 % 2`를 잘못 계산했을 가능성 | `42`는 짝수. `42 / 2 = 21`, 나머지 0. 따라서 `42 % 2 == 0`은 참 |

**비유:** 독립 `if`문은 마치 **쇼핑 체크리스트**와 같다. "우유가 필요한가? → 사자", "빵이 필요한가? → 사자", "달걀이 필요한가? → 사자" — 각각의 체크 항목은 독립적이다. 우유를 샀다고 빵을 안 사는 것이 아니다! 반면 `if-else if`는 **자판기**와 같다. 버튼 하나를 누르면(조건 하나가 참이면) 음료 하나만 나오고 나머지 버튼은 무시된다.

> ⚠️ **함정:** 이 문제는 AP CSA에서 **가장 악명 높은 함정** 중 하나이다. 코드를 빠르게 읽으면 `if` / `if` / `if`가 마치 `if-else if-else if`처럼 보인다. 하지만 `else` 키워드가 **없다**는 것이 결정적인 차이다! AP 시험에서 조건문 문제를 만나면, 반드시 **`else`의 유무를 먼저 확인**하라. `else`가 없으면 독립 `if`, `else`가 있으면 연결된 체인이다.

> 💡 **꿀팁:**
> - **독립 `if`**: `else` 없음 → 모든 조건을 **항상** 검사 → 여러 블록이 동시에 실행 가능
> - **`if-else if`**: `else` 있음 → 참인 조건을 만나면 **즉시 탈출** → 최대 하나의 블록만 실행
> - AP 시험 팁: 문제에서 "`if-else if`를 사용하라"와 "독립 `if`를 사용하라"를 혼동하면 FRQ에서도 감점을 받으므로, 이 차이를 확실히 체화해야 함

### Q6. 정답: **(B) !p && !q**

📌 **출제 의도:** 드모르간의 법칙(De Morgan's Law)을 정확히 적용할 수 있는지 테스트. 이것은 AP CSA에서 매년 1문제 이상 출제되는 필수 개념

📎 **연계:** AP CSA Unit 2(Selection/Iteration)의 핵심 이론 문제. Q5의 조건문과 연결되어, boolean 표현식을 변환하고 단순화하는 능력을 측정함. 드모르간 법칙은 FRQ에서 복잡한 조건을 부정(negate)해야 할 때, 예를 들어 "~가 아닌 경우"를 코딩할 때 직접 적용됨. Unit 2의 while 루프 종료 조건을 계산할 때도 필수

> **핵심 개념:** 드모르간의 법칙(De Morgan's Laws)은 논리 연산의 부정을 변환하는 두 가지 규칙이다:
> 1. `!(p || q)` = `!p && !q` — OR의 부정은 AND로 바뀌고, 각 항이 부정됨
> 2. `!(p && q)` = `!p || !q` — AND의 부정은 OR로 바뀌고, 각 항이 부정됨
> 핵심 패턴: **NOT을 안으로 넣으면 AND↔OR이 뒤바뀐다**

> **선행 지식:**
> - `||` (OR): 하나라도 참이면 참
> - `&&` (AND): 둘 다 참이어야 참
> - `!` (NOT): 참↔거짓 반전
> - 드모르간 법칙: `!`을 괄호 안으로 분배하면 `||`와 `&&`가 서로 바뀜

**문제:** `!(p || q)`와 동치인 표현은?

**드모르간 법칙 적용:**
```
!(p || q)
= !p && !q    ← NOT을 안으로 분배하면서 OR → AND로 변환
```

**진리표로 검증:**

| p | q | p \|\| q | !(p \|\| q) | !p | !q | !p && !q |
|---|---|---|---|---|---|---|
| T | T | T | **F** | F | F | **F** |
| T | F | T | **F** | F | T | **F** |
| F | T | T | **F** | T | F | **F** |
| F | F | F | **T** | T | T | **T** |

`!(p || q)`와 `!p && !q` 열이 **완전히 동일**함을 확인!

**오답 분석:**

| 보기 | 표현 | 왜 틀린가 |
|---|---|---|
| (A) `!p \|\| !q` | 이것은 `!(p && q)`의 결과 (드모르간 법칙의 **다른 쪽**). `||`가 `&&`로 바뀌지 않음 | `!`을 분배하면서 `||`를 `&&`로 바꿔야 하는데 바꾸지 않은 실수. 예: p=T, q=F일 때 `!p\|\|!q = F\|\|T = T`이지만 `!(p\|\|q) = !(T) = F`. 결과가 다름! |
| **(B) `!p && !q`** | ✅ 정답 | 드모르간 법칙을 정확히 적용한 결과 |
| (C) `p && q` | `!`을 완전히 무시하고 `||`만 `&&`로 바꾼 경우 | 각 항의 부정(`!p`, `!q`)도 함께 적용해야 함 |
| (D) `!p \|\| q` | 드모르간 법칙을 부분적으로만 적용한 경우. `p`만 부정하고 `q`는 그대로 두고, `||`도 바꾸지 않음 | 드모르간 법칙은 **모든 항을 부정**하고 **연산자를 반전**해야 올바르게 적용됨 |

**비유:** 드모르간 법칙을 "반대 세계의 언어 변환"이라고 생각하자. 원래 세계에서 "비가 오**거나** 눈이 온다"(`p || q`)의 부정은 "비가 안 오**고** 눈도 안 온다"(`!p && !q`)이다. "거나(OR)"가 "고(AND)"로 바뀌고, 각 항이 부정된다. "비가 안 오거나 눈이 안 온다"(`!p || !q`)는 "비만 안 오면 되거나, 눈만 안 오면 된다"는 뜻이므로, 원래 문장의 부정이 아니다! (비가 오고 눈이 안 오면 이 표현은 참이지만, 원래 의도한 부정은 거짓이어야 함)

> ⚠️ **함정:** 가장 흔한 실수는 **(A) `!p || !q`**를 고르는 것이다. `!`을 분배하면서 `||`를 `&&`로 바꾸는 것을 잊어버리는 실수다. 기억하라: **NOT을 안으로 넣으면 OR↔AND가 반드시 바뀐다!** 이것을 잊으면 드모르간 법칙이 아니라 단순한 `!` 분배가 되어버린다.

> 💡 **꿀팁:** 드모르간 법칙 암기법 — **"NOT이 문을 열면, OR과 AND가 자리를 바꾼다."**
> - `!(A || B)` → `!A && !B` (OR이 AND로)
> - `!(A && B)` → `!A || !B` (AND가 OR로)
> AP 시험에서 확신이 없으면, **진리표의 한두 행만 빠르게 검증**해도 충분하다. 예를 들어 p=T, q=F를 대입해서 원래 식과 선택지의 결과가 같은지 확인하면 오답을 빠르게 제거할 수 있다.

### Q7. 정답: **(B) "54321"**

📌 **출제 의도:** 역순(감소) `for` 루프의 동작을 정확히 추적할 수 있는지 테스트. 특히 문자열 연결(concatenation)에서 정수가 자동으로 문자열로 변환되는 과정을 이해하는지 확인

📎 **연계:** AP CSA Unit 2(Selection/Iteration)의 기본 반복문 문제. 증가 루프(`i++`)는 익숙하지만 감소 루프(`i--`)는 상대적으로 덜 연습하는 학생이 많아 출제됨. Q8의 조건부 누적(conditional accumulation)과 함께 루프 추적 능력의 기초를 형성. Unit 4(Data Collections)에서 배열을 역순으로 출력하거나, 재귀에서 역순 처리를 할 때 이 패턴이 반복됨

> **선행 지식:**
> - `for (초기화; 조건; 갱신)`: 초기화 1번 → (조건 확인 → 참이면 본문 실행 → 갱신) 반복
> - `i--`: `i`를 1 감소
> - `i >= 1`: `i`가 1 이상인 동안 반복 (1 포함!)
> - `String + int`: 정수가 문자열로 자동 변환되어 연결됨. 예: `"" + 5` → `"5"`
> - `+=`는 `result = result + i`의 축약형

**코드:**
```java
String result = "";
for (int i = 5; i >= 1; i--)
{
    result += i;
}
System.out.println(result);
```

**반복 추적:**

| 초기 | 5 | - | - | `""` |
| 1회 | 5 | `5 >= 1` → true | `"" + 5` | `"5"` |
| 2회 | 4 | `4 >= 1` → true | `"5" + 4` | `"54"` |
| 3회 | 3 | `3 >= 1` → true | `"54" + 3` | `"543"` |
| 4회 | 2 | `2 >= 1` → true | `"543" + 2` | `"5432"` |
| 5회 | 1 | `1 >= 1` → true | `"5432" + 1` | `"54321"` |
| 종료 | 0 | `0 >= 1` → false | (루프 탈출) | `"54321"` |

**오답 분석:**

| (C) "5" | 루프가 1번만 실행된다고 생각한 경우. 또는 `result`가 매 반복마다 초기화된다고 착각 | `result`는 루프 **밖에서** 한 번만 초기화되며, 루프는 `i >= 1`인 동안 총 5번 반복 |

### Q8. 정답: **(A) 315**

📌 **출제 의도:** `%` 연산자를 사용한 배수 판별 조건(`i % 3 == 0 && i % 5 == 0`)이 실제로 **15의 배수**를 걸러낸다는 것을 이해하고, 1부터 100까지의 15의 배수의 합을 정확히 계산할 수 있는지 테스트

📎 **연계:** Q1에서 배운 `%` 연산의 실전 적용. AP CSA Unit 2(Selection/Iteration) — 반복문 + 조건문의 조합 문제로, 루프 안에서 조건부 누적(conditional accumulation) 패턴을 사용. 이 패턴은 AP CSA FRQ에서 매우 빈번하게 등장하며, "특정 조건을 만족하는 원소의 합/개수/평균"을 구하는 모든 문제의 기본 구조

> **핵심 개념:** `i % 3 == 0`은 "i가 3의 배수"를, `i % 5 == 0`은 "i가 5의 배수"를 의미한다. 두 조건이 `&&`(AND)로 연결되어 있으므로 **동시에** 만족해야 한다. 즉, i가 3의 배수**이면서** 5의 배수인 수 = **15의 배수(최소공배수)**만 합산된다.

> **선행 지식:**
> - `%` 연산자: `i % n == 0`이면 `i`는 `n`의 배수
> - `&&` (AND): 양쪽 조건이 **모두** 참이어야 참
> - 3의 배수이면서 5의 배수 = 15의 배수 (3과 5는 서로소이므로 LCM = 3 × 5 = 15)
> - 조건부 누적(conditional accumulation): 특정 조건을 만족하는 값만 합산하는 패턴

**코드:**
```java
int sum = 0;
for (int i = 1; i <= 100; i++)
{
    if (i % 3 == 0 && i % 5 == 0)
        sum += i;
}
System.out.println(sum);
```

**왜 `i % 3 == 0 && i % 5 == 0`이 15의 배수인가?**

어떤 수가 3의 배수이면서 동시에 5의 배수라는 것은, 그 수가 3과 5의 **최소공배수(LCM)**의 배수라는 뜻이다. 3과 5는 공통 인수가 없는 서로소(coprime)이므로 LCM = 3 × 5 = **15**. 따라서 `i % 3 == 0 && i % 5 == 0`은 `i % 15 == 0`과 **완전히 동일**하다.

검증: 15 → 15%3=0(O), 15%5=0(O). 30 → 30%3=0(O), 30%5=0(O). 12 → 12%3=0(O), 12%5=2(X) → 탈락. 10 → 10%3=1(X) → 탈락.

**1~100에서 15의 배수 찾기:**

| 15의 배수 | 15×? | 100 이하? |
|---|---|---|
| **15** | 15×1 | O |
| **30** | 15×2 | O |
| **45** | 15×3 | O |
| **60** | 15×4 | O |
| **75** | 15×5 | O |
| **90** | 15×6 | O |
| 105 | 15×7 | X (100 초과) |

**합계 계산:**

| 단계 | 더하는 값 | sum 누적 |
|---|---|---|
| 초기 | - | 0 |
| i=15 | +15 | 15 |
| i=30 | +30 | 45 |
| i=45 | +45 | 90 |
| i=60 | +60 | 150 |
| i=75 | +75 | 225 |
| i=90 | +90 | **315** |

검산: `15 + 30 + 45 + 60 + 75 + 90` = `15 × (1 + 2 + 3 + 4 + 5 + 6)` = `15 × 21` = **315**

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| **(A) 315** | ✅ 정답 | 1~100에서 15의 배수(15, 30, 45, 60, 75, 90)의 합 |
| (B) 735 | `i % 3 == 0 && i % 5 == 0`을 `i % 3 == 0 || i % 5 == 0`(OR)으로 잘못 읽은 경우. 즉, 3의 배수 **또는** 5의 배수를 모두 합산 | `&&`는 "그리고(AND)"이지 "또는(OR)"이 아님. `||`였다면 3의 배수(33개) + 5의 배수(20개) - 15의 배수(6개, 중복 제거) = 47개의 수를 합산하여 훨씬 큰 값이 나옴 |
| (C) 2418 | 3의 배수의 합이거나, 완전히 다른 계산을 한 경우 | 1~100에서 3의 배수의 합은 3+6+9+...+99 = 1683. 이것도 2418은 아니므로 계산 실수일 가능성이 높음 |
| (D) 225 | 하나의 15의 배수를 빠뜨리거나, 합산 과정에서 계산 실수를 한 경우. 또는 15의 배수 5개만 더한 경우(15+30+45+60+75 = 225, 90을 빠뜨림) | 15×6 = 90 ≤ 100이므로 90도 포함해야 함. 6개의 15의 배수를 모두 더해야 315 |

**비유:** FizzBuzz 게임을 알고 있는가? 1부터 숫자를 세면서, 3의 배수이면 "Fizz", 5의 배수이면 "Buzz", **3과 5 모두의 배수이면 "FizzBuzz"**라고 외치는 게임이다. 이 문제는 "FizzBuzz를 외치는 순간의 숫자"만 모아서 합산하는 것이다. 15, 30, 45, 60, 75, 90에서 "FizzBuzz!"를 외치고, 그 숫자들의 합이 315이다.

> ⚠️ **함정:** 이 문제의 최대 함정은 **`&&`(AND)를 `||`(OR)로 혼동하는 것**이다. `&&`이면 15의 배수만(6개), `||`이면 3 또는 5의 배수 전부(47개)가 포함되어 합이 크게 달라진다. AP 시험에서 `&&`와 `||`를 신중히 구분하라! 또한 **225를 고르는 학생은 90을 빠뜨린 것**인데, `15 × 7 = 105 > 100`이므로 90(15×6)이 마지막이라는 것을 확인해야 한다.

> 💡 **꿀팁:** 등차수열의 합 공식을 활용하면 빠르다: `합 = 항의 개수 × (첫 항 + 끝 항) / 2`. 15의 배수: 첫 항 15, 끝 항 90, 항의 개수 6. `합 = 6 × (15 + 90) / 2 = 6 × 105 / 2 = 315`. 또한, **`i % 3 == 0 && i % 5 == 0`은 `i % 15 == 0`과 완전 동치**라는 사실을 알면, 루프를 `for (int i = 15; i <= 100; i += 15)`로 재작성하여 검증할 수 있다.

### Q9. 정답: **(D) 0.8333333333333334**

📌 **출제 의도:** 사용자 정의 클래스의 메서드 동작을 코드를 읽어서 추적(tracing)할 수 있는지 테스트. 특히 분수 덧셈의 알고리즘(`통분`)을 코드로 구현한 것을 이해하고, `(double) num / den` 캐스팅의 결과를 정확히 계산할 수 있는지 확인

📎 **연계:** AP CSA Unit 3(Class Creation)의 핵심 문제. 인스턴스 변수 접근(`this.num`, `other.den`), 새 객체 생성(`new Fraction(...)`), 메서드 체이닝(`f1.add(f2).getValue()`)을 종합적으로 테스트. Q2에서 배운 `double` 캐스팅이 `getValue()` 메서드에서 재등장

> **핵심 개념:** `Fraction` 클래스는 분자(`num`)와 분모(`den`)를 저장하는 분수 객체이다. `add` 메서드는 통분 공식 `a/b + c/d = (a*d + c*b) / (b*d)`를 구현한다. `getValue()`는 `(double) num / den`으로 분수를 소수로 변환한다.

> **선행 지식:**
> - 클래스의 인스턴스 변수: `this.num`은 현재 객체의 분자, `other.num`은 매개변수 객체의 분자
> - 통분 공식: `a/b + c/d = (a×d + c×b) / (b×d)`
> - `(double)` 캐스팅: 정수를 실수로 변환하여 실수 나눗셈을 수행
> - `new Fraction(newNum, newDen)`: 새로운 Fraction 객체를 생성하여 반환

**코드:**
```java
public class Fraction
{
    private int num;
    private int den;

    public Fraction(int n, int d) { num = n; den = d; }
    public double getValue() { return (double) num / den; }

    public Fraction add(Fraction other)
    {
        int newNum = this.num * other.den + other.num * this.den;
        int newDen = this.den * other.den;
        return new Fraction(newNum, newDen);
    }
}
```

```java
Fraction f1 = new Fraction(1, 2);
Fraction f2 = new Fraction(1, 3);
Fraction f3 = f1.add(f2);
System.out.println(f3.getValue());
```

**한 줄씩 추적:**

| 단계 | 실행 내용 | 계산 과정 | 결과 |
|---|---|---|---|
| 1 | `new Fraction(1, 2)` | num=1, den=2 저장 | f1: {num=1, den=2} (= 1/2) |
| 2 | `new Fraction(1, 3)` | num=1, den=3 저장 | f2: {num=1, den=3} (= 1/3) |
| 3 | `f1.add(f2)` 호출 | this=f1, other=f2로 add 메서드 진입 | - |
| 3a | `this.num * other.den` | `1 * 3` = 3 | 중간값: 3 |
| 3b | `other.num * this.den` | `1 * 2` = 2 | 중간값: 2 |
| 3c | `newNum = 3 + 2` | **5** | newNum = 5 |
| 3d | `newDen = this.den * other.den` | `2 * 3` = **6** | newDen = 6 |
| 3e | `return new Fraction(5, 6)` | 새 Fraction 객체 생성 | f3: {num=5, den=6} (= 5/6) |
| 4 | `f3.getValue()` | `(double) 5 / 6` → `5.0 / 6` → **0.8333333333333334** | 출력: 0.8333333333333334 |

**수학적 검증:**
```
1/2 + 1/3 = 3/6 + 2/6 = 5/6

5 ÷ 6 = 0.83333... (순환소수)
Java double 표현: 0.8333333333333334 (부동소수점 표현의 마지막 자리 반올림)
```

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) 0.3333333333333333 | `f2.getValue()` (= 1/3)의 결과를 답한 경우. 덧셈을 수행하지 않고 f2의 값만 출력한다고 착각 | `f3 = f1.add(f2)`이므로 f3는 **덧셈 결과**. f3.getValue()를 호출하므로 5/6의 값이 나와야 함 |
| (B) 0.5 | `f1.getValue()` (= 1/2)의 결과를 답한 경우. 또는 add 메서드가 원래 객체를 변경하지 않는다고 혼동 | add는 **새로운** Fraction 객체를 반환(`return new Fraction(...)`)하며, f3에 그 결과가 저장됨 |
| (C) 1.0 | 분자와 분모를 잘못 계산한 경우. 예: newNum = 1+1 = 2, newDen = 2로 계산하여 2/2 = 1.0 | 통분 공식을 정확히 적용해야 함. `1*3 + 1*2 = 5`, `2*3 = 6` → 5/6 |
| **(D) 0.8333333333333334** | ✅ 정답 | 5/6 = 0.8333...의 Java double 표현 |

**비유:** 피자 한 판을 나눈다고 생각하자. f1은 피자의 1/2(반쪽), f2는 피자의 1/3(삼등분의 하나). `add` 메서드는 "두 조각을 합치면 피자의 얼마인가?"를 계산한다. 통분하면 1/2 = 3/6, 1/3 = 2/6이므로 합은 5/6. 피자의 거의 대부분(약 83.3%)을 차지한다.

> ⚠️ **함정:** `(double) num / den`에서 `(double)`은 **`num`에만** 적용된다. `(double)(num / den)`이 아니다! `(double)(5 / 6)` = `(double)(0)` = `0.0`이 되어 완전히 다른 결과가 나온다. `(double) num / den` = `5.0 / 6` = `0.833...`이 올바른 계산. 캐스팅의 우선순위를 정확히 이해해야 한다.

> 💡 **꿀팁:** 분수 덧셈 통분 공식을 외워두면 이런 문제를 빠르게 풀 수 있다:
> **`a/b + c/d = (a*d + c*b) / (b*d)`**
> 이 공식은 약분하지 않은 결과를 주지만, `getValue()`로 소수 변환하면 약분 여부에 관계없이 같은 값이 나온다. 5/6이든 10/12이든 `getValue()`는 동일한 0.833...을 반환한다.

### Q10. 정답: **(A) true**

📌 **출제 의도:** 사용자 정의 클래스의 인스턴스 변수 상태 변화를 여러 메서드 호출에 걸쳐 추적할 수 있는지 테스트. 특히 `toggle()` 메서드가 `boolean` 값을 반전시키는 패턴과, 홀수 번 토글하면 초기 상태의 반대가 된다는 논리를 이해하는지 확인

📎 **연계:** AP CSA Unit 3(Class Creation)의 기본 문제. 객체의 **상태(state)** 가 메서드 호출에 의해 변화하는 것을 추적하는 능력은 AP CSA의 근간. Q9(Fraction)와 달리 이 문제는 void 메서드(`toggle`)가 인스턴스 변수를 직접 수정하는 패턴. Q11(StringSet)에서 더 복잡한 상태 추적으로 확장됨

> **핵심 개념:** `toggle()` 메서드는 `on = !on`으로 `boolean` 값을 반전시킨다. `false → true → false → true → ...` 패턴으로 토글된다. 홀수 번 호출하면 초기 상태의 **반대**, 짝수 번 호출하면 **원래 상태**로 돌아온다.

> **선행 지식:**
> - `boolean` 타입: `true` 또는 `false`만 가능
> - `!` 연산자: `true → false`, `false → true` (논리 부정)
> - `on = !on`: 현재 값의 반대를 다시 자기 자신에 대입 → 토글(toggle)
> - 생성자에서 `on = false`로 초기화

**코드:**
```java
public class Light
{
    private boolean on;
    public Light() { on = false; }
    public void toggle() { on = !on; }
    public boolean isOn() { return on; }
}
```

```java
Light lamp = new Light();
lamp.toggle();
lamp.toggle();
lamp.toggle();
System.out.println(lamp.isOn());
```

**상태 추적:**

| 단계 | 호출 | on 변화 | on 값 |
|---|---|---|---|
| 초기 | `new Light()` | 생성자에서 `on = false` | **false** |
| 1번째 | `lamp.toggle()` | `on = !false` | **true** |
| 2번째 | `lamp.toggle()` | `on = !true` | **false** |
| 3번째 | `lamp.toggle()` | `on = !false` | **true** |
| 결과 | `lamp.isOn()` | `return on` → `true` | 출력: **true** |

3번 토글(홀수) → 초기값(`false`)의 반대 → **`true`**

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| **(A) true** | ✅ 정답 | false → true → false → true (3번 토글, 홀수 → 반대) |
| (B) false | 토글 횟수를 잘못 세거나, 2번만 토글한다고 생각한 경우. 또는 마지막 `toggle()`을 빼먹음 | 코드에 `toggle()`이 **3번** 호출됨. 홀수 번이므로 초기값의 반대 |
| (C) 컴파일 에러 | `println(lamp.isOn())`에서 `boolean`을 출력할 수 없다고 생각한 경우 | `System.out.println(boolean)`은 합법적. `true` 또는 `false`가 문자열로 변환되어 출력됨 |
| (D) 런타임 에러 | `toggle()` 호출에서 문제가 발생한다고 생각한 경우 | `on = !on`은 완전히 합법적인 boolean 연산. 런타임 오류 없음 |

**비유:** 방의 전등 스위치를 생각하자. 처음에 꺼져 있다(false). 1번 누르면 켜지고(true), 2번 누르면 꺼지고(false), 3번 누르면 다시 **켜진다(true)**. 스위치를 홀수 번 누르면 원래 상태의 반대, 짝수 번 누르면 원래 상태. 이것이 `toggle()`의 본질이다.

> ⚠️ **함정:** 토글 문제에서 가장 흔한 실수는 **횟수를 잘못 세는 것**이다. 코드를 빠르게 읽으면 `toggle()`이 몇 번 호출되는지 놓치기 쉽다. 하나하나 손가락으로 짚으며 세라!

> 💡 **꿀팁:** 토글 문제를 만나면 **호출 횟수의 홀짝만 판단**하면 된다:
> - **홀수 번 토글** → 초기 상태의 **반대** (false였으면 true)
> - **짝수 번 토글** → **초기 상태와 동일** (false였으면 false)
> 이 규칙을 알면 100번 토글한 결과도 즉시 알 수 있다. 100은 짝수 → 초기 상태(false). 99번이면 홀수 → true.

### Q11. 정답: **(B) true / true / false / 2**

📌 **출제 의도:** 사용자 정의 클래스가 내부 `ArrayList`를 캡슐화하고, **for-loop + equals**로 직접 검색하는 패턴을 추적할 수 있는지 테스트. AP CSA의 ArrayList 6개 메서드(`size`, `add(E)`, `add(int,E)`, `get(int)`, `set(int,E)`, `remove(int)`)만 사용해 "값으로 검색"을 구현한 것이 핵심. `contains`나 `indexOf`를 직접 쓰지 않고 동일 기능을 만드는 패턴은 AP FRQ의 단골.

📎 **연계:** AP CSA Unit 3(Class Creation)의 심화 + Unit 4(Data Collections)의 ArrayList 활용. Q10(Light) 단순 상태 추적에서 발전. AP의 "중복 없는 컬렉션(set)" 패턴은 FRQ에서 자주 출제 — 특히 `addItem`이나 `contains`-style 메서드를 직접 구현하라는 문제. 본 문제의 핵심은 **메서드의 반환값**과 **내부 상태**를 동시에 추적하는 것.

> **핵심 개념: for-loop 직접 검색 패턴**
> AP의 ArrayList 6개 메서드에는 `contains`가 없으므로, 값으로 검색하려면 **수동 루프**로 구현:
> ```java
> for (int i = 0; i < items.size(); i++) {
>     if (items.get(i).equals(s)) return true;  // 발견
> }
> return false;  // 못 찾음
> ```
> `add` 메서드의 동작:
> - 루프에서 일치하는 원소 발견 → 즉시 `return false` (이미 있으니 추가 안 함)
> - 루프 끝까지 못 찾음 → `items.add(s)` 후 `return true`

> **선행 지식:**
> - AP의 ArrayList 메서드 6개만 사용 가능 (`contains`는 범위 밖)
> - String 비교는 항상 `.equals()` 사용 (== 금지)
> - 조건부 `return`: 메서드 중간에 return하면 즉시 종료
> - 빈 ArrayList에서 for-loop는 0번 실행되어 자연스럽게 "못 찾음"으로 진행

**코드:**
```java
public class StringSet
{
    private ArrayList<String> items;
    public StringSet() { items = new ArrayList<String>(); }

    public boolean add(String s)
    {
        for (int i = 0; i < items.size(); i++)
        {
            if (items.get(i).equals(s)) return false;  // 발견 → 추가 안 함
        }
        items.add(s);
        return true;
    }

    public int size() { return items.size(); }
}
```

```java
StringSet set = new StringSet();
System.out.println(set.add("a"));
System.out.println(set.add("b"));
System.out.println(set.add("a"));
System.out.println(set.size());
```

**호출별 상세 추적:**

| 호출 | items (호출 전) | for-loop 결과 | 동작 | 반환값 | items (호출 후) |
|---|---|---|---|---|---|
| `set.add("a")` | `[]` (size=0) | 루프 0번 실행 (못 찾음) | `items.add("a")` 후 return true | **true** | `["a"]` |
| `set.add("b")` | `["a"]` (size=1) | i=0: "a".equals("b")? no → 루프 끝 | `items.add("b")` 후 return true | **true** | `["a", "b"]` |
| `set.add("a")` | `["a", "b"]` (size=2) | i=0: "a".equals("a")? **yes** → return false | 추가 안 함 | **false** | `["a", "b"]` 그대로 |
| `set.size()` | `["a", "b"]` (size=2) | — | size 반환 | **2** | `["a", "b"]` |

**시각화 (for-loop 검색 흐름):**

```
세 번째 add("a") 호출 시:
  i=0  items.get(0)="a"  "a".equals("a")?  ✅ TRUE  → return false (즉시 종료)
  ──────────────────────────────────────
  (i=1 이후 코드는 실행되지 않음)
  (items.add(s) 도 실행 안 됨)
  
결과: items 그대로 ["a", "b"], 반환값 false
```

**출력:**
```
true
true
false
2
```

**오답 분석:**

| 보기 | 출력 | 왜 틀린가 |
|---|---|---|
| (A) true/true/true/3 | 모든 add가 성공한다고 착각 — 중복 검사 무시 | for-loop의 첫 일치 시 `return false` → 추가 안 함 → size 증가 안 함 |
| **(B) true/true/false/2** ✅ | 정답 | for-loop이 "a"를 찾아 즉시 return false; size는 2 그대로 |
| (C) false/false/false/0 | 모든 add가 실패한다고 착각 — 빈 리스트에서도 false 반환한다고 오해 | 빈 리스트에서 for-loop은 0번 실행 → return false 안 함 → 자연스럽게 add(s) 후 true 반환 |
| (D) true/true/false/3 | 반환값은 맞지만 size를 3이라고 착각 | `return false`로 메서드가 종료되면 `items.add(s)` 자체가 실행 안 됨 → size 변화 없음 |

**비유:** 호텔 체크인을 직접 처리하는 매니저를 생각하자. 손님이 오면 **명부 처음부터** 한 줄씩 훑으면서 같은 이름이 있는지 확인한다. 같은 이름을 발견하는 즉시 "이미 등록됨"이라고 거절하고 더 이상 명부를 보지 않는다(early return). 끝까지 봤는데 없으면 새로 등록한다. `contains` 메서드는 매니저 머릿속에 자동으로 있는 것이 아니라, **직접 한 줄씩 비교**해야 하는 수작업.

> ⚠️ **함정 1:** for-loop의 종료 조건은 `i < items.size()`. items가 비어 있으면 `0 < 0` = false → 루프 진입 안 함. 첫 번째 `add("a")` 호출 시 이 점이 중요 — 빈 리스트라서 루프는 즉시 종료, 그 다음 `items.add(s)`로 자연스럽게 추가.

> ⚠️ **함정 2:** `return false`는 메서드를 **즉시 종료**한다. 그 뒤의 `items.add(s)`는 실행되지 않는다. 따라서 중복인 경우 items의 size는 변하지 않음. **반환값과 size를 분리해서 추적**하는 습관이 중요.

> ⚠️ **함정 3:** AP CSA에서 ArrayList의 `contains`, `indexOf`, `isEmpty` 등은 **범위 밖**. 시험에서 이런 메서드를 호출하는 코드를 보면 의심해야 한다. 본 문제처럼 6개 메서드만 사용한 코드가 정통 AP 스타일.

> 💡 **꿀팁 1:** "값으로 검색" 패턴 5줄 관용구를 외워두자:
> ```java
> for (int i = 0; i < list.size(); i++) {
>     if (list.get(i).equals(target)) return true;
> }
> return false;
> ```
> contains 없이도 같은 결과를 만들 수 있다. FRQ에서도 자주 등장.

> 💡 **꿀팁 2:** 메서드의 반환값과 내부 상태를 **표로 분리**해서 추적하자. 컬럼: (호출, 호출 전 상태, 동작, 반환값, 호출 후 상태). 한 줄로 묶어 적으면 실수가 줄어든다.

> 💡 **꿀팁 3:** 이 코드는 Java의 `Set` 인터페이스를 단순화한 버전. AP에서는 `Set`을 직접 다루지 않지만, ArrayList로 set 동작을 흉내내는 패턴은 자주 출제. "중복 없이 추가"라는 요구사항이 보이면 즉시 이 패턴을 떠올리자.

### Q12. 정답: **(C) 15**

📌 **출제 의도:** 배열의 원소를 순차적으로 수정하는 루프에서, 이전 반복의 수정 결과가 다음 반복에 영향을 미치는 **누적 효과(cumulative effect)** 를 정확히 추적할 수 있는지 테스트. 이것은 **누적합(prefix sum / running sum)** 알고리즘의 핵심

📎 **연계:** AP CSA Unit 4(Data Collections) — 1D 배열의 핵심 문제. 단순히 배열을 읽는 것이 아니라, 루프 안에서 배열을 **수정**하면서 이전 수정 결과가 현재 수정에 영향을 주는 패턴. 이 패턴은 AP CSA FRQ에서 "in-place 배열 변환" 문제로 자주 출제되며, 실수하기 가장 쉬운 유형 중 하나. Q13의 배열 인덱싱과 결합하여 배열 조작의 기초를 형성

> **핵심 개념:** `arr[i] += arr[i - 1]`은 "현재 원소에 이전 원소의 값을 더한다"는 의미이다. 하지만 `arr[i - 1]`은 이미 이전 반복에서 수정된 값이므로, 원래의 `arr[i - 1]`이 아니라 **누적된 값**이 더해진다. 이것이 바로 누적합(prefix sum) 알고리즘이다.

> **선행 지식:**
> - 배열은 **참조 타입**: 배열 원소를 수정하면 원본 배열이 직접 바뀜
> - `arr[i] += arr[i-1]`은 `arr[i] = arr[i] + arr[i-1]`의 축약형
> - **누적합(prefix sum)**: arr[i]를 "인덱스 0부터 i까지의 원래 원소들의 합"으로 변환하는 알고리즘
> - 이전 반복의 결과가 다음 반복의 입력이 되는 **체이닝 효과**

**코드:**
```java
int[] arr = {1, 2, 3, 4, 5};
for (int i = 1; i < arr.length; i++)
{
    arr[i] += arr[i - 1];
}
System.out.println(arr[4]);
```

**반복별 상세 추적 (가장 중요!):**

| 반복 | i | arr[i-1] (이미 수정된!) | arr[i] (수정 전) | arr[i] += arr[i-1] | arr 전체 상태 |
|---|---|---|---|---|---|
| 초기 | - | - | - | - | `{1, 2, 3, 4, 5}` |
| 1회 | 1 | arr[0] = **1** (원본) | arr[1] = 2 | `2 + 1 = 3` | `{1, **3**, 3, 4, 5}` |
| 2회 | 2 | arr[1] = **3** (수정됨!) | arr[2] = 3 | `3 + 3 = 6` | `{1, 3, **6**, 4, 5}` |
| 3회 | 3 | arr[2] = **6** (수정됨!) | arr[3] = 4 | `4 + 6 = 10` | `{1, 3, 6, **10**, 5}` |
| 4회 | 4 | arr[3] = **10** (수정됨!) | arr[4] = 5 | `5 + 10 = 15` | `{1, 3, 6, 10, **15**}` |

최종 배열: `{1, 3, 6, 10, 15}`
`arr[4]` = **15**

**왜 누적합인가?**
각 arr[i]의 최종값은 원래 배열의 0번째부터 i번째 원소까지의 **합**이다:
- arr[0] = 1 = 1
- arr[1] = 1+2 = 3
- arr[2] = 1+2+3 = 6
- arr[3] = 1+2+3+4 = 10
- arr[4] = 1+2+3+4+5 = **15**

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) 5 | 배열이 수정되지 않는다고 생각한 경우. 원래 `arr[4] = 5`를 그대로 답함 | 루프 안에서 `arr[i] += arr[i-1]`로 배열이 **직접 수정**됨. 루프 후의 arr[4]는 원래 값이 아님 |
| (B) 10 | `arr[i-1]`이 **수정된** 값이 아니라 **원래** 값이라고 착각한 경우. arr[4] = 5 + arr[3]의 원래값(4) + arr[2]의 원래값(3)...이 아니라 단순히 arr[4] = 5 + 원래arr[3](4) + ... = 5+4+...이라고 부분적으로 계산 | **핵심 실수**: `arr[i-1]`은 이미 이전 반복에서 수정된 값! arr[3]은 원래 4가 아니라 `10`(누적된 값)이다. 이 점을 놓치면 10이나 9 같은 오답이 나옴 |
| **(C) 15** | ✅ 정답 | 누적합: 1+2+3+4+5 = 15 |
| (D) 9 | 누적 효과를 부분적으로만 적용한 경우. 또는 루프 범위를 잘못 판단하여 일부 반복만 수행 | 루프는 `i=1`부터 `i=4`까지 **4번** 반복. 모든 반복을 정확히 추적해야 함 |

**비유:** 눈덩이를 굴린다고 상상하자. 처음에 작은 눈덩이(1)가 있고, 언덕을 내려가면서 눈(2, 3, 4, 5)을 차례로 모은다. 하지만 각 지점에서 모이는 눈은 "그 지점의 눈 + 이미 굴러온 눈덩이 전체"가 된다. 마지막 지점에서는 1+2+3+4+5 = 15만큼의 눈이 쌓인다. `arr[i] += arr[i-1]`에서 `arr[i-1]`이 "이미 굴러온 눈덩이 전체"에 해당한다.

> ⚠️ **함정:** 이 문제의 **핵심 함정**은 `arr[i-1]`이 **원래 값이 아니라 이미 수정된 값**이라는 점이다. 많은 학생이 arr[i] += arr[i-1]에서 arr[i-1]이 원래 배열의 값이라고 생각하여 오답을 낸다. 예를 들어 i=2일 때 `arr[1]`은 원래의 2가 아니라, 이전 반복에서 수정된 **3**(= 1+2)이다! 이 누적 효과를 놓치면 절대로 정답에 도달할 수 없다. **반드시 각 반복 후의 배열 전체 상태를 기록하면서 추적하라!**

> 💡 **꿀팁:** 누적합(prefix sum) 패턴을 인식하는 법:
> - 코드 패턴: `arr[i] += arr[i-1]` 또는 `arr[i] = arr[i] + arr[i-1]`
> - 의미: 각 원소가 "자기 자신 + 왼쪽 이웃"이 됨 → 최종적으로 각 원소가 "0부터 자기까지의 합"이 됨
> - 빠른 계산: 원래 배열의 합을 구하면 됨! `1+2+3+4+5 = 15`
> - AP 시험에서 이 패턴이 보이면, 배열 전체를 한 줄씩 추적하는 것이 가장 안전하지만, 마지막 원소의 값이 "원래 배열 전체의 합"이라는 사실을 알면 검산에 활용할 수 있다.

### Q13. 정답: **(D) 40**

📌 **출제 의도:** 배열의 인덱싱에서 `arr.length - 1`이 마지막 원소의 인덱스임을 정확히 이해하고, 배열의 첫 원소와 마지막 원소에 접근하여 뺄셈을 수행할 수 있는지 테스트

📎 **연계:** AP CSA Unit 4(Data Collections)의 기본 문제이면서, 배열 인덱싱의 가장 기초적인 규칙을 확인하는 문제. `arr.length`는 배열의 크기(= 원소 개수)이고 `arr.length - 1`은 마지막 원소의 인덱스라는 사실은 1D 배열, ArrayList, 2D 배열 등 데이터 컬렉션을 사용하는 모든 곳에서 필수. Q12의 배열 수정 문제와 함께 배열 조작의 기초를 구성

> **핵심 개념:** Java 배열의 인덱스는 `0`부터 `length - 1`까지이다. `arr[0]`은 **첫 번째** 원소, `arr[arr.length - 1]`은 **마지막** 원소를 가리킨다. `arr.length`는 배열의 크기(원소 개수)를 반환하는 **속성**(메서드가 아니므로 괄호 없음).

> **선행 지식:**
> - 배열 인덱스: 0부터 시작 → 첫 원소 = `arr[0]`, 마지막 원소 = `arr[arr.length - 1]`
> - `arr.length`: 배열의 원소 개수 (여기서는 5)
> - `arr.length - 1`: 마지막 원소의 인덱스 (여기서는 4)
> - **주의:** `arr[arr.length]`는 `ArrayIndexOutOfBoundsException` 발생! (인덱스는 0~4인데 5를 접근)

**코드:**
```java
int[] arr = {10, 20, 30, 40, 50};
System.out.println(arr[arr.length - 1] - arr[0]);
```

**인덱스 맵핑:**
```
원소:   10   20   30   40   50
인덱스:  0    1    2    3    4
        ^                   ^
      arr[0]           arr[4] = arr[arr.length-1]
```

**한 줄씩 추적:**

| 단계 | 실행 내용 | 계산 과정 | 결과 |
|---|---|---|---|
| 1 | `int[] arr = {10, 20, 30, 40, 50};` | 배열 생성, 길이 5 | arr = {10, 20, 30, 40, 50} |
| 2 | `arr.length` | 배열 원소 개수 | 5 |
| 3 | `arr.length - 1` | `5 - 1` = 4 | 4 (마지막 인덱스) |
| 4 | `arr[arr.length - 1]` | `arr[4]` = **50** | 50 |
| 5 | `arr[0]` | **10** | 10 |
| 6 | `arr[4] - arr[0]` | `50 - 10` = **40** | 40 |
| 7 | `println(...)` | **40** 출력 | 출력: 40 |

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) 0 | `arr[arr.length - 1] - arr[0]`을 `arr[0] - arr[0]`이라고 잘못 계산한 경우. 또는 두 값이 같다고 착각 | `arr.length - 1 = 4`이므로 `arr[4] = 50`. `arr[0] = 10`. `50 - 10 = 40` |
| (B) 10 | `arr[1] - arr[0]` = 20 - 10 = 10으로 계산한 경우. `arr.length - 1`을 `1`이라고 잘못 계산했거나, 인접 원소의 차이를 구한다고 착각 | `arr.length = 5`이므로 `arr.length - 1 = 4`. 마지막 원소는 `arr[4] = 50` |
| (C) 50 | `arr[arr.length - 1]`의 값(50)만 출력된다고 생각한 경우. `- arr[0]` 부분을 빼먹음 | 전체 식은 `arr[4] - arr[0]` = `50 - 10` = **40**. 뺄셈을 잊으면 안 됨 |
| **(D) 40** | ✅ 정답 | `arr[4] - arr[0]` = `50 - 10` = 40 |

**비유:** 5일간의 주식 가격이 {10, 20, 30, 40, 50}이라고 하자. `arr[arr.length - 1] - arr[0]`은 "마지막 날 가격 - 첫 날 가격" = **총 변동폭**을 구하는 것이다. 50 - 10 = 40원의 가격 상승이 있었다. 이 "범위(range) 계산"은 통계에서 자주 사용되는 패턴이다.

> ⚠️ **함정:** 가장 흔한 실수는 `arr.length`와 `arr.length - 1`을 혼동하는 것이다. `arr.length`는 **원소 개수**(5)이고, 마지막 원소의 **인덱스**는 `arr.length - 1`(4)이다. 만약 `arr[arr.length]`(= `arr[5]`)에 접근하면 `ArrayIndexOutOfBoundsException`이 발생한다! 이것은 AP CSA에서 가장 빈번한 런타임 오류이므로, 항상 **"인덱스 = 길이 - 1"** 규칙을 떠올리라.

> 💡 **꿀팁:** 배열의 첫/마지막 원소 접근 패턴을 외워두자:
> - **첫 원소:** `arr[0]` — 항상 인덱스 0
> - **마지막 원소:** `arr[arr.length - 1]` — 길이에서 1을 빼야 함!
> - **범위(range):** `arr[arr.length - 1] - arr[0]` — 최댓값 - 최솟값 (정렬된 배열에서)
> 이 패턴은 AP FRQ에서 배열을 순회하는 거의 모든 문제의 출발점이다.

### Q14. 정답: **(C) 7**

📌 **출제 의도:** 2D 배열의 **각 행마다 두 번 순회**하는 패턴을 추적할 수 있는지 테스트. 첫 번째 패스로 행의 최댓값을 찾고, 두 번째 패스로 그 최댓값과 같은 모든 셀의 개수를 카운트하는 **two-pass per row** 알고리즘. 단순히 max만 찾는 것이 아니라 **max가 여러 번 등장**할 때 모두 세는 점이 핵심 — "duplicates of max" 패턴은 AP에서 자주 출제.

📎 **연계:** Unit 4(Data Collections)의 2D 배열 + 통계 응용. LT3 Q23(2D max + 위치 추적), LT9 Q14(second largest), LT10 Q14(mode)와 같은 "각 원소를 다른 원소와 비교"하는 패턴 계열. 이중 루프 안에 다시 두 번의 행별 처리가 들어가는 **3중 중첩** 구조라 트레이스 실수가 많음.

> **핵심 개념: Two-pass per row 패턴**
> 한 행을 한 번만 훑어서는 "max 등장 횟수"를 알 수 없다. 왜냐하면 max는 마지막에야 확정되기 때문. 따라서:
> - **1차 패스**: 행 전체 훑어서 max 결정
> - **2차 패스**: 행 전체 훑어서 max와 같은 셀 카운트
> 
> 외부 루프(행)와 내부 두 루프(max 찾기 + 카운트)가 합쳐져 시간 복잡도 O(rows × 2cols) = O(n²)에 가깝다.

> **선행 지식:**
> - `mat.length` = 행 개수 (3)
> - `mat[r].length` = r행의 열 개수 (4)
> - 직사각형 행렬(rectangular)이면 모든 행이 같은 길이
> - max 초기화는 항상 첫 원소(`mat[r][0]`)로 시작
> - `==` 비교 시 max와 같은 모든 셀 카운트 (한 행에 여러 개일 수 있음)

**코드:**
```java
int[][] mat = {{3, 7, 7, 2},
               {5, 8, 4, 8},
               {2, 6, 6, 6}};
int totalMaxPositions = 0;
for (int r = 0; r < mat.length; r++) {
    int rowMax = mat[r][0];
    for (int c = 1; c < mat[r].length; c++) {
        if (mat[r][c] > rowMax) rowMax = mat[r][c];
    }
    for (int c = 0; c < mat[r].length; c++) {
        if (mat[r][c] == rowMax) totalMaxPositions++;
    }
}
System.out.println(totalMaxPositions);
```

**행렬 시각화:**

```
        c=0  c=1  c=2  c=3     rowMax  ==max인 셀 수
       ┌────┬────┬────┬────┐
r=0    │  3 │  7 │  7 │  2 │     7      2 (c=1, 2)
       ├────┼────┼────┼────┤
r=1    │  5 │  8 │  4 │  8 │     8      2 (c=1, 3)
       ├────┼────┼────┼────┤
r=2    │  2 │  6 │  6 │  6 │     6      3 (c=1, 2, 3)
       └────┴────┴────┴────┘
                              총합:    2+2+3 = 7
```

**행별 상세 추적:**

| 행 | 1차 패스 (max 찾기) | rowMax | 2차 패스 (== max 카운트) | 행별 추가 | totalMaxPositions |
|---|---|---|---|---|---|
| 0 | start=3, c=1: 7>3 → rowMax=7. c=2: 7>7 no. c=3: 2>7 no | 7 | c=0:3≠7. c=1:7=7 ✓. c=2:7=7 ✓. c=3:2≠7 | +2 | **2** |
| 1 | start=5, c=1: 8>5 → rowMax=8. c=2: 4>8 no. c=3: 8>8 no | 8 | c=0:5≠8. c=1:8=8 ✓. c=2:4≠8. c=3:8=8 ✓ | +2 | **4** |
| 2 | start=2, c=1: 6>2 → rowMax=6. c=2: 6>6 no. c=3: 6>6 no | 6 | c=0:2≠6. c=1:6=6 ✓. c=2:6=6 ✓. c=3:6=6 ✓ | +3 | **7** |

**최종 출력: `totalMaxPositions = 7`**

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) `3` | 3 | "행 개수"라고 착각. 또는 "max 1개씩만 카운트"한 경우 (각 행 max 1번씩 = 3) |
| (B) `5` | 5 | 행별 카운트를 잘못 계산. 어느 행에서 1개 빠뜨린 경우 |
| **(C) `7`** ✅ | 7 | 정답 — 2 + 2 + 3 = 7 |
| (D) `12` | 12 | 전체 셀 개수(3×4=12). max 비교를 무시하고 모든 셀을 센 경우. 또는 max를 매번 카운트한다고 잘못 추적 |

**비유:** 3개의 농구팀이 각각 4번의 경기를 치렀고, 각 행은 한 팀의 경기별 점수. 각 팀의 **최고 점수**(rowMax)를 먼저 찾고, 그 점수와 **같은 점수를 기록한 경기**(반복 등장 가능)를 모두 카운트한다. 한 팀이 최고 기록을 여러 번 달성했으면 모두 센다. 세 팀 합계는 7번.

> ⚠️ **함정 1:** **max를 한 번만 카운트**하는 실수. 각 행의 max가 여러 번 등장할 수 있다. 예: 행 2의 `{2,6,6,6}`에서 max=6이 3번 등장 → 3번 모두 카운트. "1번씩만"이라고 잘못 가정하면 답이 (A) 3.

> ⚠️ **함정 2:** **두 번째 패스의 시작 인덱스**. `for (int c = 0; ...)` (인덱스 0부터)이지 `c = 1`부터가 아니다. 첫 번째 패스에서 `c=1`부터 시작한 것은 max 초기값을 `mat[r][0]`으로 설정했기 때문. 두 번째 패스는 c=0도 비교 대상.

> ⚠️ **함정 3:** rowMax는 매 행마다 **재초기화**된다. `int rowMax = mat[r][0];`이 외부 루프 안에 있다는 점이 핵심. 만약 외부 루프 밖에 두면 이전 행의 max가 다음 행 비교에 영향을 줘서 틀린다.

> 💡 **꿀팁 1:** "max + 등장 횟수" 패턴은 단일 패스로 풀기 어렵다. 항상 **2-pass** 패턴 사용:
> - Pass 1: max 결정
> - Pass 2: max와 같은 셀 카운트
> 1D 배열에도 같은 패턴 적용 가능.

> 💡 **꿀팁 2:** 행렬 시각화 + 행별 분석 표를 먼저 그리자. 12셀을 한 번에 추적하면 실수 확률 80%. 행별로 (rowMax, count)를 적으면 명확.

> 💡 **꿀팁 3:** AP의 2D 문제는 항상 (1) 무엇을 찾는가? (2) 어떤 인덱스로 순회하는가? (3) 행/열 길이 정확히 사용했는가? 세 가지를 검토. 본 문제는 max를 두 번 다른 목적으로 사용하므로 의도를 명확히 분리해야 함.

### Q15. 정답: **(B) 2**

📌 **출제 의도:** `ArrayList`에 대한 **enhanced for-loop** + **조건부 카운트** 패턴을 추적할 수 있는지 테스트. AP CSA의 ArrayList 6개 메서드(`size`, `add(E)`, `add(int,E)`, `get(int)`, `set(int,E)`, `remove(int)`)와 함께, `for (int v : list)` 형태의 향상된 for문을 자연스럽게 사용하는 능력을 확인. 단순한 카운트 문제로 보이지만, **`>` vs `>=` 경계 처리**와 **카운터 초기화/증가**의 정확성이 핵심.

📎 **연계:** AP CSA Unit 4(Data Collections)의 ArrayList 트래버설 + Unit 2(Selection)의 조건문 결합. LT1 Q12(enhanced for sum), LT2 Q13(enhanced for count)와 같은 패턴. AP에서 가장 흔한 "특정 조건 만족 원소 카운트" 관용구의 ArrayList 버전.

> **핵심 개념: enhanced for-loop on ArrayList**
> `for (int v : list)`는 ArrayList의 모든 원소를 처음부터 끝까지 순회한다. `v`는 각 원소의 **값 복사본**(int이므로 primitive 복사). 인덱스 없이 값만 필요할 때 가장 깔끔한 패턴.
> 카운트 패턴:
> ```java
> int count = 0;
> for (int v : list) {
>     if (조건) count++;
> }
> ```

> **선행 지식:**
> - `ArrayList<Integer>`의 enhanced for-loop은 자동으로 `Integer`를 `int`로 unboxing
> - `int v` 변수에 각 원소가 한 번씩 들어옴 (순서: 인덱스 0 → 1 → 2 → ...)
> - `>` 와 `>=` 차이: `>`는 **초과**(같은 값 제외), `>=`는 **이상**(같은 값 포함)
> - 카운터는 반드시 0으로 초기화 후 조건 만족 시 증가

**코드:**
```java
ArrayList<Integer> list = new ArrayList<Integer>();
list.add(15);
list.add(10);
list.add(25);
list.add(30);
int count = 0;
for (int v : list)
{
    if (v > 20) count++;
}
System.out.println(count);
```

**한 줄씩 추적:**

| 단계 | 동작 | list 상태 | count |
|---|---|---|---|
| 1 | `list.add(15)` | `[15]` | 0 |
| 2 | `list.add(10)` | `[15, 10]` | 0 |
| 3 | `list.add(25)` | `[15, 10, 25]` | 0 |
| 4 | `list.add(30)` | `[15, 10, 25, 30]` | 0 |
| 5 | for 시작: v=15 | — | 15>20? false → no change | 0 |
| 6 | v=10 | — | 10>20? false | 0 |
| 7 | v=25 | — | 25>20? **true** → count++ | **1** |
| 8 | v=30 | — | 30>20? **true** → count++ | **2** |
| 9 | 루프 종료, println | — | — | 출력: **2** |

**시각화:**

```
list:   [15,  10,  25,  30]
조건:    ✗   ✗    ✓    ✓     (>20)
count:   0   0    1    2

최종 count = 2
```

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) `1` | 1 | 25, 30 중 하나만 셈 (예: 25만 carry over). 카운트 누적 실수 |
| **(B) `2`** ✅ | 2 | 정답 — 25와 30 둘 다 20 초과 |
| (C) `3` | 3 | `>= 20`으로 잘못 해석 → 15보다 큰 모든 값(10 제외)을 셈. 또는 "15 >= 20"을 잘못 판정 |
| (D) `4` | 4 | 모든 원소를 카운트. if 조건을 무시하고 size를 답한 경우 |

**비유:** 놀이공원에서 키 측정. 4명이 줄을 섰다. 키가 **20cm 초과**(20cm 정확히는 안 됨)인 손님만 입장 가능. 키 15, 10, 25, 30 → 25cm와 30cm 두 명만 입장. 통과한 인원 = 2.

> ⚠️ **함정 1:** **`>` vs `>=` 구분.** "20보다 큰"은 strict `>` (20 자체는 제외). "20 이상"은 `>=` (20 포함). 코드에 `>20`이 있으니 정확히 20인 값이 있어도 카운트 안 함. 본 문제는 20이 없으므로 `>=`로 잘못 읽어도 답은 같지만, 일반적으로 두 연산자를 구분해야 함.

> ⚠️ **함정 2:** **카운터 초기화 위치.** `int count = 0;`이 for 루프 **밖**에 있어야 한다. for 루프 안에 두면 매 반복마다 0으로 리셋되어 결과는 항상 0 또는 1.

> ⚠️ **함정 3:** **enhanced for-loop의 변수는 복사본.** `for (int v : list)`에서 `v`를 변경해도 원본 list는 영향 없음. 본 문제는 `v`를 읽기만 하므로 안전하지만, "원본 수정"이 필요한 경우 `for (int i = 0; i < list.size(); i++)` 인덱스 기반으로 바꿔야 함.

> 💡 **꿀팁 1:** "조건 만족 원소 카운트"는 AP의 단골. 관용구:
> ```java
> int count = 0;
> for (int v : list) {
>     if (조건) count++;
> }
> ```
> 5줄로 외워두면 다양한 변형(>, <, ==, %, etc.)에 즉시 적용 가능.

> 💡 **꿀팁 2:** AP의 ArrayList 6개 메서드 외에 **enhanced for-loop**도 ArrayList에서 사용 가능. 인덱스가 필요 없으면 항상 enhanced for가 더 깔끔.

> 💡 **꿀팁 3:** 추적할 때 **표를 그려라**. (단계, v 값, 조건 결과, count 누적) 4개 컬럼이면 충분. 머릿속으로만 추적하면 4개 원소도 실수 가능.

### Q16. 정답: **(B) `"ant"`**

📌 **출제 의도:** ArrayList<String>에서 **사전순(lexicographic) 최솟값**을 찾는 **선형 탐색(linear search)** 패턴을 추적할 수 있는지 테스트. `String.compareTo`의 부호 의미 — 음수면 첫 인자가 사전순으로 앞 — 와 "현재까지 최솟값을 갱신"하는 누적 추적 패턴이 핵심.

📎 **연계:** Unit 4(Data Collections)의 ArrayList 트래버설 + Unit 1(Using Objects)의 String compareTo. LT3 Q23(2D max + 위치), LT9 Q14(second largest), LT10 Q23(maximin)의 "최솟값/최댓값 추적" 패턴 계열. 정수 max/min 패턴과 동일한 구조에서 비교 연산만 `<`/`>`에서 `compareTo()`로 바뀐 형태로, AP에서 자주 출제.

> **핵심 개념: compareTo의 부호 + 선형 탐색**
> `str1.compareTo(str2)` 반환값:
> - **음수**: str1이 사전순으로 **앞** (str1 < str2)
> - **0**: 같음
> - **양수**: str1이 사전순으로 **뒤** (str1 > str2)
> 
> "최솟값 찾기" 관용구:
> ```java
> String result = list.get(0);  // 첫 원소로 초기화
> for (int i = 1; ...) {
>     if (list.get(i).compareTo(result) < 0) result = list.get(i);
> }
> ```
> `< 0`이면 "현재 원소가 result보다 앞 → 더 작음 → 갱신".

> **선행 지식:**
> - String.compareTo는 첫 글자부터 비교, 다르면 그 글자의 유니코드 차로 부호 결정
> - 알파벳 순서: a < b < c < d < ... < z (소문자 기준)
> - `"ant".compareTo("dog")`: 'a'(97) - 'd'(100) = **음수** → "ant" < "dog"
> - 첫 원소를 result 초기값으로 → 인덱스 1부터 순회
> - 더 작은 게 발견되면 result 갱신

**코드:**
```java
ArrayList<String> words = new ArrayList<String>();
words.add("dog");
words.add("ant");
words.add("bee");
words.add("cat");

String result = words.get(0);
for (int i = 1; i < words.size(); i++)
{
    if (words.get(i).compareTo(result) < 0)
    {
        result = words.get(i);
    }
}
System.out.println(result);
```

**한 줄씩 추적:**

| i | words.get(i) | result (비교 전) | compareTo 결과 | < 0? | result (비교 후) |
|---|---|---|---|---|---|
| 시작 | — | "dog" (초기화: words.get(0)) | — | — | "dog" |
| 1 | "ant" | "dog" | "ant".compareTo("dog") = 'a'(97)-'d'(100) = **음수** | ✅ | **"ant"** (갱신) |
| 2 | "bee" | "ant" | "bee".compareTo("ant") = 'b'(98)-'a'(97) = **양수** | ❌ | "ant" (변화 없음) |
| 3 | "cat" | "ant" | "cat".compareTo("ant") = 'c'(99)-'a'(97) = **양수** | ❌ | "ant" (변화 없음) |

**최종 출력: `"ant"`**

**시각화 (사전순 위치):**

```
words:  ["dog", "ant", "bee", "cat"]
사전순:    4    1     2     3
                ↑
             가장 앞 = 최솟값 = "ant"

선형 탐색 진행:
  start: result="dog" (idx 0)
  i=1:   "ant" < "dog" → result="ant" ★
  i=2:   "bee" > "ant" → no change
  i=3:   "cat" > "ant" → no change
  
최종 result = "ant"
```

**오답 분석:**

| 보기 | 값 | 왜 틀린가 |
|---|---|---|
| (A) `"dog"` | "dog" | 첫 원소를 그대로 답한 경우. 알고리즘이 더 작은 값을 찾으면 갱신한다는 사실을 놓침 |
| **(B) `"ant"`** ✅ | "ant" | 정답 — 4개 단어 중 사전순으로 가장 앞 |
| (C) `"bee"` | "bee" | "현재까지 본 두 번째로 작은 값"으로 잘못 추적. compareTo 부호 헷갈림 |
| (D) `"cat"` | "cat" | 마지막 원소를 답한 경우. 또는 `< 0` 대신 `> 0`으로 비교했다고 착각 (그래도 결과는 다름) |

**비유:** 영어 사전을 만들 때 4개 단어(dog, ant, bee, cat)를 알파벳 순으로 정렬한다고 하자. 가장 앞에 와야 할 단어를 찾아야 한다. dog부터 시작해 비교: "ant가 dog보다 앞"이니 ant 선택. 그 다음 bee와 비교: "bee가 ant보다 뒤"이니 ant 유지. cat과 비교: "cat이 ant보다 뒤"이니 ant 유지. 최종 1등은 **ant**.

> ⚠️ **함정 1:** `compareTo()`의 부호 반대로 해석. **"앞의 문자열이 더 작으면 음수"**로 외우자. `"ant".compareTo("dog")` < 0이라는 것은 "ant가 사전순으로 dog보다 앞"이라는 뜻. 부호 헷갈리면 답이 (D) "cat"이 됨 (반대로 max를 찾게 됨).

> ⚠️ **함정 2:** result 초기화를 빠뜨리거나 잘못된 값으로 시작. `String result = words.get(0);`이 핵심 — 첫 원소를 후보로 시작해야 i=1부터 순회 가능. null로 초기화하면 첫 비교에서 NullPointerException.

> ⚠️ **함정 3:** for 루프의 시작 인덱스 `i = 1`. 인덱스 0은 이미 result로 설정했으므로 비교할 필요 없다. `i = 0`으로 시작하면 자기 자신과 비교 → result.compareTo(result) = 0 → 갱신 안 함 → 결과는 같지만 불필요한 비교 1회.

> 💡 **꿀팁 1:** "최솟값/최댓값 찾기" 패턴은 모두 같은 구조:
> ```java
> X best = list.get(0);
> for (int i = 1; ...) {
>     if (list.get(i) [<,>] best) best = list.get(i);
> }
> ```
> Integer면 `<`/`>`, String이면 `compareTo() < 0`/`> 0`. 비교 연산자만 다름.

> 💡 **꿀팁 2:** String compareTo의 결과는 **부호(sign)**만 보면 됨. 정확한 차이값(예: -3)은 거의 사용 안 함. AP 시험에서도 "음수/0/양수" 셋 중 하나로 분류.

> 💡 **꿀팁 3:** 알파벳 순서 빠른 확인: 단어의 **첫 글자만 다르면** 첫 글자로 결과 결정. "ant" vs "dog" → 'a' vs 'd' → 'a'가 작음 → ant가 작다. 두 번째 글자는 볼 필요 없음. AP 시험에서 단어 비교 시 첫 글자만 봐도 90% 해결.

### Q17. 정답: **(C) "3 3"**

📌 **출제 의도:** Java에서 객체 변수의 **참조(reference) 복사**와 **별칭(aliasing)** 개념을 이해하는지 테스트. `ArrayList<Integer> b = a;`는 새로운 리스트를 만드는 것이 **아니라**, `a`와 `b`가 **같은 객체**를 가리키게 만든다. 이것은 AP CSA에서 가장 자주 출제되는 함정 문제 유형 중 하나이다.

📎 **연계:** Unit 4(Data Collections — ArrayList)과 Unit 1(Using Objects — 참조)의 핵심 개념. 이 문제는 Java의 참조 의미론(reference semantics)을 정확히 이해해야 풀 수 있다. 배열에서도 동일한 원리가 적용되며(`int[] b = a;`도 별칭), FRQ에서 메서드에 ArrayList를 매개변수로 전달할 때도 같은 개념이 적용된다.

> **핵심 개념:** Java에서 객체 변수(참조 변수)는 객체 자체를 저장하는 것이 아니라, 객체가 저장된 **메모리 주소(참조)**를 저장한다. `b = a`는 `a`에 저장된 참조(주소)를 `b`에 복사하는 것이므로, `a`와 `b`는 **동일한 객체**를 가리킨다. 한쪽에서 수정하면 다른 쪽에서도 변경이 보인다.

> **선행 지식:**
> - `ArrayList<Integer> b = a;`는 **shallow copy(얕은 복사)**가 아니라 **참조 복사**이다. 새 리스트가 만들어지지 않는다!
> - 새 리스트를 만들려면 `ArrayList<Integer> b = new ArrayList<Integer>(a);`처럼 `new`를 사용해야 한다
> - `a`와 `b`가 같은 객체를 가리키므로, `b.add(4)`를 하면 `a`에서도 4가 보이고, `a.remove(0)`을 하면 `b`에서도 첫 원소가 사라진다
> - `a.size()`와 `b.size()`는 같은 객체의 크기를 묻는 것이므로 항상 같은 값을 반환한다

**코드:**
```java
ArrayList<Integer> a = new ArrayList<Integer>();
a.add(1); a.add(2); a.add(3);
ArrayList<Integer> b = a;
b.add(4);
a.remove(0);
System.out.println(a.size() + " " + b.size());
```

**메모리 다이어그램으로 상세 추적:**

**1단계: 초기 상태**
```
a ──→ [ArrayList 객체: {1, 2, 3}]
```
`a`라는 참조 변수가 힙(heap) 메모리에 있는 ArrayList 객체를 가리킨다. 리스트에는 1, 2, 3이 들어 있다.

**2단계: `ArrayList<Integer> b = a;` 실행 후**
```
a ──→ [ArrayList 객체: {1, 2, 3}] ←── b
```
`b`에 `a`의 참조(주소)가 복사되었다. 이제 `a`와 `b`는 **동일한 ArrayList 객체**를 가리킨다. 새로운 ArrayList가 만들어진 것이 **아니다**! 메모리에는 여전히 ArrayList 객체가 **하나**만 존재한다.

**3단계: `b.add(4);` 실행 후**
```
a ──→ [ArrayList 객체: {1, 2, 3, 4}] ←── b
```
`b`를 통해 `add(4)`를 호출했다. `b`가 가리키는 객체(= `a`가 가리키는 객체)에 4가 추가되었다. 이 시점에서 `a`로 접근하든 `b`로 접근하든 리스트는 `{1, 2, 3, 4}`이고, 크기는 4이다.

**4단계: `a.remove(0);` 실행 후**
```
a ──→ [ArrayList 객체: {2, 3, 4}] ←── b
```
`a`를 통해 `remove(0)`을 호출했다. 인덱스 0의 원소(값 1)가 제거되었다. `a`가 가리키는 객체(= `b`가 가리키는 객체)에서 첫 번째 원소가 삭제되었으므로, 리스트는 `{2, 3, 4}`이고, 크기는 3이다.

**5단계: 출력**
`a.size()` → `a`가 가리키는 객체의 크기 → **3**
`b.size()` → `b`가 가리키는 객체의 크기 → **3** (같은 객체이므로 당연히 같다!)
출력: **"3 3"**

**오답 분석:**

**(A) "3 4"** — `a`와 `b`가 **별도의 리스트**라고 생각한 경우. `b.add(4)` 후 `b`는 크기 4가 되고, `a.remove(0)` 후 `a`는 크기 2가 되어... 아, 이 보기는 "3 4"이므로 `a`에서 remove 후 크기 3, `b`는 add 후 크기 4라고 생각한 것이다. 하지만 `a`와 `b`는 같은 객체이므로 크기가 다를 수 없다.

**(B) "2 4"** — `b = a` 시점에 리스트의 **복사본**이 만들어졌다고 생각한 경우. 이 경우 `a`는 `{1, 2, 3}` → `remove(0)` → `{2, 3}` (크기 2), `b`는 `{1, 2, 3}` → `add(4)` → `{1, 2, 3, 4}` (크기 4)가 된다. 하지만 실제로는 복사본이 만들어지지 않는다!

**(D) "2 3"** — `b.add(4)`가 `a`에도 영향을 주지만, `a.remove(0)`은 `b`에 영향을 주지 않는다고 생각한 경우. 별칭 관계는 **양방향**이다. 어느 참조를 통해 수정하든 같은 객체가 변경되므로 둘 다 영향을 받는다.

> ⚠️ **함정:** 이 문제의 핵심 함정은 `ArrayList<Integer> b = a;`가 마치 `b`에 `a`의 복사본을 만드는 것처럼 보인다는 것이다. 많은 학생들이 `b = a` 이후에 `a`와 `b`가 독립적인 리스트라고 착각한다. Java에서 `=` 연산자는 **참조를 복사**하는 것이지, **객체를 복사**하는 것이 아니다! 이것은 `int x = 5; int y = x;`와 근본적으로 다르다. 기본 타입(primitive)은 값이 복사되지만, 객체 타입(reference)은 참조가 복사된다.

> 💡 **꿀팁:** "같은 리모컨" 비유를 기억하자. `a`와 `b`는 같은 TV를 가리키는 두 개의 리모컨이다. `b`로 채널을 바꾸면 `a`로 볼 때도 채널이 바뀌어 있다. `a`로 볼륨을 내리면 `b`로 볼 때도 볼륨이 내려가 있다. 두 리모컨이 가리키는 TV는 **하나**이기 때문이다. 독립적인 TV(리스트)를 원하면 `new ArrayList<>(a)`로 새 TV를 사야 한다.

### Q18. 정답: **(B) 7**

📌 **출제 의도:** 2차원 배열(2D array)의 **행 우선 순서(row-major order)** 채우기를 이해하고, 특정 인덱스의 값을 정확히 추적할 수 있는지 테스트. 2D 배열에서 `grid[r][c]`의 `r`은 행(row), `c`는 열(column)이라는 인덱싱 규칙을 정확히 알아야 한다.

📎 **연계:** Unit 4(Data Collections — 2D 배열)의 핵심 문제. 2D 배열의 순회 패턴(행 우선 vs 열 우선)과 인덱싱은 AP CSA에서 매년 출제된다. Q19의 행렬 곱셈 문제, Q23의 2D 정렬된 행 카운트와도 연결되며, FRQ Q4(TreasureMap)에서도 2D 배열 순회가 사용된다.

> **핵심 개념:** `new int[3][4]`는 3행 4열의 2D 배열을 생성한다. `grid[r][c]`에서 `r`은 **행 인덱스**(0부터 시작), `c`는 **열 인덱스**(0부터 시작)이다. 중첩 for 루프에서 바깥 루프가 행, 안쪽 루프가 열을 순회하면 **행 우선 순서**로 채워진다.

> **선행 지식:**
> - `new int[3][4]` → 3개의 행, 각 행에 4개의 열. 총 12개의 칸
> - 인덱스는 0부터 시작: 행 0, 1, 2 / 열 0, 1, 2, 3
> - `grid[1][2]`는 **2번째 행**(인덱스 1)의 **3번째 열**(인덱스 2)

**코드:**
```java
int[][] grid = new int[3][4];
int val = 1;
for (int r = 0; r < 3; r++)
{
    for (int c = 0; c < 4; c++)
    {
        grid[r][c] = val;
        val++;
    }
}
System.out.println(grid[1][2]);
```

**한 줄씩 추적:**

`val`이 1부터 시작하여 칸을 채울 때마다 1씩 증가한다. 행 우선 순서(r=0의 모든 열 → r=1의 모든 열 → r=2의 모든 열)로 채워진다.

```
r=0 순회: grid[0][0]=1, grid[0][1]=2, grid[0][2]=3, grid[0][3]=4
r=1 순회: grid[1][0]=5, grid[1][1]=6, grid[1][2]=7, grid[1][3]=8
r=2 순회: grid[2][0]=9, grid[2][1]=10, grid[2][2]=11, grid[2][3]=12
```

완성된 2D 배열을 표로 나타내면:

```
         열0   열1   열2   열3
행0  [   1,    2,    3,    4  ]
행1  [   5,    6,    7,    8  ]
행2  [   9,   10,   11,   12  ]
```

`grid[1][2]`는 행1, 열2에 해당하는 값이므로 → **7**

**오답 분석:**

**(A) 6** — `grid[1][2]`를 행1, 열1의 값(6)으로 착각한 경우. 열 인덱스가 2이므로 세 번째 열이지, 두 번째 열이 아니다. 인덱스와 순번을 혼동하지 않도록 주의해야 한다.

**(C) 8** — `grid[1][2]`를 `grid[1][3]`의 값(8)으로 착각한 경우. 또는 val이 0이 아니라 1부터 시작하는 것을 고려하지 않고 잘못 계산한 경우이다.

**(D) 5** — `grid[1][0]`의 값(5)이다. 열 인덱스가 0이 아니라 2라는 것을 놓친 경우이다.

> ⚠️ **함정:** 2D 배열에서 가장 흔한 실수는 행과 열을 혼동하는 것이다. `grid[1][2]`에서 `1`은 **행**, `2`는 **열**이다. "1행 2열"이라고 말하면 "두 번째 행의 세 번째 열"이다 (0부터 시작하므로). 수학에서의 행렬 표기법과 동일하다.

> 💡 **꿀팁:** 빠른 계산법이 있다. `grid[r][c]`의 값은 `r × (열의 개수) + c + 1`이다 (val이 1부터 시작하는 경우). 이 문제에서 `grid[1][2]` = `1 × 4 + 2 + 1` = `4 + 2 + 1` = **7**. 이 공식을 사용하면 전체 배열을 그리지 않고도 바로 답을 구할 수 있다.

### Q19. 정답: **(D) "1 1 1"**

📌 **출제 의도:** 행렬 곱셈(matrix multiplication) 알고리즘을 코드로 구현한 것을 추적할 수 있는지 테스트. 특히 **단위 행렬(identity matrix)**의 성질을 이해하면 빠르게 풀 수 있지만, 코드를 직접 추적해도 풀 수 있어야 한다.

📎 **연계:** Unit 4(Data Collections — 2D 배열)의 심화 문제. 3중 중첩 for 루프를 사용하는 행렬 곱셈은 2D 배열의 가장 복잡한 활용 사례 중 하나이다. AP CSA 시험에서 행렬 곱셈 자체가 직접 출제되는 경우는 드물지만, 3중 루프와 2D 배열의 인덱싱을 이해하는 능력을 테스트하는 문제는 나올 수 있다.

> **핵심 개념: 단위 행렬(Identity Matrix)**
> 단위 행렬 I는 대각선 원소가 모두 1이고 나머지가 모두 0인 정사각 행렬이다.
> ```
> I = | 1  0  0 |
>     | 0  1  0 |
>     | 0  0  1 |
> ```
> 단위 행렬의 핵심 성질: **어떤 행렬 A에 대해 I × A = A × I = A**. 즉, 단위 행렬을 곱하면 원래 행렬이 그대로 나온다. 숫자에서 1을 곱하면 원래 값이 나오는 것과 같은 원리이다.
> 따라서 **I × I = I**이다. 단위 행렬에 단위 행렬을 곱하면 단위 행렬이 나온다.

> **선행 지식: 행렬 곱셈의 원리**
> 행렬 곱셈 `result[r][c]`는 첫 번째 행렬의 r행과 두 번째 행렬의 c열의 **내적(dot product)**이다.
> ```
> result[r][c] = Σ(k=0 to n-1) mat1[r][k] * mat2[k][c]
> ```
> 코드에서 3중 루프의 역할:
> - `r` 루프: 결과 행렬의 **행**을 선택
> - `c` 루프: 결과 행렬의 **열**을 선택
> - `k` 루프: 첫 번째 행렬의 r행과 두 번째 행렬의 c열의 원소들을 하나씩 곱해서 더함 (내적)

**코드:**
```java
int[][] mat = {{1, 0, 0},
               {0, 1, 0},
               {0, 0, 1}};
int[][] result = new int[3][3];
for (int r = 0; r < 3; r++)
{
    for (int c = 0; c < 3; c++)
    {
        for (int k = 0; k < 3; k++)
        {
            result[r][c] += mat[r][k] * mat[k][c];
        }
    }
}
System.out.println(result[0][0] + " " + result[1][1] + " " + result[2][2]);
```

**대각선 원소 계산 상세 추적:**

이 코드는 `mat × mat`(= I × I)를 계산한다. 문제에서 묻는 것은 `result[0][0]`, `result[1][1]`, `result[2][2]`, 즉 **대각선 원소**이다.

**result[0][0] 계산 (r=0, c=0):**
```
k=0: mat[0][0] * mat[0][0] = 1 * 1 = 1
k=1: mat[0][1] * mat[1][0] = 0 * 0 = 0
k=2: mat[0][2] * mat[2][0] = 0 * 0 = 0
합계: 1 + 0 + 0 = 1
```
이것은 mat의 0행 `{1, 0, 0}`과 mat의 0열 `{1, 0, 0}`의 내적이다. 같은 위치에 1이 있으므로 `1×1 + 0×0 + 0×0 = 1`이다.

**result[1][1] 계산 (r=1, c=1):**
```
k=0: mat[1][0] * mat[0][1] = 0 * 0 = 0
k=1: mat[1][1] * mat[1][1] = 1 * 1 = 1
k=2: mat[1][2] * mat[2][1] = 0 * 0 = 0
합계: 0 + 1 + 0 = 1
```
이것은 mat의 1행 `{0, 1, 0}`과 mat의 1열 `{0, 1, 0}`의 내적이다.

**result[2][2] 계산 (r=2, c=2):**
```
k=0: mat[2][0] * mat[0][2] = 0 * 0 = 0
k=1: mat[2][1] * mat[1][2] = 0 * 0 = 0
k=2: mat[2][2] * mat[2][2] = 1 * 1 = 1
합계: 0 + 0 + 1 = 1
```
이것은 mat의 2행 `{0, 0, 1}`과 mat의 2열 `{0, 0, 1}`의 내적이다.

**출력: "1 1 1"**

사실 이 문제는 단위 행렬의 성질을 알면 코드를 추적할 필요도 없다. **I × I = I**이므로, 결과 행렬도 단위 행렬이고, 대각선 원소는 모두 1이다.

**오답 분석:**

**(A) "0 0 0"** — 대각선이 아닌 원소의 값을 답으로 선택한 경우. 단위 행렬의 대각선 **외** 원소는 모두 0이지만, 대각선 원소는 1이다.

**(B) "2 2 2"** — 행렬 곱셈에서 `result[r][c] += ...`의 `+=`을 `= 2 *`로 잘못 이해한 경우. 또는 단위 행렬끼리 곱하면 대각선이 2가 된다고 착각한 경우.

**(C) "3 3 3"** — 3×3 행렬이므로 대각선 값이 3이라고 추측한 경우. 행렬의 **크기**와 대각선 **값**은 관계가 없다. k 루프가 3번 돌지만, 각 반복에서 0이 아닌 곱이 나오는 것은 단 1번뿐이다.

> ⚠️ **함정:** `result[r][c] += mat[r][k] * mat[k][c]`에서 `+=`(누적 덧셈)을 사용한다는 점에 주의하자. k 루프가 3번 돌면서 각각의 곱을 누적하는 것이다. 처음에 `result[r][c]`는 `int`의 기본값인 0이므로, k=0, 1, 2의 곱을 모두 더한 값이 최종 결과이다.

> 💡 **꿀팁:** 단위 행렬의 성질 `I × A = A × I = A`를 기억하면, `I × I = I`를 바로 알 수 있다. AP 시험에서 단위 행렬이 직접 출제되는 경우는 드물지만, 2D 배열과 중첩 루프의 이해를 테스트하는 문제에서 행렬 곱셈 패턴이 나올 수 있다. 3중 루프의 구조를 파악하는 것이 핵심이다: 바깥 두 루프(r, c)는 결과 행렬의 위치를, 안쪽 루프(k)는 내적 계산을 담당한다.

### Q20. 정답: **(A) "DE-1"**

📌 **출제 의도:** `String.substring(int beginIndex)`의 **단일 매개변수** 버전과, `String.indexOf(String str)`의 **존재하지 않는 문자열 검색 시 반환값(-1)**, 그리고 String + int 연산에서의 **자동 문자열 변환**이라는 3가지 개념이 한 줄에 결합된 종합 문제. 각 메서드의 정확한 명세를 알아야 풀 수 있다.

📎 **연계:** Unit 1(Using Objects)의 String 클래스 메서드 활용 문제. AP CSA의 Java Quick Reference에 명시된 String 메서드 6개(`length`, `substring(int)`, `substring(int, int)`, `indexOf(String)`, `equals`, `compareTo`) 중 3개를 한 문제에서 다룬다. Q1, Q3, Q16 등 다른 String 관련 문제와 연결된다.

> **핵심 개념: substring(int beginIndex)와 indexOf의 반환값**
> - `s.substring(beginIndex)`: `beginIndex`부터 **문자열 끝까지** 잘라서 반환. `beginIndex`가 음수이거나 `s.length()`보다 크면 `StringIndexOutOfBoundsException` 발생
> - `s.indexOf(str)`: `str`이 `s` 안에 처음 나타나는 위치(0부터 시작하는 인덱스)를 반환. **`str`이 없으면 `-1`을 반환**
> - String + int 연산: `+` 연산자의 한 쪽이 String이면 다른 쪽은 자동으로 String으로 변환된 후 **문자열 연결**된다. 즉 `"DE" + (-1)` = `"DE" + "-1"` = `"DE-1"`

> **선행 지식:**
> - `s.length()` = 5 (문자 5개: A, B, C, D, E)
> - `s.length() - 2` = 3 → `substring(3)` = "DE" (인덱스 3부터 끝까지)
> - "ABCDE"에는 'F'가 없으므로 `s.indexOf("F")` = `-1`
> - String concatenation은 **왼쪽에서 오른쪽으로** 평가됨 (Q3에서 다룬 좌결합)

**코드:**
```java
String s = "ABCDE";
System.out.println(s.substring(s.length() - 2) + s.indexOf("F"));
```

**한 줄씩 추적:**

| 단계 | 표현식 | 값 | 설명 |
|------|--------|-----|------|
| 1 | `s.length()` | `5` | "ABCDE"의 길이 |
| 2 | `s.length() - 2` | `3` | 5 - 2 |
| 3 | `s.substring(3)` | `"DE"` | 인덱스 3부터 끝까지 (D는 인덱스 3, E는 인덱스 4) |
| 4 | `s.indexOf("F")` | `-1` | "ABCDE"에 'F'가 없으므로 -1 |
| 5 | `"DE" + (-1)` | `"DE-1"` | String + int → String 변환 후 연결 |
| 6 | 출력 | `"DE-1"` | println으로 출력 |

**시각화 (인덱스 표):**

```
인덱스:  0   1   2   3   4
문자:    A   B   C   D   E
                     ↑
              substring(3)부터 시작
              → "DE"
```

**오답 분석:**

| 선택지 | 잘못된 해석 |
|--------|-------------|
| **(A) "DE-1"** ✅ | 정답 |
| **(B) "DE0"** | `indexOf("F")`가 -1이 아니라 0을 반환한다고 착각. 인덱스 0부터 시작하지만, **존재하지 않으면 -1**이라는 명세를 모르는 경우 |
| **(C) "CDE-1"** | `substring(s.length() - 2)`를 `substring(s.length() - 3)`으로 잘못 계산. 5-2=3인데 "CDE"는 인덱스 2부터이다. 헷갈리면 손가락으로 세어보자 |
| **(D) 컴파일 에러** | String + int 연산은 컴파일 에러가 아니다. Java는 `+` 연산자의 한 쪽이 String이면 다른 쪽을 자동으로 toString() 호출하여 연결한다 |

> 🍎 **비유:** `substring(beginIndex)`은 "여기부터 끝까지 모두 가져와"라는 명령이다. 식빵을 자를 때 "3번째 조각부터 끝까지 챙겨줘"와 같다. `indexOf("F")`는 "F가 어디 있어?"라는 질문에 "없어요(-1)"라고 답하는 것과 같다. AP CSA에서 -1은 "검색 실패"의 표시이다.

> ⚠️ **함정 1:** `s.length() - 2`를 보고 "끝에서 두 번째 문자 1개"라고 착각하면 안 된다. `substring(3)`은 인덱스 **3부터 끝까지** 모두 자르므로, 결과는 **"DE"** (2글자)이다. "끝에서 2개"라는 의미는 맞지만, 한 문자가 아니다.

> ⚠️ **함정 2:** `indexOf`의 반환값이 -1이라는 것을 잊으면 안 된다. 많은 학생이 "찾지 못하면 0이나 예외"라고 잘못 기억한다. AP CSA에서 명확히: **존재하지 않으면 -1을 반환**하며, 예외는 발생하지 않는다.

> ⚠️ **함정 3:** String concatenation에서 음수가 어떻게 처리되는지 헷갈릴 수 있다. `"DE" + (-1)`은 `"DE" + "-1"` = `"DE-1"`이다. 음수 부호도 문자열의 일부로 포함된다. 만약 `"DE" - 1`이었다면 컴파일 에러였을 것이다(String에 `-` 연산은 정의되지 않음).

> 💡 **꿀팁 1:** AP CSA에서 String 메서드는 단 6개만 외우면 된다(Java Quick Reference에 명시): `length()`, `substring(int)`, `substring(int, int)`, `indexOf(String)`, `equals(Object)`, `compareTo(String)`. 그 외 메서드(`charAt`, `toUpperCase`, `replace` 등)는 출제되지 않는다. 6개 메서드의 정확한 동작과 반환값을 외우자.

> 💡 **꿀팁 2:** `indexOf`는 항상 두 가지 케이스로 결과를 분류하자: **(1) 0 이상이면 찾음**, **(2) -1이면 못 찾음**. 그래서 검색 결과를 사용할 때는 보통 `if (s.indexOf(...) >= 0)` 패턴으로 검사한다. 시험 문제에서 -1이 답인 경우가 자주 나오므로 항상 의심하자.

> 💡 **꿀팁 3:** `String + int` vs `int + int + String`의 평가 순서: Java는 **왼쪽부터 순차 평가**한다. `1 + 2 + "X"`는 `(1+2) + "X"` = `"3X"`이지만, `"X" + 1 + 2`는 `("X"+1) + 2` = `"X12"`이다. 이 문제처럼 `"DE" + (-1)`은 String이 먼저 등장했으므로 즉시 String 연결로 처리된다.

### Q21. 정답: **(A) "3 2"**

📌 **출제 의도:** `while` 루프에서 **반복적인 뺄셈**으로 "몫"과 "나머지"를 구해내는 패턴을 추적할 수 있는지 테스트. 이는 나눗셈 `/`와 모듈로 `%` 연산자를 **구현(simulate)**한 것이다. 루프 종료 조건 `a >= b`와 루프 본문 내 변수 변화를 정확히 추적하는 것이 핵심이다.

📎 **연계:** Unit 2(Selection & Iteration)의 `while` 루프 추적 문제. Q1(floating-point), Q5(do-while), Q7(for 루프) 등 다른 반복문 문제들과 연결된다. Q6의 나눗셈/모듈로 문제를 while 루프로 직접 구현하는 패턴으로, 알고리즘 사고력을 묻는다.

> **핵심 개념: while 루프 종료 조건과 반복적 뺄셈**
> - `while (조건)` 루프는 **조건이 거짓이 될 때까지** 반복 실행된다
> - 조건 검사 → 참이면 본문 실행 → 다시 조건 검사 → ... → 거짓이면 탈출
> - `a -= b`는 `a = a - b`와 동일: `a`에서 `b`를 한 번 뺀다
> - 이 코드는 **정수 나눗셈을 뺄셈으로 구현**한 것이다:
>   - `17 / 5` = 3 (몫) ← `count`에 해당
>   - `17 % 5` = 2 (나머지) ← 루프 종료 후 `a`에 해당

> **선행 지식:**
> - 초기값: `a = 17`, `b = 5`, `count = 0`
> - 루프 종료 조건: `a >= b`가 거짓(즉 `a < b`)일 때 탈출
> - `String + int + " " + int`의 좌결합 평가 순서 (Q3에서 다룸)

**코드:**
```java
int a = 17;
int b = 5;
int count = 0;
while (a >= b) {
    a -= b;
    count++;
}
System.out.println(count + " " + a);
```

**한 줄씩 추적:**

| 반복 회차 | `a` 값 (진입 시) | 조건 `a >= b` 검사 | 본문 실행 후 `a` | `count` |
|-----------|-------------------|---------------------|---------------------|---------|
| 시작      | 17                | -                   | -                   | 0       |
| 1회차     | 17                | `17 >= 5` → true    | `17 - 5 = 12`       | 1       |
| 2회차     | 12                | `12 >= 5` → true    | `12 - 5 = 7`        | 2       |
| 3회차     | 7                 | `7 >= 5` → true     | `7 - 5 = 2`         | 3       |
| 탈출      | 2                 | `2 >= 5` → **false**| (본문 실행 안 함)   | 3       |

루프 탈출 후: `a = 2`, `count = 3`.

**최종 출력 계산:**
```
count + " " + a
= 3 + " " + 2
= (3 + " ") + 2       ← 좌결합
= "3 " + 2
= "3 2"
```

**시각화 (17에서 5를 빼는 과정):**

```
17 ─[−5]→ 12 ─[−5]→ 7 ─[−5]→ 2    (여기서 멈춤: 2 < 5)
  1회        2회      3회
           count가 1씩 증가 → 최종 count = 3
           마지막 a 값 → 나머지 = 2
```

**오답 분석:**

| 선택지 | 잘못된 해석 |
|--------|-------------|
| **(A) "3 2"** ✅ | 정답 |
| **(B) "3 0"** | 루프가 `a == 0`이 될 때까지 돈다고 착각. 조건은 `a >= b`이므로 `a`가 `b`보다 작아지면 멈춘다. `a = 2`가 남은 상태로 종료 |
| **(C) "4 0"** | 루프가 한 번 더 돌아서 `a = 2 - 5 = -3`이 되고 count = 4라고 착각. 하지만 `2 >= 5`가 false이므로 4번째 반복은 시작조차 하지 않는다 |
| **(D) "2 7"** | 루프를 두 번만 돌았다고 착각 (17 → 12 → 7에서 멈춤). 하지만 `7 >= 5`가 여전히 true이므로 루프가 한 번 더 돈다 |

> 🍎 **비유:** 17개의 사과를 5개씩 담을 수 있는 상자에 나눠 담는 상황이다. "5개 이상 남았으면 한 상자에 담는다"는 규칙으로 반복하면:
> - 상자 1: 5개 담음, 12개 남음
> - 상자 2: 5개 담음, 7개 남음
> - 상자 3: 5개 담음, 2개 남음
> - 이제 2개만 남아서 상자 하나를 채울 수 없음 → **상자 수 = 3, 남은 사과 = 2**

> ⚠️ **함정 1:** 루프 탈출 조건을 헷갈리면 안 된다. `while (a >= b)`는 `a`가 `b` **이상**인 동안 반복한다. 즉 `a == b`일 때도 루프는 돈다. 이 문제에서 `a = 5`가 되는 순간은 없지만, 만약 있다면 "5 >= 5" = true이므로 한 번 더 돈다.

> ⚠️ **함정 2:** 루프 탈출 **직전**의 `a` 값과 탈출 **직후**의 `a` 값을 헷갈리지 말자. 3회차 본문 실행 직후 `a = 2`가 되고, 이 값으로 다시 조건 검사 → false → 탈출. 탈출 시점의 `a`는 **2**이다. count++가 마지막에 실행된 이후 조건을 한 번 더 검사한다는 점이 핵심.

> ⚠️ **함정 3:** `count + " " + a`의 평가 순서. `count + " "`가 먼저 평가되어 `"3 "`이 되고, 그 다음 `"3 " + 2` = `"3 2"`가 된다. 만약 `a + " " + count`였다면 `2 + " " + 3` = `"2 3"`. **왼쪽부터 순차 평가**되며, String이 등장한 순간부터 문자열 연결로 처리된다.

> 💡 **꿀팁 1:** `while (a >= b) { a -= b; count++; }` 패턴은 **정수 나눗셈 시뮬레이션**이다. 종료 후:
> - `count` = `원래 a / b` (몫)
> - `a` = `원래 a % b` (나머지)
> 
> 이 공식을 암기하면 추적 없이도 답을 구할 수 있다: `17 / 5 = 3` (count), `17 % 5 = 2` (a).

> 💡 **꿀팁 2:** while 루프 추적 시 **표**를 그리자. 반복 회차마다 (1) 조건 검사 전 변수 상태, (2) 조건 결과, (3) 본문 실행 후 상태를 기록한다. 이 표가 없으면 4-5회 반복부터 반드시 실수한다. AP 시험장에서도 반드시 표를 그려 추적하자.

> 💡 **꿀팁 3:** "루프가 몇 번 도는가?"와 "변수의 최종값"을 구하는 문제가 헷갈릴 때는, 마지막 반복 이후 **조건 검사 한 번 더 실행**된다는 점을 기억하자. count는 "조건이 true였던 횟수" = "본문이 실행된 횟수"이다. 이 문제에서 본문은 3번 실행되었으므로 count = 3.

### Q22. 정답: **(C) 4**

📌 **출제 의도:** **캡슐화(encapsulation)**의 기본 개념을 이해하는지, 그리고 **getter가 내부 참조를 그대로 반환할 때 발생하는 캡슐화 누수(encapsulation leak)**를 인지하는지 테스트. 참조 타입(ArrayList)의 getter는 단순히 `return this.field`를 하면 외부에서 내부 상태를 직접 변경할 수 있게 된다.

📎 **연계:** Unit 3(Class Creation) 핵심 문제. Q9(생성자), Q10(정적 변수), Q11(this) 등 클래스 설계 관련 문제의 연장선. Unit 4(Data Collections)의 ArrayList 참조 개념과 결합되어 있다. AP CSA에서 "mutable field의 getter" 패턴은 시험 문제와 FRQ에서 모두 자주 등장한다.

> **핵심 개념: 캡슐화와 참조 누수(reference leak)**
> 캡슐화는 **내부 상태를 숨기고 제어된 접근**을 제공하는 것이 목적이다. 하지만 `private ArrayList<String> names;`를 `public ArrayList<String> getNames() { return names; }`로 그대로 반환하면, 외부 코드가 `names`의 **참조를 획득**하게 된다. 그 참조를 통해 외부에서 add, remove, set 등을 호출하면 **내부 상태가 변경**된다. 이는 `private` 선언의 보호 효과를 무력화시킨다.
>
> 올바른 방어: `return new ArrayList<>(names);` (방어적 복사, defensive copy)

> **선행 지식:**
> - ArrayList는 **참조 타입**이다. 변수에 저장되는 것은 객체 자체가 아니라 메모리 주소(참조)
> - `return names;`는 **참조값을 복사해서 반환**한다. 반환된 참조와 내부 필드는 **같은 객체**를 가리킨다
> - `ref.add(...)`는 그 객체에 직접 요소를 추가한다. 따라서 내부 필드 `names`에도 반영된다
> - `names.size()`는 ArrayList의 현재 요소 개수를 반환한다

**코드:**
```java
public class Roster {
    private ArrayList<String> names;
    public Roster() {
        names = new ArrayList<>();
        names.add("Alice");
        names.add("Bob");
    }
    public ArrayList<String> getNames() { return names; }  // 내부 참조 그대로 반환!
    public int size() { return names.size(); }
}
```

```java
Roster r = new Roster();
ArrayList<String> ref = r.getNames();
ref.add("Charlie");
ref.add("Dave");
System.out.println(r.size());
```

**메모리 참조 다이어그램:**

```
1단계: new Roster() 후
    r ──→ [Roster 객체]
              │
              names ──→ [ArrayList: "Alice", "Bob"]

2단계: ArrayList<String> ref = r.getNames();
    r ──→ [Roster 객체]
              │
              names ──┐
                      ├──→ [ArrayList: "Alice", "Bob"]
    ref ──────────────┘
    
    ★ 중요: r.names와 ref는 같은 ArrayList 객체를 가리킨다!

3단계: ref.add("Charlie"); ref.add("Dave");
    r ──→ [Roster 객체]
              │
              names ──┐
                      ├──→ [ArrayList: "Alice", "Bob", "Charlie", "Dave"]
    ref ──────────────┘
    
    ref로 add한 것이 r.names에도 영향을 미친다 (같은 객체이므로)
```

**한 줄씩 추적:**

| 단계 | 코드 | r.names의 상태 | size() 결과 |
|------|------|----------------|-------------|
| 1 | `Roster r = new Roster();` | `["Alice", "Bob"]` | 2 |
| 2 | `ArrayList<String> ref = r.getNames();` | `["Alice", "Bob"]` (ref = names) | 2 |
| 3 | `ref.add("Charlie");` | `["Alice", "Bob", "Charlie"]` | 3 |
| 4 | `ref.add("Dave");` | `["Alice", "Bob", "Charlie", "Dave"]` | 4 |
| 5 | `System.out.println(r.size());` | - | **4 출력** |

**최종 출력: 4**

**오답 분석:**

| 선택지 | 잘못된 해석 |
|--------|-------------|
| **(A) 2** | getter가 **방어적 복사**를 수행한다고 착각. 즉 `ref`는 별개의 ArrayList이므로 `r.names`에는 영향 없다고 생각. 하지만 코드는 `return names;`로 **참조 그대로** 반환 |
| **(B) 3** | 추가한 요소 중 하나만 반영됐다고 착각. 두 번의 `ref.add()`가 모두 같은 객체에 추가되므로 둘 다 반영된다 |
| **(C) 4** ✅ | 정답 |
| **(D) 컴파일 에러** | `private` 필드를 외부에서 수정한다고 컴파일 에러가 나는 것은 아니다. getter를 통해 참조를 얻은 후에는 **어디든** 가공할 수 있다. 이것이 **캡슐화 누수**의 위험 |

> 🍎 **비유:** "Roster 클래스의 `names`"는 **집 안에 있는 장부**라고 생각하자. `private`은 "집 안에 있다"는 의미. `getNames()`가 `return names;`라면, 외부인이 집에 와서 장부를 요청했을 때 **원본을 건네주는** 것이다. 그 사람이 장부에 낙서를 하면 집주인의 원본이 훼손된다. 올바른 방법은 **복사본을 건네주는** 것(`return new ArrayList<>(names);`).

> ⚠️ **함정 1:** `private`만 붙이면 안전하다고 착각하기 쉽다. `private`는 **직접 접근**을 막지만, `public` 메서드를 통해 **참조를 유출**하면 의미가 없다. AP CSA 시험에서 "다음 getter 중 캡슐화를 유지하는 것은?" 같은 문제가 자주 출제된다.

> ⚠️ **함정 2:** `ArrayList`는 참조 타입이고, `int`나 `double`은 값 타입(primitive)이다. `int age`의 getter `return age;`는 **값 복사**를 반환하므로 안전하다. 하지만 `ArrayList`, `int[]`, 사용자 정의 객체 등은 참조 복사이므로 주의해야 한다.

> ⚠️ **함정 3:** "ref와 r.names는 다른 변수니까 다른 객체"라고 착각하면 안 된다. **변수는 다르지만 같은 객체를 가리킨다**(aliasing). 이는 Q2, Q13 등 다른 문제에서도 반복되는 AP CSA의 핵심 패턴이다.

> 💡 **꿀팁 1:** 캡슐화를 완전히 보장하려면 getter에서 **방어적 복사**를 반환하자:
> ```java
> public ArrayList<String> getNames() {
>     return new ArrayList<>(names);  // 복사본 반환
> }
> ```
> 이러면 외부에서 반환된 리스트를 수정해도 `names`에는 영향이 없다.

> 💡 **꿀팁 2:** 혹은 **불변 뷰(immutable view)**를 제공할 수도 있다. 하지만 이는 AP CSA 범위 밖이므로, 시험에서는 "내부 참조를 그대로 반환하는가, 복사본을 반환하는가"만 구분하면 된다.

> 💡 **꿀팁 3:** AP CSA FRQ에서 자주 나오는 패턴: "ArrayList 필드를 직접 반환하는 getter는 캡슐화를 어떻게 해치는가?"라는 질문에 대비하여, 아래 문구를 기억하자: **"외부 클라이언트가 반환된 참조를 통해 내부 상태를 변경할 수 있다."** 이 한 줄로 부분 점수를 챙길 수 있다.

### Q23. 정답: **(C) 2**

📌 **출제 의도:** 2D 배열의 **행 단위 순회**와 **각 행 내에서의 쌍(pair) 비교**를 결합한 문제. "엄격히 증가하는 행(strictly increasing row)"을 찾으려면 각 행에서 `arr[r][c] < arr[r][c+1]`을 모든 인접 쌍에 대해 확인해야 한다. 2D 배열 + 조건 누적의 복합 문제이다.

📎 **연계:** Unit 4(Data Collections)의 2D 배열 심화 문제. LT1(대각선), LT2(삼각), LT3(max 위치), LT4(테두리)에서 발전된 패턴. 행 단위 분석은 AP CSA FRQ에서도 자주 출제되는 핵심 스킬이다.

> **핵심 개념: "엄격히 증가"의 정의**
> 배열 `a[0..n-1]`이 **엄격히 증가(strictly increasing)**한다는 것은:
> - 모든 인접한 쌍에 대해 `a[i] < a[i+1]`가 성립하는 것
> - 즉 `a[0] < a[1] < a[2] < ... < a[n-1]`
> - **같은 값도 허용되지 않는다**: `{5, 5, 6}`은 엄격히 증가가 아님 (5 == 5이므로)
> - "비감소(non-decreasing)"는 `<=`를 허용하지만, "엄격히 증가"는 `<`만 허용

> **선행 지식:**
> - 2D 배열 `mat[r][c]`: r은 행 인덱스, c는 열 인덱스
> - `mat.length` = 행 개수, `mat[r].length` = r번째 행의 열 개수
> - 각 행을 확인할 때 `for (int c = 0; c < mat[r].length - 1; c++)` 로 인접 쌍을 비교
> - 하나라도 `mat[r][c] >= mat[r][c+1]`이면 그 행은 증가하지 않음 → break 또는 flag 사용

**코드:**
```java
int[][] mat = {
    {1, 2, 3, 4},
    {5, 5, 6, 7},
    {2, 4, 6, 8}
};

int count = 0;
for (int r = 0; r < mat.length; r++) {
    boolean strictlyIncreasing = true;
    for (int c = 0; c < mat[r].length - 1; c++) {
        if (mat[r][c] >= mat[r][c+1]) {
            strictlyIncreasing = false;
        }
    }
    if (strictlyIncreasing) count++;
}
System.out.println(count);
```

**각 행별 분석:**

| 행 | 배열 | 인접 쌍 검사 | 엄격히 증가? |
|---|------|--------------|-------------|
| 0 | `{1, 2, 3, 4}` | 1<2 ✓, 2<3 ✓, 3<4 ✓ | **YES** ✅ |
| 1 | `{5, 5, 6, 7}` | **5<5 ✗** (같음, 증가 아님) | **NO** ❌ |
| 2 | `{2, 4, 6, 8}` | 2<4 ✓, 4<6 ✓, 6<8 ✓ | **YES** ✅ |

조건을 만족하는 행: 행 0, 행 2 → **총 2개**

**최종 출력: 2**

**시각화:**

```
    c=0   c=1   c=2   c=3
    ┌────┬────┬────┬────┐
r=0 │  1 │  2 │  3 │  4 │  → 1<2<3<4 ✅ 엄격히 증가
    ├────┼────┼────┼────┤
r=1 │  5 │  5 │  6 │  7 │  → 5==5 ❌ 엄격히 증가 아님
    ├────┼────┼────┼────┤
r=2 │  2 │  4 │  6 │  8 │  → 2<4<6<8 ✅ 엄격히 증가
    └────┴────┴────┴────┘
    
    count = 2
```

**오답 분석:**

| 선택지 | 잘못된 해석 |
|--------|-------------|
| **(A) 0** | "모든 인접 쌍이 다른지"까지 조건을 강하게 걸어서 0이라고 생각. 하지만 행 0은 1,2,3,4로 모두 다르고 증가하므로 포함 |
| **(B) 1** | 행 2 `{2, 4, 6, 8}`에서 "값이 하나도 겹치지 않아야" 같은 추가 조건을 걸거나, "2로 시작하니 1로 시작하는 것과 다르다" 같은 이상한 필터를 적용한 경우. 실제로는 행 0과 행 2 **둘 다** 엄격히 증가 |
| **(C) 2** ✅ | 정답 |
| **(D) 3** | **비감소(non-decreasing)**와 **엄격히 증가(strictly increasing)**를 혼동. 행 1 `{5, 5, 6, 7}`은 비감소이지만 엄격히 증가는 아니다. 5 == 5는 **증가가 아니다** (같은 값 허용 안 함) |

> 🍎 **비유:** "엄격히 증가하는 행"을 **등산 루트**에 비유하면: (1) 행 0은 4m → 8m → 12m → 16m처럼 계속 올라간다. (2) 행 1은 중간에 **평평한 구간(5m → 5m)**이 있어서 순수한 오르막이 아니다. (3) 행 2도 꾸준히 올라간다. 평지도 허용되지 않는 **엄격한 오르막**이 조건이다.

> ⚠️ **함정 1:** "엄격히 증가"와 "증가(비감소)"의 차이. 수학에서:
> - 엄격히 증가(strictly increasing): `a[i] < a[i+1]` (<만 허용)
> - 증가/비감소(non-decreasing): `a[i] <= a[i+1]` (<=, 같음도 허용)
> 영어 시험에서는 **"strictly increasing"**이라는 표현이 자주 등장하는데, **같은 값은 허용되지 않는다**는 뜻이다.

> ⚠️ **함정 2:** 첫 번째 조건 위반 시 break를 해야 효율적이지만, 위 코드처럼 **flag를 쓰고 끝까지 검사**해도 결과는 같다. AP에서는 둘 다 허용. 다만 "인접 쌍 중 하나라도 위반하면 그 행은 탈락"이라는 **OR 로직**을 정확히 구현해야 한다. AND 로직으로 잘못 짜면 결과가 달라진다.

> ⚠️ **함정 3:** 열 개수가 다른 **들쑥날쑥(ragged) 배열**은 AP CSA 2025-26 범위에서 제외되었다. 이 문제의 배열은 모두 4열로 같은 직사각형 배열이지만, 혹시라도 행별 길이가 다를 수 있다고 의심해 `mat[r].length`를 참조하는 코드를 짜는 것은 방어적이고 좋은 습관이다.

> 💡 **꿀팁 1:** "각 행에 대해 조건 확인" 패턴은 AP CSA의 단골이다. 기본 틀:
> ```java
> for (int r = 0; r < mat.length; r++) {
>     // 행 r에 대한 조건 확인
>     for (int c = ...; ...; c++) { ... }
>     // 결과 누적
> }
> ```
> 이중 for 루프에서 바깥 루프는 "어떤 행?", 안쪽 루프는 "그 행 내 처리"를 담당한다.

> 💡 **꿀팁 2:** "연속한 쌍" 검사는 반드시 `c < mat[r].length - 1`로 범위를 설정해야 한다. `c+1`이 유효한 인덱스가 되려면 `c`가 마지막 인덱스보다 작아야 하기 때문. `c < mat[r].length`로 하면 `c+1`이 배열 범위를 벗어나 ArrayIndexOutOfBoundsException이 발생한다.

> 💡 **꿀팁 3:** 엄격히 증가 체크의 간결한 한 줄 구현:
> ```java
> boolean inc = true;
> for (int c = 0; c < row.length - 1 && inc; c++)
>     if (row[c] >= row[c+1]) inc = false;
> ```
> short-circuit(`&& inc`)으로 false가 되면 즉시 루프를 탈출해 효율적이다. AP에서 이런 최적화는 필수는 아니지만 FRQ에서 부분 점수를 챙기는 데 도움이 된다.

### Q24. 정답: **(C) 6**

📌 **출제 의도:** 유클리드 호제법(Euclidean Algorithm)을 재귀(recursion)로 구현한 코드를 추적하여 최대공약수(GCD)를 구할 수 있는지 테스트. 재귀 호출의 매개변수가 어떻게 변하는지, 그리고 종료 조건(base case)에 도달했을 때 무엇을 반환하는지 추적하는 능력이 필요하다.

📎 **연계:** Unit 4(Data Collections) — 재귀의 핵심 문제. 유클리드 호제법은 재귀의 대표적인 응용 사례이며, AP CSA에서 자주 출제된다. Q25의 회문 판별 재귀와 함께, 재귀 호출을 단계별로 추적하는 연습이 필요하다.

> **핵심 개념: 유클리드 호제법**
> 두 양의 정수 a, b의 최대공약수를 구하는 알고리즘이다.
> - **종료 조건(base case):** `b == 0`이면 `a`가 최대공약수
> - **재귀 단계(recursive case):** `gcd(a, b)` = `gcd(b, a % b)`
>
> 원리: `gcd(a, b) = gcd(b, a mod b)` → `b`가 0이 될 때까지 반복하면, 그때의 `a`가 GCD.
> 이것은 기원전 300년경 유클리드가 발견한 알고리즘으로, 가장 오래된 알고리즘 중 하나이다.

> **선행 지식:**
> - `a % b`는 `a`를 `b`로 나눈 나머지이다. 예: `48 % 18 = 12` (48 = 18×2 + 12)
> - 재귀 호출이 반복될 때마다 `b`의 값이 줄어들고, 결국 0이 된다
> - `b == 0`일 때 `a`를 반환하면 그것이 최대공약수이다

**코드:**
```java
public static int gcd(int a, int b)
{
    if (b == 0)
        return a;
    return gcd(b, a % b);
}
```

**gcd(48, 18) 호출 추적:**

**호출 1: gcd(48, 18)**
- `b == 0`? → `18 == 0`? → **거짓**. 종료 조건 미충족.
- `return gcd(18, 48 % 18)` → `48 % 18 = 12` (48 = 18×2 + 12) → `return gcd(18, 12)`

**호출 2: gcd(18, 12)**
- `b == 0`? → `12 == 0`? → **거짓**. 종료 조건 미충족.
- `return gcd(12, 18 % 12)` → `18 % 12 = 6` (18 = 12×1 + 6) → `return gcd(12, 6)`

**호출 3: gcd(12, 6)**
- `b == 0`? → `6 == 0`? → **거짓**. 종료 조건 미충족.
- `return gcd(6, 12 % 6)` → `12 % 6 = 0` (12 = 6×2 + 0) → `return gcd(6, 0)`

**호출 4: gcd(6, 0)**
- `b == 0`? → `0 == 0`? → **참**! 종료 조건 충족!
- `return a` → `return 6`

**반환 체인:** 호출 4가 6을 반환 → 호출 3이 6을 반환 → 호출 2가 6을 반환 → 호출 1이 6을 반환.

**최종 결과: 6**

검증: 48의 약수는 {1, 2, 3, 4, 6, 8, 12, 16, 24, 48}이고, 18의 약수는 {1, 2, 3, 6, 9, 18}이다. 공통 약수는 {1, 2, 3, 6}이고, 최대공약수는 **6**이다.

**오답 분석:**

**(A) 2** — 재귀 추적을 잘못한 경우. 또는 48과 18의 공약수 중 하나인 2를 답으로 선택한 경우. 2는 공약수이긴 하지만, **최대**공약수는 아니다.

**(B) 3** — 마찬가지로 공약수이긴 하지만 최대공약수가 아니다. `a % b` 계산을 잘못하여 중간에 다른 값이 나온 경우이다.

**(D) 18** — `gcd(48, 18)` 호출에서 `b` 값(18)을 바로 반환한다고 생각한 경우. 또는 "48을 18로 나눌 수 있으니 18이 GCD"라고 잘못 판단한 경우. 48 ÷ 18 = 2.666...이므로 18은 48의 약수가 아니다.

> ⚠️ **함정:** `a % b`를 계산할 때 실수하기 쉽다. 48 % 18을 계산할 때, 48 ÷ 18 = 2 나머지 12이므로 48 % 18 = 12이다. 만약 48 % 18 = 30이라고 계산하면(48 - 18 = 30으로 잘못 계산), 전혀 다른 결과가 나온다. `%` 연산은 "나누기의 나머지"이지 "빼기"가 아니다.

> 💡 **꿀팁:** 유클리드 호제법의 호출 횟수는 보통 매우 적다. 48과 18처럼 비교적 큰 수도 4번의 호출만에 GCD를 찾는다. 재귀 추적 문제에서는 각 호출의 매개변수를 표로 정리하면 실수를 줄일 수 있다: `(48,18)` → `(18,12)` → `(12,6)` → `(6,0)` → 반환 6.

### Q25. 정답: **(D) isPalindrome("abcba")**

📌 **출제 의도:** 재귀 함수의 **호출 횟수**를 정확히 추적할 수 있는지 테스트. 각 보기마다 `isPalindrome`이 총 몇 번 호출되는지(초기 호출 포함) 세어야 한다. 종료 조건이 `length <= 1`인 경우와 `첫 글자 != 마지막 글자`인 경우를 모두 고려해야 한다.

📎 **연계:** Unit 4(Data Collections) — 재귀의 심화 문제. Q24(유클리드 호제법)와 마찬가지로 재귀 호출을 단계별로 추적해야 하지만, 이 문제는 4개의 보기 **모두**에 대해 호출 횟수를 계산해야 하므로 더 많은 추적이 필요하다.

> **핵심 개념:** 이 재귀 함수에는 **두 가지 종료 조건**이 있다:
> 1. `s.length() <= 1` → 길이가 0 또는 1이면 즉시 `true` 반환 (더 이상 확인할 문자가 없으므로 회문)
> 2. `!s.substring(0,1).equals(s.substring(s.length()-1))` → 첫 글자와 마지막 글자가 다르면 즉시 `false` 반환 (회문이 아님)
>
> 두 종료 조건 모두 만족하지 않으면 → 첫 글자와 마지막 글자를 제거하고 재귀 호출: `isPalindrome(s.substring(1, s.length()-1))`

> **선행 지식:**
> - `s.substring(0, 1)`은 첫 번째 문자, `s.substring(s.length() - 1)`은 마지막 문자
> - `s.substring(1, s.length() - 1)`은 첫 글자와 마지막 글자를 **제거한** 부분 문자열
> - 각 재귀 호출마다 문자열 길이가 **2씩 줄어든다** (양쪽에서 1글자씩 제거)
> - "총 호출 횟수"는 최초의 호출도 **포함**한다

**코드:**
```java
public static boolean isPalindrome(String s)
{
    if (s.length() <= 1)
        return true;
    if (!s.substring(0, 1).equals(s.substring(s.length() - 1)))
        return false;
    return isPalindrome(s.substring(1, s.length() - 1));
}
```

**각 보기별 호출 추적:**

**(A) isPalindrome("abc") — 총 2회 호출**

호출 1: `isPalindrome("abc")`
- `"abc".length()` = 3. `3 <= 1`? → 거짓.
- 첫 글자 = `"a"`, 마지막 글자 = `"c"`. `"a".equals("c")`? → **거짓**. `!"a".equals("c")` → **참**.
- `return false` → **즉시 종료!** 첫 글자와 마지막 글자가 다르므로 재귀 호출 없이 false 반환.

잠깐, 이것은 1회 호출이다. 다시 확인하자.

호출 1: `isPalindrome("abc")` → 첫 글자 `a` != 마지막 글자 `c` → `return false`. 재귀 호출 없음.
**총 1회 호출.** 3회가 아니다.

**(B) isPalindrome("aba") — 총 2회 호출**

호출 1: `isPalindrome("aba")`
- `"aba".length()` = 3. `3 <= 1`? → 거짓.
- 첫 글자 = `"a"`, 마지막 글자 = `"a"`. `"a".equals("a")`? → **참**. `!true` → **거짓**. 두 번째 조건 통과.
- `return isPalindrome("aba".substring(1, 2))` → `return isPalindrome("b")`

호출 2: `isPalindrome("b")`
- `"b".length()` = 1. `1 <= 1`? → **참**. `return true`.
**총 2회 호출.** 3회가 아니다.

**(C) isPalindrome("abcdef") — 총 1회 호출**

호출 1: `isPalindrome("abcdef")`
- `"abcdef".length()` = 6. `6 <= 1`? → 거짓.
- 첫 글자 = `"a"`, 마지막 글자 = `"f"`. `"a".equals("f")`? → **거짓**. `!false` → **참**.
- `return false` → **즉시 종료!** 첫 글자와 마지막 글자가 다르므로 재귀 호출 없이 false 반환.
**총 1회 호출.** 3회가 아니다.

**(D) isPalindrome("abcba") — 총 3회 호출** ✅

호출 1: `isPalindrome("abcba")`
- `"abcba".length()` = 5. `5 <= 1`? → 거짓.
- 첫 글자 = `"a"`, 마지막 글자 = `"a"`. `"a".equals("a")`? → **참**. 조건 통과.
- `return isPalindrome("abcba".substring(1, 4))` → `return isPalindrome("bcb")`

호출 2: `isPalindrome("bcb")`
- `"bcb".length()` = 3. `3 <= 1`? → 거짓.
- 첫 글자 = `"b"`, 마지막 글자 = `"b"`. `"b".equals("b")`? → **참**. 조건 통과.
- `return isPalindrome("bcb".substring(1, 2))` → `return isPalindrome("c")`

호출 3: `isPalindrome("c")`
- `"c".length()` = 1. `1 <= 1`? → **참**. `return true`.

**총 3회 호출!** ← 정확히 3회!

**반환 체인:** 호출 3이 `true` 반환 → 호출 2가 `true` 반환 → 호출 1이 `true` 반환.

**종합 정리:**

| 보기 | 입력 | 호출 추적 | 총 호출 횟수 |
|---|---|---|---|
| (A) | "abc" | "abc"(a!=c → false) | **1회** |
| (B) | "aba" | "aba" → "b" | **2회** |
| (C) | "abcdef" | "abcdef"(a!=f → false) | **1회** |
| (D) | "abcba" | "abcba" → "bcb" → "c" | **3회** ✅ |

**오답 분석:**

**(A) isPalindrome("abc")** — "abc"는 회문이 아니다. 첫 글자 `a`와 마지막 글자 `c`가 다르므로 **첫 번째 호출에서 즉시 false를 반환**한다. 재귀 호출이 한 번도 일어나지 않으므로 총 **1회** 호출이다. 이 보기를 고른 학생은 문자열의 각 글자마다 한 번씩 호출된다고 잘못 생각한 것이다.

**(B) isPalindrome("aba")** — "aba"는 회문이다. `a`==`a`이므로 재귀 호출하여 `"b"`를 검사한다. `"b"`는 길이 1이므로 바로 true를 반환한다. 총 **2회** 호출이다. 3회가 아니다.

**(C) isPalindrome("abcdef")** — "abcdef"는 회문이 아니다. 첫 글자 `a`와 마지막 글자 `f`가 다르므로 **첫 번째 호출에서 즉시 false를 반환**한다. 총 **1회** 호출이다. 문자열이 길다고 호출 횟수가 많아지는 것이 아니다! 첫 글자와 마지막 글자가 다르면 즉시 종료된다.

> ⚠️ **함정:** 이 문제에서 가장 흔한 실수는 **종료 조건 2(첫 글자 != 마지막 글자 → 즉시 false)**를 간과하고, 모든 경우에서 문자열 끝까지 재귀 호출이 진행된다고 가정하는 것이다. "abc"와 "abcdef"는 첫 글자와 마지막 글자가 다르므로 **1회 호출로 즉시 종료**된다. 재귀는 첫 글자와 마지막 글자가 같을 때**만** 더 깊이 들어간다.

> 💡 **꿀팁:** 회문 재귀의 호출 횟수를 빠르게 계산하는 방법: 문자열의 양쪽 끝에서 시작하여, **같은 글자가 연속으로 몇 쌍 매칭되는지** 센다. "abcba"는 `a`-`a`(1쌍), `b`-`b`(2쌍), 남은 `c`(종료) → 2쌍 + 1(종료) = **3회 호출**. "abc"는 `a`-`c`(불일치, 0쌍) → 0쌍 + 1(즉시 종료) = **1회 호출**. 매칭된 쌍의 수 + 1이 총 호출 횟수이다.

#### Part (a) — `wordCount(String text)` (4점)

**이 문제가 뭘 원하는가:**

주어진 문자열 `text`에 들어 있는 단어의 개수를 반환해야 한다. 단어는 단일 공백(" ")으로 구분되며, 문자열 앞뒤에 공백이 없다. 최소 한 단어는 항상 존재한다(precondition).

예를 들어:
- `wordCount("hello world")` -> 2
- `wordCount("one")` -> 1
- `wordCount("a b c d")` -> 4

**풀이 전략:**

핵심 관찰: **단어 수 = 공백 수 + 1** 이다. "hello world"에는 공백이 1개이므로 단어는 2개다. "a b c d"에는 공백이 3개이므로 단어는 4개다. 따라서 문자열을 처음부터 끝까지 순회하면서 공백 문자의 개수를 세고, 거기에 1을 더하면 된다.

AP CSA에서는 `charAt()`이 Quick Reference에 없으므로, `substring(i, i+1).equals(" ")`로 한 글자씩 비교하는 것이 안전하다.

**모범 답안 (접근법 1: 공백 카운팅):**

```java
public static int wordCount(String text)
{
    // 공백 개수를 세기 위한 카운터
    int spaceCount = 0;

    // 문자열의 모든 위치를 순회
    for (int i = 0; i < text.length(); i++)
    {
        // 현재 위치의 문자가 공백인지 확인
        if (text.substring(i, i + 1).equals(" "))
        {
            spaceCount++;
        }
    }

    // 단어 수 = 공백 수 + 1
    return spaceCount + 1;
}
```

**대안 풀이 (접근법 2: indexOf 루프):**

`indexOf(" ")`를 사용하여 공백의 위치를 반복적으로 찾는 방법이다. `indexOf`는 찾지 못하면 -1을 반환한다는 점을 이용한다.

```java
public static int wordCount(String text)
{
    // 단어 카운터 (최소 1개)
    int count = 1;

    // 첫 번째 공백 위치 탐색
    int idx = text.indexOf(" ");

    // 공백이 존재하는 동안 반복
    while (idx != -1)
    {
        count++;                          // 공백 발견 -> 단어 하나 더 있음
        idx = text.indexOf(" ", idx + 1); // 다음 공백 탐색 (현재 위치+1부터)
    }

    return count;
}
```

**예시 추적 (접근법 1, text = "a b c d"):**

```
문자열: "a b c d"
인덱스:  0 1 2 3 4 5 6

i=0: "a" → 공백 아님 → spaceCount = 0
i=1: " " → 공백!     → spaceCount = 1
i=2: "b" → 공백 아님 → spaceCount = 1
i=3: " " → 공백!     → spaceCount = 2
i=4: "c" → 공백 아님 → spaceCount = 2
i=5: " " → 공백!     → spaceCount = 3
i=6: "d" → 공백 아님 → spaceCount = 3

return 3 + 1 = 4  ✓
```

**예시 추적 (접근법 2, text = "hello world"):**

```
count = 1
idx = text.indexOf(" ") = 5  → 5 != -1 → count = 2, idx = text.indexOf(" ", 6) = -1
idx == -1 → 루프 종료
return 2  ✓
```

**채점 기준 (4점):**

| 배점 | 기준 |
|------|------|
| +2 | 문자열 순회하며 공백을 올바르게 카운트 (또는 indexOf 루프로 공백 탐색) |
| +1 | 단어 수 = 공백 수 + 1 관계를 올바르게 적용 |
| +1 | 올바른 값 반환 (return) |

**자주 하는 실수:**

1. **`count`를 0으로 시작하고 +1을 안 함:** 공백 수만 반환하면 항상 1 적게 나온다. "hello world" -> 1이 되어 버림.
2. **`charAt()` 사용:** AP CSA 시험에서는 String의 Quick Reference에 `charAt()`이 없으므로, `substring(i, i+1).equals(" ")`를 써야 한다. `charAt()`을 쓰면 감점될 수 있다.
3. **`==` 로 문자열 비교:** `text.substring(i, i+1) == " "`는 참조 비교이므로 동작하지 않을 수 있다. 반드시 `.equals()`를 사용해야 한다.
4. **빈 문자열 케이스 걱정:** precondition에 "최소 한 단어"라고 명시되어 있으므로 빈 문자열은 고려할 필요 없다.
5. **indexOf 접근에서 시작 위치를 idx+1이 아닌 idx로 설정:** 무한 루프에 빠진다.

#### Part (b) — `longestWord(String text)` (5점)

**이 문제가 뭘 원하는가:**

문자열 `text`에서 가장 긴 단어를 찾아 반환해야 한다. 길이가 같은 단어가 여러 개 있으면 **첫 번째** 것을 반환한다. 단어는 공백으로 구분되며, 최소 한 단어가 있다(precondition).

**풀이 전략:**

문자열에서 단어를 하나씩 추출해야 한다. `indexOf(" ")`를 사용하여 공백 위치를 찾고, `substring(start, spaceIndex)`로 단어를 잘라낸다. 각 단어의 길이를 현재까지의 최장 단어와 비교하여, 더 길면 갱신한다.

마지막 단어 처리가 중요하다 -- 마지막 단어 뒤에는 공백이 없으므로 `indexOf`가 -1을 반환한다. 이때 `substring(start)`로 마지막 단어를 추출해야 한다.

**"동률이면 첫 번째"** 를 보장하려면 비교 시 `>` (strictly greater than)를 사용해야 한다. `>=`를 쓰면 뒤에 나오는 같은 길이의 단어로 덮어쓰게 된다.

**모범 답안:**

```java
public static String longestWord(String text)
{
    // 지금까지 찾은 가장 긴 단어
    String longest = "";

    // 현재 단어의 시작 인덱스
    int start = 0;

    // 첫 번째 공백 위치 탐색
    int spaceIdx = text.indexOf(" ");

    // 공백이 존재하는 동안 반복
    while (spaceIdx != -1)
    {
        // 현재 단어 추출: start부터 공백 직전까지
        String word = text.substring(start, spaceIdx);

        // 현재 단어가 지금까지의 최장 단어보다 길면 갱신
        if (word.length() > longest.length())
        {
            longest = word;
        }

        // 다음 단어의 시작 위치 = 공백 다음
        start = spaceIdx + 1;

        // 다음 공백 탐색
        spaceIdx = text.indexOf(" ", start);
    }

    // 마지막 단어 처리 (공백 뒤에 남은 부분)
    String lastWord = text.substring(start);
    if (lastWord.length() > longest.length())
    {
        longest = lastWord;
    }

    return longest;
}
```

**대안 풀이 (단일 루프, 인덱스 순회):**

```java
public static String longestWord(String text)
{
    String longest = "";
    int start = 0;

    // i를 text.length()까지 포함하여 마지막 단어도 처리
    for (int i = 0; i <= text.length(); i++)
    {
        // 문자열 끝이거나 공백을 만나면 -> 단어 하나 완성
        if (i == text.length() || text.substring(i, i + 1).equals(" "))
        {
            String word = text.substring(start, i);
            if (word.length() > longest.length())
            {
                longest = word;
            }
            start = i + 1; // 다음 단어 시작 위치
        }
    }

    return longest;
}
```

**예시 추적 (text = "hi world ab programming ok"):**

```
목표: 가장 긴 단어 찾기 → "programming" (11글자)

[indexOf 루프 접근]
start = 0, spaceIdx = 2, longest = ""

반복 1: word = "hi" (길이 2) → 2 > 0 → longest = "hi"
         start = 3, spaceIdx = 8

반복 2: word = "world" (길이 5) → 5 > 2 → longest = "world"
         start = 9, spaceIdx = 11

반복 3: word = "ab" (길이 2) → 2 > 5? 아니오 → longest 유지 = "world"
         start = 12, spaceIdx = 23

반복 4: word = "programming" (길이 11) → 11 > 5 → longest = "programming"
         start = 24, spaceIdx = -1 → 루프 종료

마지막 단어: lastWord = "ok" (길이 2) → 2 > 11? 아니오

return "programming"  ✓
```

**동률 예시 추적 (text = "cat dog ant"):**

```
모든 단어 길이 3

반복 1: word = "cat" (3) → 3 > 0 → longest = "cat"
반복 2: word = "dog" (3) → 3 > 3? 아니오 (> 이므로!) → longest 유지 = "cat"
마지막: word = "ant" (3) → 3 > 3? 아니오 → longest 유지 = "cat"

return "cat"  ✓  (동률 시 첫 번째 반환)
```

**채점 기준 (5점):**

| 배점 | 기준 |
|------|------|
| +2 | 문자열에서 단어를 올바르게 분리 (indexOf+substring 또는 인덱스 순회) |
| +1 | 각 단어의 길이를 비교하여 최장 단어를 추적 |
| +1 | `>` 사용으로 동률 시 첫 번째 단어 반환 보장 (>= 사용 시 감점) |
| +1 | 마지막 단어 포함하여 올바른 값 반환 |

**자주 하는 실수:**

1. **마지막 단어 누락:** `indexOf`가 -1을 반환하면 루프가 끝나는데, 마지막 단어는 공백 뒤에 있으므로 루프 밖에서 별도로 처리해야 한다. 이걸 빼먹으면 마지막 단어가 무시된다.
2. **`>=` 사용:** `if (word.length() >= longest.length())`로 하면 동률 시 나중에 나오는 단어로 갱신되어 "첫 번째"가 아닌 "마지막" 것이 반환된다.
3. **off-by-one 에러:** `substring(start, spaceIdx)`에서 `spaceIdx`는 공백 위치이므로 exclusive end로 정확하다. `spaceIdx - 1`로 하면 마지막 글자가 잘린다.
4. **단어가 하나뿐일 때:** 공백이 없으면 `indexOf`가 처음부터 -1을 반환한다. 마지막 단어 처리 코드에서 `text.substring(0)` = text 전체가 되어 정상 작동하는지 확인해야 한다.
5. **`longest`를 `null`로 초기화:** `null.length()`에서 NullPointerException이 발생한다. 빈 문자열 `""`로 초기화해야 안전하다.

### FRQ Q2: ParkingLot (9 Points)

---

#### Part (a) — `addCar`, `findCar`, `getTotalRevenue` (5점)

**이 문제가 뭘 원하는가:**

`Car` 클래스가 이미 제공된 상태에서, `ParkingLot` 클래스를 완성해야 한다. Part (a)에서는:
- `private ArrayList<Car>` 필드 선언과 no-arg 생성자
- `addCar(Car c)`: 주차장에 차를 추가
- `findCar(String plate)`: 번호판으로 차를 검색하여 반환, 없으면 `null`
- `getTotalRevenue(int ratePerHour)`: 모든 차의 (주차 시간 x 시간당 요금)을 합산하여 총 수입 반환

**풀이 전략:**

- `addCar`: 단순히 `ArrayList.add()` 호출
- `findCar`: 리스트를 순회하며 `getPlate().equals(plate)`로 비교. 찾으면 즉시 해당 `Car` 반환, 루프 끝까지 못 찾으면 `null` 반환
- `getTotalRevenue`: 누적합 패턴. 각 차의 `getHours() * ratePerHour`를 더해 나감

**모범 답안:**

```java
public class ParkingLot
{
    // 주차장에 있는 차들의 리스트
    private ArrayList<Car> cars;

    // 빈 주차장 생성
    public ParkingLot()
    {
        cars = new ArrayList<Car>();
    }

    // 주차장에 차 추가
    public void addCar(Car c)
    {
        cars.add(c);
    }

    // 번호판으로 차 검색 — 없으면 null 반환
    public Car findCar(String plate)
    {
        // 모든 차를 순회
        for (Car c : cars)
        {
            // 번호판이 일치하면 해당 Car 객체 반환
            if (c.getPlate().equals(plate))
            {
                return c;
            }
        }
        // 끝까지 못 찾으면 null
        return null;
    }

    // 총 수입 계산: 각 차의 주차시간 × 시간당 요금의 합
    public int getTotalRevenue(int ratePerHour)
    {
        int total = 0;
        for (Car c : cars)
        {
            total += c.getHours() * ratePerHour;
        }
        return total;
    }
```

**예시 추적 (getTotalRevenue):**

```
cars: [Car("ABC", 3), Car("DEF", 5), Car("GHI", 2)]
ratePerHour = 10

total = 0
Car("ABC", 3): total += 3 * 10 = 30  → total = 30
Car("DEF", 5): total += 5 * 10 = 50  → total = 80
Car("GHI", 2): total += 2 * 10 = 20  → total = 100

return 100  ✓
```

**예시 추적 (findCar):**

```
cars: [Car("ABC", 3), Car("DEF", 5), Car("GHI", 2)]
findCar("DEF")

Car("ABC"): "ABC".equals("DEF")? false → 다음
Car("DEF"): "DEF".equals("DEF")? true  → return Car("DEF", 5)  ✓

findCar("XYZ")

Car("ABC"): false → 다음
Car("DEF"): false → 다음
Car("GHI"): false → 다음
루프 종료 → return null  ✓
```

**채점 기준 (5점):**

| 배점 | 기준 |
|------|------|
| +1 | `private ArrayList<Car>` 필드 선언 + 생성자에서 초기화 |
| +1 | `addCar` — `cars.add(c)` 올바르게 구현 |
| +1 | `findCar` — `.equals()`로 번호판 비교 + 못 찾으면 `null` 반환 |
| +2 | `getTotalRevenue` — `getHours() * ratePerHour` 곱셈 + 누적합 + 반환 |

**자주 하는 실수:**

1. **`findCar`에서 `==` 사용:** `c.getPlate() == plate`는 참조 비교이므로 실패할 수 있다. 반드시 `.equals()` 사용.
2. **`findCar`에서 루프 안에 `return null` 배치:** 첫 번째 차가 아니면 바로 null을 반환해 버린다. `return null`은 반드시 루프 **바깥**에 있어야 한다.
3. **`getTotalRevenue`에서 total 초기화 누락:** `int total;`로 선언만 하고 0으로 초기화하지 않으면 컴파일 에러.
4. **ArrayList를 `new ArrayList()` 없이 선언만:** `cars`가 `null`인 상태에서 `.add()` 호출 시 NullPointerException.
5. **`getTotalRevenue`에서 ratePerHour를 곱하지 않고 시간만 합산:** 문제를 제대로 읽지 않은 경우.

#### Part (b) — `getCarsOverLimit`, `removeCarByPlate` (4점)

**이 문제가 뭘 원하는가:**

- `getCarsOverLimit(int maxHours)`: 주차 시간이 `maxHours`를 **초과**하는 차들의 리스트를 새로 만들어 반환
- `removeCarByPlate(String plate)`: 번호판이 일치하는 차를 리스트에서 **제거**하고 `true` 반환. 없으면 `false` 반환

**풀이 전략:**

- `getCarsOverLimit`: 새 ArrayList를 생성하고, 조건(`getHours() > maxHours`)을 만족하는 차를 추가. 필터링 패턴이다.
- `removeCarByPlate`: **인덱스 기반 for 루프** 사용이 필수. `remove(i)` 호출 후 즉시 `return true`하면 된다. for-each 루프에서 `remove`를 하면 `ConcurrentModificationException`이 발생하므로 주의!

**모범 답안:**

```java
    // maxHours를 초과하는 차들의 리스트 반환
    public ArrayList<Car> getCarsOverLimit(int maxHours)
    {
        // 결과를 담을 새 리스트
        ArrayList<Car> result = new ArrayList<Car>();

        for (Car c : cars)
        {
            // 주차 시간이 제한을 초과하면 결과에 추가
            if (c.getHours() > maxHours)
            {
                result.add(c);
            }
        }

        return result;
    }

    // 번호판으로 차를 찾아 제거 — 성공하면 true, 없으면 false
    public boolean removeCarByPlate(String plate)
    {
        // 인덱스 기반 루프 (remove 사용을 위해)
        for (int i = 0; i < cars.size(); i++)
        {
            if (cars.get(i).getPlate().equals(plate))
            {
                cars.remove(i);  // 해당 차 제거
                return true;     // 제거 성공
            }
        }
        return false;  // 해당 번호판의 차 없음
    }
}
```

**예시 추적 (getCarsOverLimit):**

```
cars: [Car("A", 2), Car("B", 8), Car("C", 5), Car("D", 10)]
maxHours = 6

result = []
Car("A", 2): 2 > 6? false → 건너뜀
Car("B", 8): 8 > 6? true  → result = [Car("B", 8)]
Car("C", 5): 5 > 6? false → 건너뜀
Car("D", 10): 10 > 6? true → result = [Car("B", 8), Car("D", 10)]

return [Car("B", 8), Car("D", 10)]  ✓
```

**예시 추적 (removeCarByPlate):**

```
cars: [Car("A", 2), Car("B", 8), Car("C", 5)]
removeCarByPlate("B")

i=0: cars.get(0).getPlate() = "A" → "A".equals("B")? false
i=1: cars.get(1).getPlate() = "B" → "B".equals("B")? true
     → cars.remove(1) → cars = [Car("A", 2), Car("C", 5)]
     → return true  ✓

removeCarByPlate("Z")

i=0: "A".equals("Z")? false
i=1: "C".equals("Z")? false
루프 종료 → return false  ✓
```

**채점 기준 (4점):**

| 배점 | 기준 |
|------|------|
| +1 | `getCarsOverLimit` — `getHours() > maxHours` 조건으로 올바르게 필터링 |
| +1 | `getCarsOverLimit` — 새 ArrayList 생성 + 반환 |
| +1 | `removeCarByPlate` — 번호판 검색 + `cars.remove(i)` 올바르게 호출 |
| +1 | `removeCarByPlate` — 제거 시 `true`, 없으면 `false` 올바르게 반환 |

**자주 하는 실수:**

1. **for-each에서 remove 호출:** `for (Car c : cars) { cars.remove(c); }`는 `ConcurrentModificationException`을 발생시킨다. 반드시 인덱스 기반 for 루프를 사용해야 한다.
2. **`getCarsOverLimit`에서 `>=` 사용:** 문제에서 "exceeding maxHours"라고 했으므로 `>` (초과)를 사용해야 한다. `>=`를 쓰면 경계값에서 틀린다. (예: maxHours=6일 때 정확히 6시간인 차는 포함 안 됨)
3. **`removeCarByPlate`에서 삭제 후 `return`을 빠뜨림:** 삭제 후 계속 루프를 돌면 인덱스가 밀려서 원소를 건너뛰거나 `IndexOutOfBoundsException`이 발생할 수 있다. 즉시 `return true`가 안전하다.
4. **`getCarsOverLimit`에서 원본 리스트를 수정:** 원본 `cars`에서 제거하면 안 된다. 새 리스트를 만들어 반환해야 한다.
5. **`removeCarByPlate`에서 `==` 사용:** String 비교는 항상 `.equals()`.

### FRQ Q3: GradeBook (9 Points)

#### Part (a) — `countInRange`, `applyCurve` (5점)

**이 문제가 뭘 원하는가:**

- `countInRange(int low, int high)`: 배열 `grades`에서 `low` 이상 `high` 이하인 점수의 개수를 반환. **경계값 포함** (`[low, high]` inclusive).
- `applyCurve(int curve)`: 각 점수에 `curve`를 더한 새 배열을 반환. 단, 100을 초과하면 100으로 캡(cap)한다. **원본 배열은 변경하지 않는다.**

**풀이 전략:**

- `countInRange`: 표준 카운팅 패턴. for-each로 순회하며 `grade >= low && grade <= high` 조건을 확인.
- `applyCurve`: 새 배열을 생성(`grades.length` 크기)하고, 각 원소에 대해 `Math.min(grades[i] + curve, 100)`으로 계산. `Math.min`을 사용하면 100 캡을 깔끔하게 처리할 수 있다.

**모범 답안:**

```java
// [low, high] 범위에 들어가는 점수 개수 반환
public int countInRange(int low, int high)
{
    int count = 0;

    // 모든 점수를 순회
    for (int g : grades)
    {
        // low 이상이고 high 이하인 경우 카운트 증가
        if (g >= low && g <= high)
        {
            count++;
        }
    }

    return count;
}

// 커브 적용된 새 배열 반환 (최대 100)
public int[] applyCurve(int curve)
{
    // 원본과 같은 크기의 새 배열 생성
    int[] curved = new int[grades.length];

    for (int i = 0; i < grades.length; i++)
    {
        // 커브 적용 후 100 초과 시 100으로 제한
        curved[i] = Math.min(grades[i] + curve, 100);
    }

    return curved;
}
```

**예시 추적 (countInRange, grades = {72, 85, 91, 60, 88, 77}):**

```

g=72: 72>=70 && 72<=85? true  → count = 1
g=85: 85>=70 && 85<=85? true  → count = 2  (경계값 포함!)
g=91: 91>=70 && 91<=85? false → count = 2
g=60: 60>=70 && 60<=85? false → count = 2
g=88: 88>=70 && 88<=85? false → count = 2
g=77: 77>=70 && 77<=85? true  → count = 3

return 3  ✓
```

**예시 추적 (applyCurve, grades = {72, 85, 91, 60, 95, 99}):**

```

curved = new int[6]

i=0: Math.min(72 + 10, 100) = Math.min(82, 100)  = 82   → curved[0] = 82
i=1: Math.min(85 + 10, 100) = Math.min(95, 100)  = 95   → curved[1] = 95
i=2: Math.min(91 + 10, 100) = Math.min(101, 100) = 100  → curved[2] = 100  ← 캡 적용!
i=3: Math.min(60 + 10, 100) = Math.min(70, 100)  = 70   → curved[3] = 70
i=4: Math.min(95 + 10, 100) = Math.min(105, 100) = 100  → curved[4] = 100  ← 캡 적용!
i=5: Math.min(99 + 10, 100) = Math.min(109, 100) = 100  → curved[5] = 100  ← 캡 적용!

return {82, 95, 100, 70, 100, 100}  ✓
```

**채점 기준 (5점):**

| 배점 | 기준 |
|------|------|
| +1 | `countInRange` — `>= low && <= high` 조건 올바르게 작성 (inclusive) |
| +1 | `countInRange` — 순회 + 카운트 + 반환 |
| +1 | `applyCurve` — 새 배열 생성 (`new int[grades.length]`) |
| +1 | `applyCurve` — 각 원소에 curve 더하기 |
| +1 | `applyCurve` — 100 초과 시 100으로 캡 (`Math.min` 또는 if문) |

**자주 하는 실수:**

1. **`countInRange`에서 경계값 제외:** `> low` 또는 `< high`를 쓰면 경계값이 빠진다. 문제에서 "inclusive"라고 명시했으므로 `>=`와 `<=`를 써야 한다.
2. **`applyCurve`에서 원본 배열 수정:** `grades[i] += curve`로 하면 원본이 변경된다. 문제에서 "returns a new array"라고 했으므로 새 배열을 만들어야 한다.
3. **100 캡을 안 함:** `curved[i] = grades[i] + curve`만 하면 100을 초과하는 값이 그대로 들어간다.
4. **`Math.min` 대신 if문 사용 시 else 누락:**
   ```java
   // 올바른 if문 버전:
   if (grades[i] + curve > 100)
       curved[i] = 100;
   else
       curved[i] = grades[i] + curve;
   ```
   `else`를 빼먹으면 100 이하인 경우 curved[i]가 0(기본값)이 된다.
5. **새 배열 크기를 잘못 지정:** `new int[100]` 같이 하드코딩하면 안 된다. `grades.length`를 사용해야 한다.

#### Part (b) — `getMedian` (4점)

**이 문제가 뭘 원하는가:**

정렬된 배열(precondition)에서 중앙값(median)을 반환한다.
- **홀수 길이:** 정확히 가운데 원소 반환
- **짝수 길이:** 가운데 두 원소의 평균 반환 (정수 나눗셈, 즉 소수점 이하 버림)

**풀이 전략:**

배열의 길이가 `n`이라 할 때:
- 중간 인덱스: `mid = n / 2`
- 홀수(`n % 2 == 1`): `grades[mid]` 반환
- 짝수(`n % 2 == 0`): `(grades[mid - 1] + grades[mid]) / 2` 반환

예를 들어:
- 길이 5: 인덱스 0,1,**2**,3,4 → `grades[2]`
- 길이 6: 인덱스 0,1,**2,3**,4,5 → `(grades[2] + grades[3]) / 2`

정수 나눗셈이므로 `(grades[mid-1] + grades[mid]) / 2`에서 자동으로 소수점 이하가 버려진다.

**모범 답안:**

```java
// 중앙값 반환 (배열이 정렬되어 있다고 가정)
public int getMedian()
{
    // 중간 인덱스 계산
    int mid = grades.length / 2;

    // 홀수 길이: 정확히 가운데 원소
    if (grades.length % 2 == 1)
    {
        return grades[mid];
    }

    // 짝수 길이: 가운데 두 원소의 평균 (정수 나눗셈)
    return (grades[mid - 1] + grades[mid]) / 2;
}
```

**예시 추적 (홀수 길이, grades = {60, 70, 80, 90, 95}):**

```
grades.length = 5 (홀수)
mid = 5 / 2 = 2

5 % 2 == 1? true → return grades[2] = 80  ✓

배열 시각화:
인덱스: [0]  [1]  [2]  [3]  [4]
값:      60   70  *80*  90   95
                   ↑ 중앙값
```

**예시 추적 (짝수 길이, grades = {60, 70, 85, 90}):**

```
grades.length = 4 (짝수)
mid = 4 / 2 = 2

4 % 2 == 1? false → 짝수 처리
return (grades[1] + grades[2]) / 2
     = (70 + 85) / 2
     = 155 / 2
     = 77  ← 정수 나눗셈! (77.5가 아님)

배열 시각화:
인덱스: [0]  [1]  [2]  [3]
값:      60  *70* *85*  90
              ↑    ↑
         mid-1=1  mid=2
```

**예시 추적 (짝수 길이, 정수 나눗셈 확인, grades = {10, 20, 30, 40, 50, 60}):**

```
grades.length = 6 (짝수)
mid = 6 / 2 = 3

return (grades[2] + grades[3]) / 2
     = (30 + 40) / 2
     = 70 / 2
     = 35  ✓

배열 시각화:
인덱스: [0]  [1]  [2]  [3]  [4]  [5]
값:      10   20  *30* *40*  50   60
              mid-1=2  mid=3
```

**채점 기준 (4점):**

| 배점 | 기준 |
|------|------|
| +1 | 중간 인덱스를 `grades.length / 2`로 올바르게 계산 |
| +1 | 홀수 길이 처리: `grades[mid]` 반환 |
| +1 | 짝수 길이 처리: `(grades[mid - 1] + grades[mid]) / 2` 반환 |
| +1 | 정수 나눗셈으로 올바르게 처리 (int / int = int) |

**자주 하는 실수:**

1. **홀짝 구분을 안 함:** 모든 경우에 `grades[mid]`만 반환하면 짝수 길이에서 틀린다.
2. **짝수일 때 인덱스 계산 오류:** `mid`와 `mid + 1`을 사용하는 실수. 길이 4일 때 `mid = 2`이므로 `grades[2]`와 `grades[3]`이 가운데 두 원소인 것처럼 보이지만, 사실 `grades[1]`과 `grades[2]`가 가운데다. `mid - 1`과 `mid`가 올바르다.
   - 길이 4: 인덱스 0,**1,2**,3 → mid=2이므로 `grades[mid-1]=grades[1]`, `grades[mid]=grades[2]`
3. **double 나눗셈 사용:** `(grades[mid-1] + grades[mid]) / 2.0`으로 하면 반환 타입이 double이 되어 컴파일 에러 (메서드 반환 타입이 `int`). 문제에서도 "int division"이라고 명시했다.
4. **정렬을 직접 하려고 함:** precondition에 "assumes grades is sorted"라고 명시되어 있으므로 정렬 코드를 작성할 필요가 없다. 시간 낭비이며 불필요한 코드로 실수를 유발할 수 있다.
5. **길이 1인 배열 미고려:** 길이 1이면 `mid = 0`, 홀수이므로 `grades[0]` 반환. 정상 작동하는지 머릿속으로 확인하자.

**대안 풀이 (if-else 없이 한 줄):**

```java
public int getMedian()
{
    int mid = grades.length / 2;
    // 홀수면 grades[mid], 짝수면 두 원소 평균
    // 홀수일 때 grades[mid-1]과 grades[mid]가 같은 값은 아니지만,
    // 이 한줄 방식은 홀수일 때도 동작... 하지 않음.
    // 따라서 if-else가 가장 안전하고 명확함.
}
```

결론: **if-else로 홀짝을 분기하는 것이 가장 안전하고 가독성이 높다.** 시험에서는 명확한 코드가 가장 좋다.

### FRQ Q4: TreasureMap (9 Points)

---

#### Part (a) — `countTreasures`, `treasuresPerRow` (5점)

**이 문제가 뭘 원하는가:**

`boolean[][] map`에서 `true`는 보물이 있는 칸을 의미한다.
- `countTreasures()`: 전체 맵에서 보물(true)의 총 개수를 반환
- `treasuresPerRow()`: 각 행(row)별 보물 개수를 담은 1차원 `int[]` 배열을 반환. `result[i]`는 `map[i]` 행의 보물 수.

**풀이 전략:**

- `countTreasures`: 2중 for 루프(또는 for-each)로 모든 칸을 순회하며 `true`인 칸을 카운트.
- `treasuresPerRow`: `map.length` 크기의 int 배열을 만들고, 각 행을 순회하며 `true` 개수를 카운트하여 해당 인덱스에 저장.

**모범 답안:**

```java
// 전체 보물 수 반환
public int countTreasures()
{
    int count = 0;

    // 모든 행 순회
    for (boolean[] row : map)
    {
        // 각 행의 모든 칸 순회
        for (boolean cell : row)
        {
            // true이면 보물 → 카운트 증가
            if (cell)
            {
                count++;
            }
        }
    }

    return count;
}

// 행별 보물 수를 담은 배열 반환
public int[] treasuresPerRow()
{
    // 행의 수만큼 결과 배열 생성
    int[] counts = new int[map.length];

    // 각 행을 순회
    for (int r = 0; r < map.length; r++)
    {
        // 해당 행의 각 칸을 순회
        for (int c = 0; c < map[r].length; c++)
        {
            if (map[r][c])
            {
                counts[r]++;
            }
        }
    }

    return counts;
}
```

**예시 추적 (map 시각화):**

```
map:
        col0  col1  col2  col3
row0: [ true  false true  false ]
row1: [ false false false true  ]
row2: [ true  true  true  false ]

countTreasures():
  row0: true(+1), false, true(+1), false → 2
  row1: false, false, false, true(+1)    → 1
  row2: true(+1), true(+1), true(+1), false → 3
  총 count = 2 + 1 + 3 = 6  ✓

treasuresPerRow():
  counts = new int[3] → {0, 0, 0}
  r=0: map[0][0]=true(+1), [0][1]=false, [0][2]=true(+1), [0][3]=false → counts[0] = 2
  r=1: map[1][0]=false, [1][1]=false, [1][2]=false, [1][3]=true(+1)   → counts[1] = 1
  r=2: map[2][0]=true(+1), [2][1]=true(+1), [2][2]=true(+1), [2][3]=false → counts[2] = 3

  return {2, 1, 3}  ✓
```

**채점 기준 (5점):**

| 배점 | 기준 |
|------|------|
| +1 | `countTreasures` — 2D 배열 전체 순회 |
| +1 | `countTreasures` — `true`인 칸 카운트 + 반환 |
| +1 | `treasuresPerRow` — `new int[map.length]` 결과 배열 생성 |
| +1 | `treasuresPerRow` — 각 행을 순회하며 `true` 카운트 |
| +1 | `treasuresPerRow` — 행별 카운트를 올바른 인덱스에 저장 + 반환 |

**자주 하는 실수:**

1. **`if (cell == true)` 작성:** 동작은 하지만, `if (cell)`이 더 간결하고 관용적이다. `boolean` 변수를 `true`와 비교하는 것은 불필요하다. (감점은 아니지만 비추천)
2. **`map[0].length` 하드코딩:** `map[r].length` 대신 `map[0].length`를 써도 직사각형 배열이면 동작하지만, `map[r].length`가 더 안전하다.
3. **`treasuresPerRow`에서 결과 배열 크기를 `map[0].length`로 설정:** 행의 수는 `map.length`이고, 열의 수는 `map[0].length`이다. 행별 카운트이므로 `map.length`가 맞다.
4. **`countTreasures`에서 count를 반환하지 않음:** 메서드 끝에 `return count;`를 빼먹는 실수.
5. **int 배열 초기화 걱정:** Java에서 `new int[n]`은 자동으로 0으로 초기화되므로, 별도로 0을 넣을 필요 없다.

**대안 풀이 (countTreasures를 treasuresPerRow 활용):**

```java
public int countTreasures()
{
    int[] perRow = treasuresPerRow();
    int total = 0;
    for (int c : perRow)
    {
        total += c;
    }
    return total;
}
```

이 방법은 `treasuresPerRow()`를 재사용하므로 코드 중복을 줄인다. 시험에서 이렇게 작성해도 만점이다. 다만 별도의 배열을 생성하므로 메모리 효율은 약간 떨어진다.

#### Part (b) — `hasUnevenDistribution` (4점)

**이 문제가 뭘 원하는가:**

어떤 행이 다른 행보다 보물이 더 많거나 적으면 `true`(고르지 않은 분포), 모든 행의 보물 수가 동일하면 `false`(균등 분포)를 반환한다.

다시 말해: **모든 행의 보물 수가 같으면 `false`, 하나라도 다르면 `true`.**

**풀이 전략:**

`treasuresPerRow()`를 호출하여 행별 보물 수 배열을 얻은 뒤, 그 배열에서 **최솟값과 최댓값이 다른지** 확인한다. 최솟값 != 최댓값이면 고르지 않은 분포이다.

또는 더 간단하게: 첫 번째 행의 보물 수와 다른 행을 하나라도 찾으면 `true`를 반환하는 방법도 있다.

**모범 답안 (접근법 1: 첫 번째 원소와 비교):**

```java
// 행별 보물 분포가 고르지 않으면 true
public boolean hasUnevenDistribution()
{
    // 행별 보물 수 배열 얻기
    int[] counts = treasuresPerRow();

    // 첫 번째 행의 보물 수를 기준으로 삼음
    for (int i = 1; i < counts.length; i++)
    {
        // 어떤 행이든 첫 번째 행과 다르면 → 고르지 않음
        if (counts[i] != counts[0])
        {
            return true;
        }
    }

    // 모든 행이 같은 수의 보물 → 균등 분포
    return false;
}
```

**대안 풀이 (접근법 2: 최솟값/최댓값 비교):**

```java
public boolean hasUnevenDistribution()
{
    int[] counts = treasuresPerRow();

    // 최솟값과 최댓값 찾기
    int min = counts[0];
    int max = counts[0];

    for (int i = 1; i < counts.length; i++)
    {
        if (counts[i] < min)
            min = counts[i];
        if (counts[i] > max)
            max = counts[i];
    }

    // 최솟값과 최댓값이 다르면 고르지 않은 분포
    return min != max;
}
```

**예시 추적 (접근법 1, 고르지 않은 경우):**

```
map:
row0: [ true  false true  false ]  → 보물 2개
row1: [ false false false true  ]  → 보물 1개
row2: [ true  true  true  false ]  → 보물 3개

counts = treasuresPerRow() = {2, 1, 3}

i=1: counts[1]=1, counts[0]=2 → 1 != 2? true → return true  ✓

결과: true (행별 보물 수가 2, 1, 3으로 다름)
```

**예시 추적 (접근법 1, 균등한 경우):**

```
map:
row0: [ true  false true  ]  → 보물 2개
row1: [ false true  true  ]  → 보물 2개
row2: [ true  true  false ]  → 보물 2개

counts = treasuresPerRow() = {2, 2, 2}

i=1: counts[1]=2, counts[0]=2 → 2 != 2? false → 다음
i=2: counts[2]=2, counts[0]=2 → 2 != 2? false → 다음
루프 종료

return false  ✓

결과: false (모든 행이 보물 2개로 동일)
```

**예시 추적 (접근법 2, min/max 비교):**

```
counts = {2, 1, 3}

min = 2, max = 2 (초기값)
i=1: counts[1]=1 < min=2 → min = 1
     counts[1]=1 > max=2? false
i=2: counts[2]=3 < min=1? false
     counts[2]=3 > max=2 → max = 3

min = 1, max = 3
return 1 != 3 → return true  ✓
```

**채점 기준 (4점):**

| 배점 | 기준 |
|------|------|
| +1 | `treasuresPerRow()` 호출하여 행별 카운트 배열 획득 |
| +1 | 배열의 원소들을 비교하는 로직 구현 |
| +1 | 다른 값이 있으면 `true` 반환하는 조건 올바르게 작성 |
| +1 | 모든 값이 같으면 `false` 반환 |

**자주 하는 실수:**

1. **`treasuresPerRow()`를 호출하지 않고 직접 2D 순회:** 동작은 하지만, 문제에서 "may call treasuresPerRow"라고 힌트를 주었고, 이미 구현한 메서드를 활용하는 것이 AP 시험에서 높은 점수를 받는 핵심이다. 코드 재사용은 좋은 설계의 지표다.
2. **인접한 행만 비교:** `counts[i] != counts[i+1]`으로 인접 비교만 하면 `{2, 3, 2}`에서도 잡히긴 하지만, 첫 번째 원소와 비교하는 것이 더 간결하고 확실하다.
3. **`==` 대신 `!=` 혼동:** `counts[i] == counts[0]`이면 같은 것이므로 `return true`가 아니라 "continue". `!=`일 때 `return true`가 맞다. 논리를 거꾸로 쓰는 실수를 주의하자.
4. **빈 맵 또는 행이 1개인 경우:** 행이 1개이면 루프가 실행되지 않고 바로 `return false`가 된다. 이것은 올바르다 -- 행이 하나뿐이면 "다른 행과 차이"가 있을 수 없으므로 균등이다.
5. **반환 타입 실수:** `return true`/`return false` 대신 `return 1`/`return 0` 등을 쓰면 컴파일 에러. Java에서 boolean과 int는 호환되지 않는다.
