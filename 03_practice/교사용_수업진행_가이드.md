## Part 1: AP CSA 시험 3분 브리핑

### 시험이 뭔지

AP Computer Science A(AP CSA)는 미국 College Board가 운영하는 대학 수준 시험이다. Java 프로그래밍 언어로 코드를 **읽고, 분석하고, 작성하는 능력**을 평가한다.

### 왜 보는지

- **대학 학점 인정**: 4점 이상이면 대부분의 미국 대학에서 Computer Science 입문 과목 학점을 인정해준다 (1학기, 3~4학점).
- **대학 입시**: 특히 이공계 지원 시 CS 관련 AP를 보유하면 경쟁력이 된다.
- **비용 절감**: 대학에서 수강하면 수백만 원짜리 과목을 AP 시험 한 번으로 대체 가능.

### 점수 체계

| 점수 | 의미 | 대략적 위치 |
|------|------|------------|
| 5 | Extremely Well Qualified | 상위 약 25% |
| 4 | Well Qualified | 상위 약 45% |
| 3 | Qualified | 상위 약 67% |
| 2 | Possibly Qualified | |
| 1 | No Recommendation | |

- **대부분의 대학이 요구하는 최소 점수**: 4점
- **목표**: 4점 확보, 가능하면 5점

### 시험 구조

| 섹션 | 형식 | 문제 수 | 시간 | 배점 비중 |
|------|------|--------|------|----------|
| Section I | MCQ (4지선다) | 42문제 | 90분 | 55% |
| Section II | FRQ (서술형 코딩) | 4문제 | 90분 | 45% |

- **오답 감점 없음**: 모르면 무조건 찍어야 한다.
- **Quick Reference 제공**: 시험 중 String, Math, ArrayList 등의 메서드 시그니처가 적힌 참고 자료가 주어진다.
- **디지털 시험**: Bluebook 앱으로 치르는 컴퓨터 기반 시험이다.
- **계산기 사용 불가**.

### 2025-26 개편 핵심 (반드시 숙지!)

2025-26 시험부터 AP CSA 교육과정이 대폭 개편되었다:

| 항목 | 변경 내용 |
|------|----------|
| **상속(Inheritance) 제거** | 기존 Unit 9(Inheritance/Polymorphism)가 시험 범위에서 완전 제거됨. extends, super, override, abstract, interface 등이 더 이상 출제되지 않음 |
| **파일 I/O 추가** | File, Scanner를 이용한 파일 읽기가 새로 추가됨. hasNext(), split() 등 |
| **4 Unit 구조** | 기존 10개 Unit이 4개 Unit으로 재편: Unit 1(Using Objects and Methods), Unit 2(Selection and Iteration), Unit 3(Classes and Objects), Unit 4(Data Structures) |
| **FRQ 변경** | Q1: Methods/Control, Q2: Class Design, Q3: ArrayList (only), Q4: 2D Array |

**교사 주의**: 인터넷에서 AP CSA 자료를 검색하면 상속 관련 내용이 포함된 옛날 자료가 많다. 2025-26 이후 시험에서는 상속이 나오지 않으므로, 상속을 가르치는 데 시간을 쓰지 않는다.

## Part 2: 수업 원칙

이 5가지 원칙은 매 수업마다 의식적으로 적용해야 한다.

### 원칙 1: 코드를 직접 치게 하라

- **눈으로 보는 것과 손으로 치는 것은 완전히 다르다.** 교사가 코드를 보여주며 설명하면 학생은 "이해했어요"라고 말하지만, 직접 쓰라고 하면 못 쓴다.
- 교사는 코드를 보여준 뒤 **반드시 학생이 직접 타이핑**하게 한다.
- 복사-붙여넣기를 시키지 않는다. 한 줄씩 직접 치면서 몸에 익힌다.
- "저도 처음에는 코드를 보고 치는 거부터 시작했어요"라고 말해주면 학생이 부담을 덜 느낀다.

### 원칙 2: 코드 tracing을 매 수업 하라

- AP CSA 시험의 **37-53%가 코드 분석(Analyze Code) 문제**이다. 코드를 읽고 출력을 정확히 예측하는 능력이 곧 점수다.
- 매 수업 시작이나 중간에 **5-10분 코드 tracing 드릴**을 한다.
- 방법: 코드를 보여주고, 종이에 변수 추적표를 그려서 한 줄씩 실행시킨다.
- 참고 자료: `코드트레이싱_워크시트.md`에 Level 1~5 문제가 준비되어 있다.

### 원칙 3: Quick Reference를 항상 옆에 두게 하라

- Quick Reference는 **실제 시험장에서도 제공**되는 공식 참고 자료이다.
- 수업 중에도 항상 Quick Reference를 화면이나 인쇄물로 열어두게 한다.
- 메서드 이름이 기억 안 나면 Quick Reference에서 찾는 습관을 들인다.
- 이것은 커닝이 아니라 **시험에서 허용되는 정당한 전략**이다.
- 참고: `QuickReference_한국어해설.md`에 한국어 해설 버전이 있다.

### 원칙 4: 개념보다 패턴

- Java 완전 초보에게 "왜 public static void main이 필요한지"를 설명하면 혼란만 가중된다.
- **"이렇게 쓰면 이렇게 된다"를 먼저 알려준다.** 이유는 나중에 자연스럽게 알게 된다.
- 예: `System.out.println("Hello");`는 "화면에 Hello를 출력하는 주문"이라고 말한다. println의 의미, System과 out의 관계 등은 나중에 설명한다.
- 학생이 "왜요?"라고 물으면: "지금은 패턴으로 외워두고, 3회차쯤 되면 자연스럽게 이해될 거예요."

### 원칙 5: 에러를 두려워하지 않게 하라

- 초보 학생은 빨간 에러 메시지를 보면 공포를 느낀다. **에러는 실패가 아니라 피드백**이라는 인식을 심어줘야 한다.
- "에러가 나면 좋은 거예요. 컴퓨터가 뭐가 틀렸는지 알려주니까요."
- 에러 메시지를 같이 읽는 습관: "몇 번째 줄에서 무슨 에러가 났는지 봅시다."
- **일부러 에러를 내보는 활동**도 좋다: 세미콜론을 빼보고, 타입을 잘못 써보고, 어떤 에러가 나는지 관찰한다.

| 상황 | 대처 |
|------|------|
| && 와 \|\| 헷갈림 | "&&는 '그리고', \|\|는 '또는'. 일상 대화에 대입해보세요. '비 오고(&&) 우산 없으면' vs '비 오거나(\|\|) 눈 오면'" |
| De Morgan이 이해 안 됨 | 진리표를 직접 그리게 한다. 4줄밖에 안 되니 시간이 오래 걸리지 않는다. 직접 확인하면 납득한다 |
| if-else if 순서 혼동 | "위에서 아래로 내려가면서 처음 true인 곳에서 멈춘다"를 여러 예제로 반복 |
| == vs = 혼동 | "= 한 개는 넣는다(assign), == 두 개는 물어본다(compare). 대입은 화살표 →, 비교는 저울 ⚖" |

#### 수업 후 확인 질문 3개

1. `!(true && false)`의 결과는? → 정답: true
2. `!(A || B)`를 De Morgan 법칙으로 바꾸면? → 정답: `!A && !B`
3. `&&`에서 왼쪽이 false이면 오른쪽은 어떻게 되나요? → 정답: 평가하지 않음 (단축 평가)

교사: "2D 배열은 표(grid) 형태의 데이터예요. 행과 열이 있습니다."

      int[][] grid = {
          {1, 2, 3},
          {4, 5, 6},
          {7, 8, 9}
      };

      // 또는

교사: "접근 방법: grid[행][열]"
      grid[0][0] → 1 (첫째 행, 첫째 열)
      grid[1][2] → 6 (둘째 행, 셋째 열)
      grid[2][0] → 7 (셋째 행, 첫째 열)

교사: "크기 확인:"
      grid.length      → 3 (행 수)
      grid[0].length    → 3 (열 수)
      "★ grid.length는 행, grid[0].length는 열. 거꾸로 외우면 안 돼요!"

교사: "순회 방법 2가지:"

      // Row-major (행 우선, 가장 흔함)
      for (int r = 0; r < grid.length; r++) {
          for (int c = 0; c < grid[0].length; c++) {
              System.out.print(grid[r][c] + " ");
          }
          System.out.println();
      }
      // 출력: 1 2 3 / 4 5 6 / 7 8 9

      // Column-major (열 우선)
      for (int c = 0; c < grid[0].length; c++) {
          for (int r = 0; r < grid.length; r++) {
              System.out.print(grid[r][c] + " ");
          }
          System.out.println();
      }
      // 출력: 1 4 7 / 2 5 8 / 3 6 9

교사: "Row-major는 가로로 읽기, Column-major는 세로로 읽기.
      시험에서 '이 코드의 출력은?'이라고 물으면 순회 방향을 먼저 확인하세요."

교사: "데이터에서 원하는 값을 찾는 것이 '탐색'이에요."

      // Linear Search (선형 탐색): 처음부터 끝까지 하나씩
      for (int i = 0; i < arr.length; i++) {
          if (arr[i] == target) return i;
      }
      return -1;  // 못 찾음

      // Binary Search (이진 탐색): 반씩 줄여가며 찾기
      // ★ 전제 조건: 배열이 정렬되어 있어야 함!
      int lo = 0, hi = arr.length - 1;
      while (lo <= hi) {
          int mid = (lo + hi) / 2;
          if (arr[mid] == target) return mid;
          else if (arr[mid] < target) lo = mid + 1;
          else hi = mid - 1;
      }
      return -1;

교사: "★ Binary Search의 핵심 함정: 배열이 정렬되어 있어야만 써요!

## Part 4: 5점 달성을 위한 교수 전략

### 4점 vs 5점의 차이

| 구분 | 4점 학생 | 5점 학생 |
|------|---------|---------|
| **MCQ** | 42문제 중 25~29개 정답 (~60-69%) | 42문제 중 33개+ 정답 (~80%+) |
| **FRQ** | 25점 중 15~17점 (~60-68%) | 25점 중 20점+ (~80%+) |
| **코드 tracing** | 대부분 맞지만 복잡한 중첩에서 실수 | 거의 100% 정확 |
| **함정 회피** | 흔한 함정에 가끔 걸림 | 함정을 인식하고 피함 |
| **FRQ 시간** | 일부 문제 미완성 | 4문제 모두 완성 + 검토 시간 |
| **실수 패턴** | 반복적인 실수가 2~3개 남아 있음 | 오답노트로 실수를 거의 제거 |

핵심: **4점과 5점의 차이는 실력이 아니라 정확성과 시간 관리**이다.

### 5점 달성 핵심 전략

#### 1. 코드 tracing 정확도 95%+

- 매 수업 시작 5분에 tracing 드릴을 한다.
- `코드트레이싱_워크시트.md`의 Level 4~5를 반복 풀게 한다.
- **규칙**: 머릿속으로 하지 않는다. 반드시 종이에 변수 추적표를 그린다.
- 목표: Level 5 문제를 90% 이상 맞추면 5점 수준.

#### 2. FRQ 시간 관리

| 문제 | 권장 시간 | 전략 |
|------|----------|------|
| Q1 (Methods/Control, 7점) | 15분 | 가장 쉬움. 빠르게 끝내서 뒤에 시간 확보 |
| Q2 (Class Design, 7점) | 20분 | 구조 패턴을 외워서 빠르게: private → 생성자 → getter → 메서드 |
| Q3 (ArrayList, 5점) | 15분 | 삭제 패턴만 확실히 |
| Q4 (2D Array, 6점) | 15분 | 중첩 루프 패턴 |
| 검토 | 25분 | 세미콜론, 중괄호, 리턴타입 확인 |

검토 시간 25분이 핵심이다. 4점 학생은 검토 시간이 없고, 5점 학생은 검토로 실수를 잡는다.

#### 3. 실수 패턴 파악 — 오답노트

- 틀린 문제마다 **왜 틀렸는지** 한 줄로 기록하게 한다.
- 3회 이상 같은 이유로 틀리면 그것이 학생의 "실수 패턴"이다.
- 시험 직전에 자기 실수 패턴 목록을 한 번 읽는다.

오답노트 형식:
```
날짜 | 문제 | 내 답 | 정답 | 틀린 이유
4/10 | MCQ #15 | B | D | substring의 to가 미포함인 것을 잊음
4/10 | FRQ Q3 | - | - | 삭제 시 i--를 안 함
```

#### 4. 모의시험 5세트 전부 풀기

- `05_mock_exams/` 디렉토리와 외부 자료(CrackAP, AP Classroom 등)에서 모의시험 5세트를 구한다.
- 시험 2주 전부터 일주일에 2세트씩 풀되, 매번 시험과 동일한 환경(시간 제한, Quick Reference만)으로 진행한다.
- 채점 후 오답 분석이 풀기보다 더 중요하다.

### 흔한 실수 TOP 10과 교정법

#### 실수 1: `substring(from, to)`에서 to 포함으로 착각

- **왜 틀리나**: 일상적 감각에서 "1부터 3까지"는 1, 2, 3이지만, Java에서 substring(1, 3)은 인덱스 1, 2만 포함.
- **교정 설명**: "substring(from, to)는 [from, to) — 수학의 반개구간이에요. to는 빠져요."
- **연습**: 문자열 아래에 인덱스를 적고, 원하는 부분을 형광펜으로 칠하게 한다. 10번 반복하면 체화된다.

#### 실수 2: `==`로 String 비교

- **왜 틀리나**: 초보는 == 가 "같은지 비교"라고 직관적으로 이해한다.
- **교정 설명**: "int끼리는 == 써도 돼요. 그런데 String은 객체라서 .equals()를 써야 합니다. '객체는 .equals()'를 주문처럼 외우세요."
- **연습**: 코드에서 String 비교가 나올 때마다 물어보기: "이거 == 써야 돼요, .equals() 써야 돼요?"

#### 실수 3: 정수 나눗셈에서 소수점 기대

- **왜 틀리나**: 수학에서 7 / 2 = 3.5이니까 Java에서도 그럴 거라고 생각.
- **교정 설명**: "int끼리 나누면 소수점은 버려져요. 3.5가 아니라 3이 돼요. 소수점이 필요하면 (double)을 붙이세요."
- **연습**: 정수 나눗셈 문제를 10개 연속으로 풀게 한다.

#### 실수 4: ArrayList 삭제 시 인덱스 밀림 무시

- **왜 틀리나**: 삭제 후 뒤 원소들이 앞으로 밀린다는 것을 잊음.
- **교정 설명**: 종이에 리스트를 그리고, 삭제할 때마다 원소들이 왼쪽으로 이동하는 것을 눈으로 확인시킨다.
- **연습**: "뒤에서부터 삭제하기" 또는 "삭제 후 i--" 패턴을 3번 이상 코딩하게 한다.

#### 실수 5: 배열 인덱스 off-by-one

- **왜 틀리나**: `<=` vs `<`, 마지막 인덱스가 length인지 length-1인지 혼동.
- **교정 설명**: "배열 크기가 5면 인덱스는 0, 1, 2, 3, 4. length는 5이지만 마지막 인덱스는 4예요. 항상 length - 1!"
- **연습**: `< length` vs `<= length`가 포함된 코드 tracing 5문제.

#### 실수 6: enhanced for에서 원소 수정 시도

- **왜 틀리나**: for (int x : arr)에서 x를 바꾸면 원래 배열도 바뀐다고 착각.
- **교정 설명**: "for-each의 변수는 복사본이에요. 원본을 바꾸려면 arr[i] = 값으로 해야 해요."
- **연습**: "이 코드는 배열을 바꾸나요, 안 바꾸나요?" 문제 5개.

#### 실수 7: `null` 체크를 안 함

- **왜 틀리나**: 변수가 null일 수 있다는 생각 자체를 안 함.
- **교정 설명**: "객체 변수는 null일 수 있어요. null인 상태에서 메서드를 호출하면 NullPointerException이 터져요."
- **연습**: null 관련 MCQ 3문제.

#### 실수 8: De Morgan 법칙 적용 실수

- **왜 틀리나**: NOT을 분배할 때 && ↔ || 전환을 잊음.
- **교정 설명**: "NOT이 안으로 들어가면 AND와 OR가 바뀌고, 각 조건도 뒤집혀요."
- **연습**: De Morgan 변환 문제 10개를 기계적으로 풀게 한다.

#### 실수 9: 2D 배열에서 행/열 혼동

- **왜 틀리나**: grid.length가 행인지 열인지 헷갈림.
- **교정 설명**: "첫 번째 length는 행(세로 칸수), [0].length는 열(가로 칸수). 아파트 5층 3호에서 층이 먼저인 것처럼."
- **연습**: 2D 배열 크기 확인 문제 5개.

#### 실수 10: FRQ에서 빈칸을 남김

- **왜 틀리나**: 완벽한 답을 못 쓰면 아예 안 쓰는 심리.
- **교정 설명**: "FRQ는 부분점수제예요. for 루프만 써도 1점, private 변수만 써도 1점. 빈칸은 0점이지만 뭐라도 쓰면 점수가 나와요."
- **연습**: 일부러 시간을 부족하게 주고, 부분 답안이라도 쓰는 연습을 한다.

## Part 5: 자주 묻는 교사 질문

### Q1: "학생이 for 루프를 이해 못 하면 어떻게 해요?"

for 루프를 처음 접하는 학생은 세 부분(초기화, 조건, 증감)이 한 줄에 있는 것 자체가 혼란스러울 수 있다.

**접근법:**
1. **while 루프부터 시작한다.** while은 구조가 단순해서 "조건이 true인 동안 반복"이라는 개념을 먼저 잡을 수 있다.
2. while로 1~5를 출력하는 코드를 쓴 후, 같은 코드를 for로 바꿔보여준다.
3. for의 세 부분을 "시작 / 끝까지? / 한 칸씩"으로 풀어서 설명한다.
4. 그래도 안 되면 **구체적인 숫자를 대입해서 한 줄씩 실행**시킨다. "i가 0일 때, 조건 0 < 5는 true이니까 안으로 들어가요..."

for 루프는 4회차에서 배우고, 이후 6~10회차에서 계속 반복하므로, 4회차에서 완벽히 이해 못 해도 반복적인 노출로 체화된다.

### Q2: "FRQ를 어떻게 채점하죠?"

College Board가 매년 공식 채점 기준(Scoring Guidelines)을 공개한다.

**채점 방법:**
1. `02_frq_archive/` 폴더에 연도별 FRQ 문제와 채점 기준이 있다.
2. `FRQ_풀이해설.md`에도 채점 기준이 정리되어 있다.
3. 채점 기준은 "이것이 포함되어 있으면 +1점" 형태의 체크리스트이다.

**예시 (Q2 Class Design):**
- private 인스턴스 변수 선언 → +1점
- 생성자에서 올바르게 초기화 → +1점
- 메서드 A의 반환값이 올바름 → +1점
- 메서드 B의 로직이 올바름 → +1점
- ...

세미콜론 누락, 중괄호 불일치 같은 사소한 문법 에러는 보통 감점하지 않는다 (다만, 의미가 달라지는 에러는 감점).

### Q3: "10회로 5점이 가능한가요?"

**솔직한 답변:** 학생의 기본 자질과 숙제 성실도에 달려 있다.

- **수학적 사고력이 있고, 숙제를 성실히 하는 학생**: 10회 + 충분한 자습으로 5점 가능하다.
- **일반적인 학생**: 10회로 4점은 충분히 가능하다. 5점을 노리려면 추가 2~3회 또는 자습량을 늘려야 한다.
- **학습이 느린 학생**: 10회로 3~4점. 기초에 더 시간을 써야 하므로 15회가 이상적이다.

**핵심은 숙제**이다. 수업 시간만으로는 부족하고, 수업 사이에 최소 주 2~3시간의 자습이 필요하다. 숙제를 안 하면 다음 수업에서 이전 내용을 복습하느라 진도가 밀린다.

### Q4: "학생이 코딩 자체를 싫어하면?"

이 경우는 동기 부여가 핵심이다.

**접근법:**
1. **작은 성공 경험**: 아주 쉬운 문제부터 시작해서 "나도 할 수 있다"는 감각을 준다. CodingBat의 Warmup-1이 이 목적에 좋다.
2. **실용적 연결**: "게임, 앱, 인스타 필터 — 이런 것 다 코드로 만들어요."라고 연결한다.
3. **시험 점수에 집중**: "코딩을 좋아할 필요 없어요. 시험에서 4점만 받으면 대학 학점이 인정되잖아요. 그걸 위해 10번만 하는 거예요."
4. **지루한 반복 줄이기**: 같은 유형을 10문제 풀리지 말고, 3문제 풀고 다른 유형으로 넘어간다.
5. **칭찬 많이**: 작은 진전도 인정한다. "저번 주에 if문도 못 쓰셨는데, 오늘 for 루프를 쓰셨어요. 엄청 빠른 속도예요."

마지막 수단: 학생이 정말 코딩 자체를 거부하면, 코드 tracing에 집중한다. 코드를 "쓰는" 것은 최소화하고 "읽는" 연습을 극대화하면 MCQ에서 점수를 올릴 수 있다.

### Q5: "수업 순서를 바꿔도 되나요?"

**원칙적으로는 바꾸지 않는 것이 좋다.** 각 회차는 이전 회차의 내용을 전제로 한다.

- 1~2회(변수, String, Math)는 반드시 먼저
- 3~4회(조건문, 반복문)는 1~2회 이후
- 5회(클래스)는 독립적이므로 3~4회와 순서 교환 가능하나, 메서드 개념이 2회에서 나오므로 비추천
- 6~8회(배열, ArrayList, 2D Array)는 순서대로 해야 함
- 9~10회(모의시험)는 반드시 마지막

**바꿔도 되는 경우:**
- 학생이 다른 과목에서 Python 등을 배운 적 있어서 변수/조건문을 이미 아는 경우 → 1~2회를 압축하고, 배열/ArrayList에 시간을 더 준다.
- 학생이 클래스를 특히 어려워하면 → 5회를 두 번(5a, 5b)으로 나누고, 10회를 약점 보강으로 대체한다.

### Q6: "학생이 수업 중에 집중을 못 하면?"

- **코딩 실습 비율을 높인다**: 설명 20분 + 실습 40분이 아니라, 설명 10분 + 실습 20분을 반복한다.
- **10분 단위로 활동을 바꾼다**: 설명 → 실습 → tracing → MCQ → 실습 순환.
- **학생이 직접 설명하게 한다**: "이 코드가 뭘 하는지 저에게 설명해보세요."
- **쉬는 시간을 엄격히 지킨다**: 2시간 연속은 누구나 힘들다. 정시에 10분 쉬기.

### Q8: "Java를 설치하는 것부터 막히면?"

`00_Java_환경설정_가이드.md`에 스크린샷 수준으로 상세한 설치 가이드가 있다. 그래도 문제가 생기면:

1. **IntelliJ가 안 열리는 경우**: 대부분 JDK 미설치. IntelliJ 설치 시 Bundled JDK를 선택하면 해결.
2. **코드가 실행이 안 되는 경우**: main 메서드가 없거나, 클래스 이름과 파일 이름이 다른 경우.
3. **최후의 수단**: 온라인 IDE(replit.com 등)를 사용한다. 설치 없이 브라우저에서 바로 코딩 가능.

### Q9: "수업 전에 어떤 준비를 해야 하나요?"

각 회차의 "수업 전 준비 체크리스트"를 확인하면 되지만, 공통적으로:

1. 해당 Unit의 강의노트를 **한 번 읽는다** (완벽히 이해할 필요 없음, 흐름 파악).
2. 학생 숙제 완료 여부를 확인한다 (수업 시작 전에 물어볼 수 있음).
3. 코드 tracing 워밍업 문제 1~2개를 미리 골라둔다.
4. 실습용 코드 예제를 미리 타이핑해서 확인해둔다 (교사가 수업 중 에러 나면 당황스러움).

### Q10: "과외 비용/횟수 상담은 어떻게?"

이 가이드의 범위 밖이지만, 참고로:

- **10회는 최소 권장 횟수**이다. 학생이 성실하면 10회로 4점 가능하지만, 여유가 있으면 12~15회가 이상적이다.
- **각 수업은 2시간** 기준. 1.5시간으로 줄이면 실습 시간이 부족해진다.
- **수업 간격**: 주 1~2회가 적당. 주 1회면 총 10~15주, 주 2회면 5~8주 소요.
- 시험일(5월 초)로부터 역산하여 시작일을 정한다.
