## 2. MCQ 전략

### 시간 관리
- 42문제 / 90분 = **문제당 약 2분 7초**
- 쉬운 문제는 1분 이내에 풀고, 어려운 문제에 시간을 확보
- 절반(45분) 시점에서 21문제 정도가 정상 페이스
- **Flag 기능** 활용: 어려운 문제는 표시하고 넘긴 뒤, 남는 시간에 재도전

### 코드 Tracing 문제 (가장 중요)
종이에 **변수 추적표**를 그려서 푼다:

```
반복  | i  | j  | arr[i] | 결과
------|----|----|---------|---------
1회   | 0  | 2  | 5      | "5 "
2회   | 1  | 1  | 3      | "5 3 "
...
```

- 머릿속으로 하면 반드시 실수한다. **무조건 종이에 쓸 것.**
- 특히 for 루프 조건의 `<` vs `<=`, 초기값, 증감식을 정확히 확인

### 소거법
1. 선택지 4개 중 **확실히 틀린 것** 먼저 제거
2. 남은 2-3개에서 코드를 다시 tracing
3. 특히 "off-by-one" 오류를 유발하는 선택지에 주의

### 감점 없음 → 무조건 찍기!
- 빈 칸으로 두면 0점, 찍으면 25% 확률로 맞음
- **절대로 빈 칸을 남기지 말 것**

### 자주 나오는 유형별 공략

#### (1) "코드 출력 예측" (출제 빈도 1위)
- 변수 추적표를 그리며 한 줄씩 실행
- `System.out.print` vs `System.out.println` 구분 (줄바꿈 여부)
- String 연결 시 `+` 연산자 우선순위 주의: `"a" + 1 + 2` → `"a12"` (왼→오)

#### (2) "코드 오류 찾기"
세 가지 오류 유형을 구분하라:

| 유형 | 설명 | 예시 |
|------|------|------|
| **Compile Error** | 문법 오류, 컴파일 자체 불가 | 타입 불일치, 세미콜론 누락 |
| **Runtime Error** | 실행 중 크래시 | ArrayIndexOutOfBounds, NullPointer |
| **Logic Error** | 실행은 되지만 결과가 틀림 | `<` 대신 `<=` 사용 |

#### (3) "빈칸 채우기"
- 빈칸 **앞뒤 코드**를 먼저 분석 → 반환 타입, 매개변수 타입 확인
- 메서드 호출이면 Quick Reference에서 시그니처 확인

#### (4) "동치 코드" (같은 동작을 하는 코드 고르기)
- **De Morgan의 법칙**: `!(A && B)` == `!A || !B`, `!(A || B)` == `!A && !B`
- **루프 변환**: `for` ↔ `while` 변환 시 초기화/조건/증감 위치 확인
- **단축 평가(short-circuit)**: `&&`는 앞이 false면 뒤를 안 봄, `||`는 앞이 true면 뒤를 안 봄

#### (5) File/Scanner 문제 (2025-26 신규!)
- `new Scanner(new File("파일명"))` 패턴이 핵심
- `throws IOException` 선언이 반드시 필요 (없으면 컴파일 에러)
- `hasNext()`로 파일 끝 확인, `nextLine()`으로 한 줄 읽기
- **함정**: `nextInt()` 후 `nextLine()` 호출하면 남은 개행문자를 먼저 소비 → 빈 문자열 반환
- MCQ에서 Scanner 코드의 출력 예측, 올바른 생성 방식 선택 등이 출제 가능

#### (6) 재귀 트레이싱 (MCQ 전용)
- 재귀 코드를 **작성**할 필요 없음 — **추적만**
- 호출 스택을 종이에 그려라: `f(5)` → `f(4)` → `f(3)` → ... → base case → 역순으로 반환
- base case를 먼저 확인하고, 어떤 값에서 멈추는지 파악
- 재귀 호출이 2번인 경우(피보나치 등) 트리 형태로 그리면 명확

## 3. FRQ 전략

### 문제별 구조와 배점

#### Q1: Methods and Control Structures (7점)
- **Part A (4점)** + **Part B (3점)**
- 주로 String 처리, 조건문, 반복문 조합
- **필수 암기**: `substring()`, `indexOf()`, `length()`, `equals()`, `compareTo()`
- Part B는 Part A의 메서드를 호출해서 풀 수 있음 (Part A를 못 풀어도 호출은 가능)

#### Q2: Class Design (7점)
- **완전한 클래스를 처음부터 작성**
- 반드시 포함할 것:
  - `private` 인스턴스 변수
  - 생성자 (매개변수로 초기화)
  - 요구된 메서드들
- 실수 방지: `public` vs `private` 접근제어자, `this` 키워드

#### Q3: ArrayList (5점)
- ArrayList 탐색, 수정, 삭제 문제
- **핵심 함정**: 삭제 시 인덱스가 앞으로 밀림!
  ```java
  // 잘못된 코드 (요소를 건너뜀)
  for (int i = 0; i < list.size(); i++) {
      if (조건) list.remove(i);  // i번 삭제 후 i+1이 i가 됨
  }
  // 올바른 방법 1: 뒤에서부터
  for (int i = list.size() - 1; i >= 0; i--) {
      if (조건) list.remove(i);
  }
  // 올바른 방법 2: 삭제 시 i-- 추가
  ```

#### Q4: 2D Array (6점)
- 중첩 루프로 2D 배열 탐색/처리
- Row-major 순회:
  ```java
  for (int r = 0; r < arr.length; r++) {
      for (int c = 0; c < arr[0].length; c++) {
          // arr[r][c] 처리
      }
  }
  ```
- `arr.length` = 행 수, `arr[0].length` = 열 수

### 부분점수 전략
- AP FRQ는 **부분점수제**. 완벽하지 않아도 맞는 부분은 점수를 준다.
- **빈 칸으로 두지 말 것!** 아는 만큼이라도 작성
- 예: 루프 구조만 맞아도 1점, 조건문만 맞아도 1점
- 메서드 시그니처(반환타입, 이름, 매개변수)가 문제에 주어지면 그대로 복사

### 컴파일 에러 방지 체크리스트
- [ ] 모든 문장 끝에 세미콜론 `;`
- [ ] 여는 중괄호 `{`에 대응하는 닫는 중괄호 `}`
- [ ] 변수 타입 선언 (`int`, `String`, `double` 등)
- [ ] 메서드 이름 정확히 (대소문자 구분: `substring` O, `subString` X)
- [ ] 반환 타입과 실제 return 값 일치
- [ ] ArrayList는 `<Type>` 제네릭 사용

### Task Verb (지시 동사) 해석

| 동사 | 의미 | 예시 |
|------|------|------|
| **Assume** | ~라고 가정하라 | "Assume the array is sorted" |
| **Complete** | 빈칸/메서드 본문 채우기 | "Complete method getScore" |
| **Implement / Write** | 전체를 처음부터 작성 | "Write the class Student" |

### FRQ 채점 시 감점하지 않는 실수 (Errors to Ignore)

> 출처: CED 2025-26 Scoring Guidelines (p.173). 아래 실수들은 **사소한 실수(minor error)**로 간주되어 **감점되지 않는다**.

- 부수 효과 없는 불필요한 코드 (예: precondition 체크, no-op)
- 스펠링/대소문자 실수 (문맥에서 모호하지 않으면) — 예: `ArayList` → `ArrayList`
- 지역변수를 선언하지 않은 경우 (다른 변수는 선언되어 있을 때)
- 블록 안에서 선언한 지역변수를 블록 밖에서 사용
- 지역변수에 `private` 또는 `public` 붙임
- 클래스/생성자 헤더에 `public` 누락
- 메서드 헤더에서 변수 이름만 변경 (시그니처는 동일)
- 예약어를 식별자로 사용
- 수학 기호 혼용 (×, ÷, ≤, ≥, ≠ 등)
- `[]` vs `()` vs `<>` 혼동
- `=` 대신 `==` 또는 반대
- 배열/String/ArrayList의 `length`/`size`/`length()` 혼용 (괄호 유무 포함)
- 배열 전체 참조 시 불필요한 `[]`
- 배열 선언에 크기 포함 (예: `int[size] nums = new int[size];`)
- `[i,j]` 대신 `[i][j]` (또는 반대)
- 의도가 명확한 곳에서 `;` 누락
- 들여쓰기로 의도가 보이는 곳에서 `{}` 누락
- 파라미터 없는 메서드/생성자 호출 시 `()` 누락
- `if`, `for`, `while` 조건 주위 `()` 누락

> **시험 팁**: 학생에게 "이런 사소한 실수는 감점 안 된다"고 알려주면 FRQ 시 심리적 부담이 크게 줄어든다. 완벽한 문법보다 **로직이 맞는지**에 집중하자.

### FRQ 1점 감점 항목 (1-Point Penalty)

> 감점은 문항당 **최대 3점까지만** 적용된다.

| 코드 | 감점 사유 | 예시 |
|------|---------|------|
| **v)** | 배열/컬렉션 접근 혼동 (`[]` vs `.get()`) | `list[0]` 대신 `list.get(0)` 써야 함 |
| **w)** | 부수 효과가 있는 불필요한 코드 | 요구하지 않은 `System.out.println()` 추가, 잘못된 precondition 체크 |
| **x)** | 지역변수를 사용하되 하나도 선언하지 않음 | `sum += arr[i];`를 쓰면서 `int sum = 0;` 선언 없음 |
| **y)** | 영구 데이터 파괴 | 파라미터로 받은 ArrayList를 수정, 인스턴스 변수를 의도 없이 변경 |
| **z)** | void 메서드/생성자에서 값 return | `public void setName(...)` 안에서 `return name;` |

> **핵심**: v)와 y)가 가장 흔한 감점. 특히 `[]` vs `.get()` 혼동은 학생들이 가장 많이 하는 실수이므로, 배열은 `[]`, ArrayList는 `.get()`으로 확실히 구분하자.

## 4. 4점 달성 로드맵

### 목표 점수 분석

| Unit | 출제 비중 | 예상 문제 수 | 목표 정답 |

### Unit별 구체적 학습 전략

**Unit 1 (Using Objects and Methods)** — 가장 기본이자 다른 Unit의 토대
- **우선 연습**: `String` 메서드 5종 (`substring`, `indexOf`, `length`, `equals`, `compareTo`) 자유자재로 사용
- **자주 출제**: 정수 나눗셈 함정 (`7/2 = 3`), `==` vs `.equals()`, `Math.random()` 범위 계산
- **전략**: 개념 자체는 쉬우나 함정이 많다. 치트시트의 "함정 체크리스트" 반복 숙지

**Unit 2 (Selection and Iteration)** — 시험의 핵심. MCQ에서 가장 많이 출제
- **우선 연습**: 코드 트레이싱 (변수 추적표를 종이에 그려서 연습). 최소 30문제 이상
- **자주 출제**: De Morgan's Law, 중첩 루프 실행 횟수, short-circuit evaluation, off-by-one 에러
- **전략**: 머릿속으로 풀지 말고 **반드시 종이에 표를 그려라**. 특히 for 루프의 초기값/조건/증감 확인

**Unit 3 (Class Creation)** — FRQ Q2와 직결
- **우선 연습**: 전체 클래스 작성 (private 변수 + 생성자 + getter/setter + 메서드). FRQ Q2 3문제 이상 직접 작성
- **자주 출제**: `private` vs `public` 접근, `this` 키워드, `static` vs instance, 생성자 매개변수와 인스턴스 변수 이름 충돌
- **전략**: Q2는 7점으로 배점이 높으므로, 클래스 뼈대(선언+변수+생성자)만 정확히 써도 3점 확보 가능

**Unit 4 (Data Collections)** — 비중 최대 (30-40%). 여기서 승패 결정
- **우선 연습**: ArrayList 삭제 패턴 3가지 (역순, i--, while), 2D 배열 row/column-major 순회, File/Scanner 기본 패턴
- **자주 출제**: `ArrayIndexOutOfBoundsException`, ArrayList 삭제 시 인덱스 밀림, Selection/Insertion Sort 1-2패스 추적, 재귀 트레이싱
- **전략**: FRQ Q3(ArrayList)과 Q4(2D Array)를 각 3문제 이상 실전 연습. 코딩 없이 "알고리즘만 설계"하는 연습도 효과적

### FRQ별 목표

| FRQ | 만점 | 목표 | 전략 |
|-----|------|------|------|
| Q1 Methods | 7점 | **5점** | String 메서드만 잘 쓰면 확보 가능 |
| Q2 Class Design | 7점 | **4점** | private 변수 + 생성자 + 일부 메서드 |
| Q3 ArrayList | 5점 | **3점** | 탐색 루프 + 기본 조건문 |

| **총합** | **25점** | **15점** | |

### 학습 우선순위 (효율 극대화)
1. **1순위**: String 메서드, for 루프, if/else (Q1 + MCQ 다수)
2. **2순위**: ArrayList 메서드, 삭제 패턴 (Q3)
3. **3순위**: 2D 배열 순회 (Q4)

- [ ] Selection Sort vs Insertion Sort 차이: Selection은 최소값 찾아 교환, Insertion은 적절한 위치에 삽입
- [ ] ArrayList 삭제 시 역순 순회 또는 `i--` 패턴
- [ ] 정수 나눗셈: `7/2 = 3`, `(double) 7/2 = 3.5`

## 6. 자주 틀리는 TOP 20 함정 모음

### 함정 1: `==` vs `equals()` (String 비교)
```java
String a = new String("hello");
String b = new String("hello");
System.out.println(a == b);      // false (주소 비교)
System.out.println(a.equals(b)); // true  (값 비교)
```
- **왜 틀리나**: `==`가 값을 비교한다고 착각
- **규칙**: 객체 비교는 항상 `.equals()` 사용

### 함정 2: `substring(from, to)`에서 to는 미포함
```java
String s = "ABCDE";
s.substring(1, 3); // "BC" (인덱스 1, 2만 포함. 3은 미포함!)
```
- **왜 틀리나**: to 인덱스도 포함된다고 착각
- **규칙**: `[from, to)` — from 포함, to 미포함

### 함정 3: 배열 인덱스는 0부터 시작
```java
int[] arr = {10, 20, 30};
arr[0]; // 10 (첫 번째)
arr[3]; // ArrayIndexOutOfBoundsException!
```
- **왜 틀리나**: 마지막 인덱스가 `length`라고 착각
- **규칙**: 유효 인덱스는 `0` ~ `length - 1`

### 함정 4: ArrayList 삭제 시 인덱스 밀림
```java
ArrayList<String> list = new ArrayList<>();
// list: ["A", "B", "C"]
list.remove(0); // "A" 삭제 → list: ["B", "C"]
// 이제 "B"가 인덱스 0!
```
- **왜 틀리나**: 삭제 후에도 인덱스가 그대로라고 착각
- **규칙**: 앞에서 삭제하면 뒤 요소들의 인덱스가 1씩 감소

### 함정 5: `Math.random()` 범위
```java
Math.random();              // 0.0 이상 1.0 미만
(int)(Math.random() * 6);   // 0, 1, 2, 3, 4, 5
(int)(Math.random() * 6) + 1; // 1, 2, 3, 4, 5, 6 (주사위)
```
- **왜 틀리나**: 1.0이 포함된다고 착각, 또는 캐스팅 순서 혼동
- **규칙**: `[0.0, 1.0)`, int 캐스팅은 소수점 버림(floor)

### 함정 6: `int / int`는 int (정수 나눗셈)
```java
int a = 7, b = 2;
System.out.println(a / b);       // 3 (소수점 버림!)
System.out.println((double) a / b); // 3.5
```
- **왜 틀리나**: 자동으로 소수점이 나온다고 착각
- **규칙**: 양쪽 다 int면 결과도 int. 하나라도 double이면 double.

### 함정 7: `for` 루프의 `<` vs `<=`
```java
for (int i = 0; i < 5; i++)  // i: 0,1,2,3,4 (5번)
for (int i = 0; i <= 5; i++) // i: 0,1,2,3,4,5 (6번!)
```
- **왜 틀리나**: 반복 횟수를 잘못 셈
- **규칙**: `< n`은 n번, `<= n`은 n+1번

### 함정 8: `null` 참조에서 메서드 호출
```java
String s = null;
s.length(); // NullPointerException!
```
- **왜 틀리나**: null 체크를 안 함
- **규칙**: 객체가 null일 수 있으면 `if (s != null)` 먼저

### 함정 9: `compareTo()` 반환값 해석
```java
"apple".compareTo("banana"); // 음수 (apple < banana)
"banana".compareTo("apple"); // 양수 (banana > apple)
"apple".compareTo("apple");  // 0 (같음)
```
- **왜 틀리나**: 양수/음수의 의미를 반대로 기억
- **규칙**: 호출 객체가 사전순으로 앞이면 음수, 뒤면 양수

### 함정 10: `private` 변수 직접 접근 불가
```java
class Dog {
    private String name;
}
Dog d = new Dog();
d.name = "Rex"; // 컴파일 에러! private이므로 외부 접근 불가
```
- **왜 틀리나**: private 접근제어를 무시
- **규칙**: private 변수는 같은 클래스 내에서만 접근 가능. 외부에서는 getter/setter 필요

### 함정 11: `String`은 immutable (불변)
```java
String s = "hello";
s.substring(1);       // s는 여전히 "hello"! (반환값을 받지 않으면 사라짐)
s = s.substring(1);   // 이제 s는 "ello" (반환값을 다시 대입해야 변함)
```
- **왜 틀리나**: 메서드가 원본을 바꾼다고 착각
- **규칙**: Quick Reference에 등재된 모든 String 메서드(`substring`, `indexOf`, `equals`, `compareTo`, `split`)는 **새 값을 반환**할 뿐 원본을 변경하지 않는다. 결과를 사용하려면 **반드시 변수에 다시 대입**.
- ⚠️ `toUpperCase`/`toLowerCase`/`charAt` 등은 Quick Reference에 없으므로 시험 답안에 사용 금지.

### 함정 12: `indexOf()` 없으면 -1 반환
```java
"hello".indexOf("xyz"); // -1 (없음)
"hello".indexOf("ll");  // 2
```
- **왜 틀리나**: 0을 반환한다고 착각하거나, 예외를 던진다고 착각
- **규칙**: 못 찾으면 `-1`. 예외 아님.

### 함정 13: ArrayList<Integer> remove의 인덱스 해석
```java
ArrayList<Integer> list = new ArrayList<>();
list.add(10); list.add(20); list.add(30);
list.remove(1);    // 인덱스 1의 요소 삭제 → list = [10, 30] (값 1을 찾는 게 아님!)
```
- **왜 틀리나**: `remove(1)`이 값 1을 지운다고 착각
- **규칙**: AP CSA Quick Reference에는 **`remove(int index)` 한 가지만** 등재되어 있다. `int` 인자는 **항상 인덱스**로 해석된다. 값으로 삭제하려면 직접 인덱스를 찾는 루프(헬퍼 `findIndex`)를 작성해야 한다.
- ⚠️ `remove(Object)` 형태는 Quick Reference에 없으므로 시험 답안에 사용 금지.

### 함정 14: 2D 배열의 행/열 혼동
```java
int[][] grid = new int[3][4]; // 3행 4열
grid.length;    // 3 (행 수)
grid[0].length; // 4 (열 수)
```
- **왜 틀리나**: `length`와 `[0].length`를 반대로 사용
- **규칙**: 첫 번째 `length`는 행(row), `[0].length`는 열(column)

### 함정 15: `Math.pow()` 반환 타입은 double
```java
int result = Math.pow(2, 3); // 컴파일 에러! double을 int에 대입 불가
int result = (int) Math.pow(2, 3); // OK, 8
```
- **왜 틀리나**: int를 반환한다고 착각
- **규칙**: `Math.pow()`는 항상 `double` 반환. int로 쓰려면 캐스팅 필요

### 함정 16: enhanced for 루프에서 수정 불가
```java
int[] arr = {1, 2, 3};
for (int x : arr) {
    x = x * 2; // arr은 변하지 않음! x는 복사본
}
```
- **왜 틀리나**: `x`를 바꾸면 배열도 바뀐다고 착각
- **규칙**: enhanced for의 변수는 복사본. 배열 수정하려면 인덱스 for 사용

### 함정 17: `static` 메서드에서 인스턴스 변수 접근
```java
public class Example {
    private int count = 0;
    public static void reset() {
        count = 0; // 컴파일 에러! static에서 instance 접근 불가
    }
}
```
- **왜 틀리나**: static과 instance의 차이를 혼동
- **규칙**: static 메서드는 인스턴스 변수/메서드에 직접 접근 불가. 객체를 매개변수로 받아야 함

> **참고**: 상속(extends, super, override)은 2025-26 시험 범위에서 제거됨

### 함정 18: `&&`와 `||`의 단축 평가
```java
if (arr != null && arr.length > 0) // 안전 (null이면 뒤를 안 봄)
if (arr.length > 0 && arr != null) // 위험! (null이면 NullPointerException)
```
- **왜 틀리나**: 조건 순서가 상관없다고 착각
- **규칙**: `&&`는 앞이 false면 뒤를 건너뜀. null 체크를 앞에 둘 것.

### 함정 19: `ArrayList` 순회 중 삭제 시 인덱스 밀림
```java
ArrayList<String> list = new ArrayList<>();
// list: ["a", "b", "b", "c"]
for (int i = 0; i < list.size(); i++) {
    if (list.get(i).equals("b")) {
        list.remove(i); // 삭제 후 뒤 요소가 앞으로 밀림!
    }
}
// 결과: ["a", "b", "c"] — 두 번째 "b"를 놓침!
```
- **왜 틀리나**: `remove()` 후 인덱스가 밀리는 것을 고려 안 함
- **규칙**: 순방향 삭제 시 `i--` 추가하거나, **역방향(뒤에서 앞으로) 순회**로 해결

### 함정 20: File/Scanner 사용 시 예외 처리
```java
public static void main(String[] args) throws IOException {
    Scanner sc = new Scanner(new File("data.txt"));
    while (sc.hasNext()) {
        String line = sc.nextLine();
    }
}
```
- **왜 틀리나**: `throws IOException` 없이 `new Scanner(new File(...))`를 사용하면 컴파일 에러
- **규칙**: File 객체를 Scanner에 전달할 때는 반드시 `throws IOException` 선언 필요. `nextInt()`로 숫자를 읽은 후 `nextLine()`을 호출하면 남은 개행문자를 먼저 소비함에 주의
