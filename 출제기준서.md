## 3. FRQ 출제 기준 (4문항, 총 25점)

### 3.1 FRQ 유형 고정 구조 (CED p.145, 147)

| FRQ# | 유형 | 배점 | 구조 |
|------|------|------|------|
| **Q1** | Methods and Control Structures | **7점** | Part (a) 4점 + Part (b) 3점 |
| **Q2** | Class Design | **7점** | 전체 클래스 1개 작성 |
| **Q3** | Data Analysis with `ArrayList` | **5점** | 메서드 1개 |
| **Q4** | 2D Array | **6점** | 메서드 1개 |

### 3.2 각 FRQ 상세 출제 규칙

#### Q1: Methods and Control Structures (7점)
**CED p.147, 174-175 근거**

- **Part (a) -- 4점**: 메서드 또는 생성자 1개 작성
  - 반복문(while/for) **또는** 조건문(if/else) **또는 둘 다** 필수
  - 주어진 클래스의 다른 메서드를 **호출**해야 함
  - Scoring: (1) 올바른 메서드 호출 (2) 올바른 루프 종료 조건 (3) 카운트/누적 등 부분 로직 (4) 전체 알고리즘 정합성

- **Part (b) -- 3점**: 메서드 또는 생성자 1개 작성
  - **`String` 메서드 호출 필수** (`substring`, `indexOf`, `length`, `equals`, `compareTo` 등)
  - Scoring: (1) 문자열의 필요한 부분 순회 (2) 공백/문자 위치 식별 (3) 알고리즘 전체 완성

- **출제 시 체크리스트**:
  - [ ] 시나리오와 클래스 명세 제공
  - [ ] Part (a)에 loop 또는 conditional 요구
  - [ ] Part (a)에 클래스 내 다른 메서드 호출 요구
  - [ ] Part (b)에 String 메서드 사용 필수 명시
  - [ ] 예시 입출력 테이블 제공
  - [ ] Precondition 명시

#### Q2: Class Design (7점)
**CED p.147, 164-165, 176-177 근거**

- 학생이 **전체 클래스를 처음부터 설계/구현**
- 필수 포함 요소: class header, instance variables, constructor, method(s)
- 명세는 **시나리오 설명 + 실행 예시 테이블** 형태로 제공
- 두 번째 클래스가 포함될 수 있음
- Scoring 기준 (7점):
  1. class header + 적절한 instance variables 선언
  2. constructor가 파라미터를 instance variables에 올바르게 저장
  3. 메서드의 조건 분기 로직
  4. 메서드의 상태 업데이트 (instance variable 변경)
  5. 메서드의 올바른 반환값 생성
  6. 주문 번호 등 순차적 카운터 관리
  7. 전체 알고리즘 정합성

- **출제 시 체크리스트**:
  - [ ] 현실적 시나리오 (자판기, 은행계좌, 게임 등)
  - [ ] 생성자 파라미터 2개 이상
  - [ ] 최소 1개의 메서드에 조건 분기 포함
  - [ ] 상태 변경 (instance variable 수정) 요구
  - [ ] String 반환값 생성 포함
  - [ ] 실행 예시 테이블 (Statement / Return Value / Explanation)

#### Q3: Data Analysis with ArrayList (5점)
**CED p.147, 166-167, 178-179 근거**

- **ArrayList 구조에서** 데이터 분석/조작하는 메서드 1개 작성
- **2025-26부터 array는 Q3에서 제외** -- ArrayList만
- 보조 클래스(e.g., `ItemInfo`)의 getter 메서드를 호출해야 함
- Scoring 기준 (5점):
  1. ArrayList의 모든 요소를 올바르게 순회
  2. 조건 확인을 위해 보조 클래스 메서드 올바르게 호출
  3. 조건에 맞는 요소 필터링/비교
  4. 누적/카운트/평균 등 집계 로직
  5. 전체 알고리즘 정합성 (올바른 반환)

- **출제 시 체크리스트**:
  - [ ] 보조 클래스 (getter 2-3개) partial declaration 제공
  - [ ] `ArrayList<보조클래스>` instance variable 포함하는 메인 클래스
  - [ ] 필터링 조건 2개 이상 (e.g., 가격 범위 + 재고 여부)
  - [ ] 예시 데이터 테이블 제공
  - [ ] Precondition에 null 없음, 최소 1개 조건 충족 요소 존재 명시

#### Q4: 2D Array (6점)
**CED p.147, 168-169, 180-182 근거**

- **2D 배열 구조에서** 데이터 분석/조작하는 메서드 1개 작성
- 행-열 순회(row-major) 또는 열-행 순회(column-major) 패턴 사용
- Scoring 기준 (6점):
  1. 외부 루프 (행 또는 열 순회)
  2. 내부 루프 (내부 차원 순회)
  3. 각 요소에서 올바른 메서드 호출 / 값 비교
  4. 열(또는 행)별 카운트/집계 로직
  5. 최소/최대 등 비교 결과 추적
  6. 전체 알고리즘 정합성 (올바른 반환)

- **출제 시 체크리스트**:
  - [ ] 보조 클래스 partial declaration (getter 1-2개)
  - [ ] `보조클래스[][]` 또는 `int[][]` instance variable
  - [ ] 행-열 또는 열-행 순회 필요 (단순 행-열 순회보다 열 기준 집계가 더 도전적)
  - [ ] 예시 2D 데이터 테이블 + 기대 결과 제공
  - [ ] String 비교 시 `.equals()` 사용 필요

### 3.3 FRQ 공통 출제 규칙

1. **Java Quick Reference에 있는 것만 사용 가능**: `String`, `Integer`, `Double`, `Math`, `ArrayList`, `File`, `Scanner`의 명시된 메서드만

3. **재귀 작성 요구 금지**: 재귀는 MCQ 트레이싱에서만 출제
4. **Precondition/Postcondition 반드시 명시**
5. **Task verb 사용**: "Assume", "Complete", "Implement/Write" (CED p.148)

## 5. 채점 기준 (FRQ)

### 5.1 채점 원칙 (CED p.173)

- 각 기준 **1점씩 독립 채점** (all-or-nothing per criterion)
- **Algorithm Points** `(algorithm)`: 모든 필수 단계를 올바른 순서로 완성해야 득점
- **Non-Algorithm Points**: 더 좁은 범위, 다른 에러가 있어도 해당 기준만 맞으면 득점 가능

### 5.2 Algorithm Points 불인정 사유
- 필수 단계 누락
- 단계 순서 오류
- 추가 코드가 다른 기준의 정답을 망치는 경우 (e.g., 불필요한 print, 잘못된 precondition 체크)
- `[]`와 `.get()` 혼동 (해당 문제에서 평가되지 않는 경우)
- 파라미터로 전달된 데이터를 수정
- void 메서드/생성자에서 값 반환
- 메서드 헤더의 파라미터 수/타입 변경

### 5.3 무시하는 에러 (Errors to Ignore) -- CED p.173

이하 에러는 의도가 명확할 때 **감점하지 않음**:
- 세미콜론 `;` 누락 (구조가 명확할 때)
- `{` `}` 누락 (인덴테이션으로 의도 파악 가능할 때)
- `=` vs `==` 혼동
- `length`/`size` 혼동 (array, String, ArrayList 간)
- `[i,j]` vs `[i][j]`
- `[]` vs `()` vs `<>` 혼동
- 지역변수 미선언 / 블록 안에서 선언하고 밖에서 사용
- `private`/`public` 지역변수 한정자
- 클래스/생성자 헤더에 `public` 누락
- 메서드 헤더의 변수명 변경
- 키워드를 식별자로 사용
- 수학 기호 (`x`, `+`, `<=` 대신 실제 기호) 사용
- 파라미터 없는 메서드/생성자 호출에서 `()` 누락
- 배열 선언에서 불필요한 크기 지정
- 전체 배열 참조에서 불필요한 `[]`
- 스펠링/대소문자 -- **맥락에서 모호하지 않을 때만** (`ArayList` -> `ArrayList` OK, `G` vs `g` 모호하면 안 됨)

## 6. 난이도 조절 가이드라인 (내 기준)

### 6.1 MCQ 난이도 조절 요소

| 요소 | 쉽게 | 어렵게 |
|------|------|--------|
| 코드 길이 | 3줄 이하 | 10줄 이상, 메서드 다중 호출 |
| 개념 수 | 1개 | 2-3개 조합 (e.g., 중첩 루프 + 배열 + 조건) |
| 선택지 차이 | 선택지 간 차이 큼 | off-by-one 등 미세한 차이 |
| 함정 유형 | 없음 | `==` vs `.equals()`, 정수 나눗셈, 배열 경계 |
| 추적 단계 | 1-3단계 | 5단계 이상 (재귀 3층, 중첩 루프) |

### 6.2 FRQ 난이도 조절 요소

| 요소 | 쉽게 | 어렵게 |
|------|------|--------|
| 조건 수 | 단순 조건 1개 | 복합 조건 2-3개 AND/OR |
| 보조 메서드 호출 | getter 1개 | getter 2-3개 조합 판단 |
| 데이터 변환 | 직접 반환 | 누적 -> 평균, 문자열 조립 |
| 엣지 케이스 | precondition으로 배제 | 빈 리스트, 경계값 고려 필요 |
| 2D 순회 방향 | row-major (기본) | column-major (비직관적) |

## 7. 실전 모의고사 출제 프로토콜

### 7.1 MCQ 출제 절차

1. **Unit/Practice 매트릭스 채우기**: 42문항을 Unit(행) x Practice(열)로 배분
2. **토픽 태깅**: 각 문항에 Learning Objective + Essential Knowledge 태그 부여
3. **난이도 배분**: Easy 10 / Medium 21 / Hard 11
4. **Set-based 문항 포함**: 최소 2-3세트 (4-6문항)
5. **함정 포인트 삽입**: 정수 나눗셈, off-by-one, `==` vs `.equals()`, autoboxing 등
6. **정답 분포 확인**: A/B/C/D 각 10-11개 (편향 금지)
7. **Quick Reference만으로 풀 수 있는지 확인**: 범위 외 API 사용 금지

### 7.2 FRQ 출제 절차

1. **현실적 시나리오 선정**: 학교, 가게, 게임, 일정 등 학생 친숙한 맥락
2. **클래스 구조 설계**: 보조 클래스 partial declaration + 메인 클래스
3. **예시 데이터 & 기대 출력 작성**: 테이블 형태, 최소 3-4개 케이스
4. **Precondition/Postcondition 명시**
5. **채점 기준 작성**: 각 1점짜리 독립 기준 (algorithm + non-algorithm)
6. **Canonical solution 작성**: 모범 답안
7. **Common errors 예측**: 학생 실수 패턴 미리 파악
